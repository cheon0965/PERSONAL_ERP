# 메인 비즈니스 흐름 개선 실행계획

기준일: `2026-04-05`

문서 상태: `아카이브`

이 문서는 초기 실행계획을 바탕으로, 현재는 `실행계획 + 반영 현황 + 남은 후속 범위`를 함께 관리하는 기준 문서다.
신규 기능 제안서라기보다, 메인 비즈니스 흐름 개선 작업이 어디까지 끝났고 무엇이 아직 남아 있는지 빠르게 판단하기 위한 운영 문서로 본다.

## 목적

- `작업 문맥 -> 기준 데이터 -> 월 운영 -> 계획/수집 -> 전표 -> 마감/보고` 흐름이 제품 안에서 끊기지 않게 유지한다.
- 화면이 다음 행동 판단에 필요한 정보를 충분히 보여주게 만든다.
- 도메인 문서, API 계약, Web 화면의 상태 의미를 서로 맞춘다.
- 개선 범위를 `월 운영 backbone 강화`에 집중하고, 범용 ERP 확장으로 번지지 않게 관리한다.

## 범위와 제외

현재 이 문서의 범위:

- 메인 운영 루프의 가시성, 정합성, 추적성, 설명력 강화
- 기준 데이터 준비 경로의 제한적 보강
- 실제 build 결과물 기준 대표 브라우저 스모크 자동화

현재 의도적으로 제외하는 범위:

- 범용 기준 데이터 관리자
- 사용자 자유 계정과목/거래유형 설계 UI
- 다중 workspace 선택/전환 UI
- 자금수단의 잔액 직접 수정, 유형 변경, 참조가 있는 자금수단의 강제 삭제

## 현재 상태 요약

- `1차. 흐름 가시성 복구`: 완료
- `2차. 추적성과 설명력 강화`: 완료
- `3차. 준비 경로 보강`: 완료
- 후속 보강:
  - 주요 empty state/경고 CTA 보강 완료
  - 기준 데이터 관리 Web/E2E 보강 완료
  - 반복 규칙 Web/E2E 보강 완료
  - 자금수단 `CLOSED` lifecycle 제한적 확장 완료
  - build 결과물 기준 스모크 자동화 완료
  - 운영 체크리스트 기반 수동 스모크 일부 자동화 완료

현재 기준으로 메인 비즈니스 흐름 개선의 `핵심 목표`는 대부분 반영됐다.
이 문서의 남은 역할은 `지금 바로 이어갈 후속 작업`과 `나중에 다시 판단할 확장 범위`를 분리해서 관리하는 것이다.

## 지금 반영되어야 할 남은 항목

현재 기준으로 `즉시 반영이 꼭 필요한 남은 항목`은 없다.

직전에 완료된 범위:

- `운영 체크리스트 기반 수동 스모크 일부 자동화`
  - `dashboard`의 운영 기간 없음 empty state와 CTA
  - `transactions`의 기준 데이터 readiness 경고와 운영 월 없음 안내
  - `reference-data`의 readiness 부족 상태
  - `financial-statements`, `carry-forwards`의 대표 보고 empty state CTA
  - `settings`와 `Topbar`의 작업 문맥 fallback
  - 위 항목을 dev smoke와 build-output smoke 양쪽에서 검증

현재부터는 핵심 흐름 복구보다 `운영 품질 심화` 또는 `후순위 확장 재검토` 성격이 강하다.

## 나중에 다시 고려해야 할 항목

### 1. 운영 체크리스트 심화 자동화

현재 위치:

- 대표 empty state, readiness 경고, 문맥 fallback은 자동화됐다.
- 이후에는 릴리스 전 수동 체크 중 더 세밀한 운영 점검 항목을 점진적으로 옮길 수 있다.

후속 후보:

- 운영 기간 재오픈/잠금 관련 경고 흐름
- 기준 데이터 부족 조합별 안내 문구
- 보고 화면의 권한별 생성 가능/불가 안내

### 2. workspace switcher

이 항목은 이후 구현 완료로 전환됐다.

- `GET /auth/workspaces`로 사용자별 active membership 기반 접근 가능 사업장 목록을 노출한다.
- `POST /auth/current-workspace`로 현재 세션의 작업 문맥을 명시적으로 전환한다.
- Topbar `기준` 팝오버에서 여러 사업장에 속한 일반 사용자가 현재 사업장/장부와 권한을 확인하고 전환할 수 있다.
- 전환 후 보호 API와 화면은 새 `currentWorkspace` 기준으로 동작한다.

남는 후속 후보:

- 한 사업장 안에 장부가 여러 개인 경우를 위한 별도 장부 선택 UI
- 전환 후 캐시/권한/라우팅 일관성의 브라우저 회귀 자동화

### 3. 범용 기준 데이터 관리자 확장

현재 보류 이유:

- 현재 방향성은 월 운영 backbone 강화이지, 범용 마스터 데이터 관리자 확장이 아니다.
- 계정과목, 거래유형은 여전히 system-managed가 더 안전하다.

재검토 조건:

- 실제 운영 막힘이 현재 제한적 CRUD 범위를 넘어서는지 확인될 때
- 계정과목/거래유형 편집이 도메인 정합성을 해치지 않는 별도 설계가 준비될 때

### 4. 자금수단 lifecycle 추가 확장

현재 보류 이유:

- 현재 안전 범위는 `ACTIVE -> INACTIVE -> CLOSED`와 종료 후 읽기 전용이다.
- `CLOSED` 재활성화, 유형 변경, 잔액 직접 수정은 기존 기록 해석을 흔들 수 있다.

재검토 조건:

- 운영상 실제 종료 해제 요구가 반복 확인될 때
- 기존 거래/반복 규칙/전표/보고에 대한 영향 규칙이 먼저 정리될 때

### 5. 반복 규칙 세부 정책 확장

현재 보류 이유:

- 기본 생성/수정/삭제와 브라우저 검증은 충분히 올라왔다.
- 지금은 세부 recurrence 정책보다 backbone 안정성이 우선이다.

재검토 조건:

- 실제 운영에서 반복 규칙 예외 정책 요구가 누적될 때
- 계획 생성 정책과 월 운영 루프에서 우선순위가 올라갈 때

### 6. 빌드 후 보고/아티팩트 가시성 정리

현재 위치:

- build 결과물 smoke 자체는 이미 반영됐다.
- 이후에는 리포트 업로드 범위, 실패 시 artifact 구성, 체크리스트 연동 같은 보조 장치를 다듬을 수 있다.

우선순위 판단:

- 핵심 흐름 강화보다 운영 편의성 보강에 가까우므로, 현재는 `지금 반드시 해야 할 항목`보다 한 단계 뒤로 둔다.

## 공통 작업 원칙

- 모든 변경은 `도메인 문서 -> 계약 타입 -> API -> Web UI -> 테스트 -> 문서` 순서로 영향을 점검한다.
- 상태 이름이 바뀌거나 늘어나도 원본 업무 상태 의미를 잃지 않는다.
- 새 UI는 항상 “사용자가 다음 행동을 판단할 수 있는가”를 완료 기준에 포함한다.
- 서버가 가진 업무 판정을 우선 노출하고, 임의 클라이언트 계산은 최소화한다.
- 각 단계는 최소 `테스트 1종 이상 + 문서 1건 이상`을 함께 남긴다.

## 검증 기준

- API 요청 테스트:
  - 권한 검증, workspace 범위 검증, 상태 정책 검증, 자동 판정 응답 검증
- Web 테스트:
  - 상태 칩, 링크, empty state, context 노출, 기준 데이터 readiness 노출
- 브라우저 E2E:
  - 대표 운영 루트 smoke
  - 기준 데이터 관리/반복 규칙/대표 입력 흐름 smoke
  - build 결과물 기준 smoke
- 문서 검증:
  - `DEMO_GUIDE`, `business-logic-draft`, `API`, `VALIDATION_NOTES` 드리프트 점검

## 완료 판정

이 개선 계획은 아래 조건을 만족한 상태로 본다.

1. 운영자가 앱 안에서 메인 운영 루프의 현재 위치와 다음 단계를 스스로 따라갈 수 있다.
2. 운영자는 제품 안에서 현재 준비 상태와 다음 행동을 스스로 판단할 수 있다.
3. 수집 거래와 업로드 자동화 상태를 화면만 보고 해석할 수 있다.
4. 계획, 수집, 전표의 연결이 화면에서 추적 가능하다.
5. 장부 문맥 착오 가능성을 줄이는 상시 문맥 노출이 있다.
6. 문서와 구현이 같은 상태 용어와 같은 추천 순서를 사용한다.

현재 기준으로 위 조건은 `현재 범위 안에서 충족`된 상태다.
따라서 이후 작업은 핵심 흐름 복구가 아니라 `운영 검증 품질`과 `확장 판단 보류 항목 관리` 중심으로 이어간다.
