# UI 화면 정리 실행계획

기준일: `2026-04-14`

문서 상태: `completed`

이 문서는 현재 Web 화면 구조를 기준으로, PERSONAL ERP를 더 읽기 쉽고 덜 복잡한 ERP 화면 체계로 정리하기 위한 실행계획이다.
목표는 단순히 예쁘게 바꾸는 것이 아니라, `한 화면에서 지금 무엇을 해야 하는지`가 즉시 보이도록 정보 우선순위와 공통 패턴을 다시 세우는 데 있다.

## 목적

- 화면 전반의 가시성과 통일성을 높인다.
- 각 화면을 `한 화면 = 한 핵심 작업` 구조로 재정리한다.
- 헤더, 보조 내비게이션, 카드, 표, 폼의 공통 규칙을 다시 정의한다.
- 월 운영 중심 ERP 흐름이 화면 구조만 봐도 자연스럽게 이어지게 만든다.

## 이번 검토 기준

이번 계획은 아래 기준을 함께 보고 정리했다.

- 현재 화면 흐름 문서: `docs/SCREEN_FLOW_GUIDE.md`
- 현재 디자인 규칙 문서: `docs/DESIGN_SYSTEM.md`
- 공통 레이아웃/헤더: `apps/web/src/shared/layout/*`, `apps/web/src/shared/ui/*`
- 실제 주요 화면 구현: `apps/web/src/features/*`

즉, 이번 문서는 시안 문서라기보다 `현재 코드 구조와 화면 책임 분포를 바탕으로 한 UI 재정리 착수 계획`으로 본다.

## 현재 진행 현황

현재 Web UI는 실행계획의 큰 축을 대부분 반영한 상태다.

- `Phase 1`
  - App Shell, 사이드바 그룹, 브랜드 문구 축약, `PageHeader`, `SectionTabs`, `DataTableCard.toolbar` 공통화가 반영되었다.
- `Phase 2`
  - `월 운영`, `계획 항목`, `수집 거래`, `전표 조회`, `업로드 배치`, `보험 계약`, `차량 운영`, `반복 규칙`이 `한 화면 = 한 핵심 작업` 구조로 재정리되었다.
- `Phase 3`
  - `운영 지원`, `기준 데이터`, `관리자`, `설정` 화면군이 `헤더 기준 -> 짧은 상단 상태 -> 실제 처리 목록` 패턴으로 정리되었다.
- `Phase 4`
  - `재무제표`, `차기 이월`, `기간 전망`, `대시보드` 등 보고 화면군이 `기준 선택 -> 결과 해석` 흐름으로 압축되었다.
- `Phase 5`
  - 문서 동기화와 완료 상태 정리가 반영되었다.

즉, 이 실행계획의 범위 안에서 예정했던 주요 구조 개편은 모두 반영되었다. 이후 남는 일은 별도 실행계획 없이 이어갈 수 있는 `문구 polishing`, `간격 미세 조정`, `추가 검증 보강` 성격에 가깝다.

## 현재 문제 요약

### 1. 전역 셸이 ERP 화면 구조를 충분히 정리해 주지 못한다

- `apps/web/src/shared/layout/sidebar-nav.tsx`의 브랜드 블록은 `PERSONAL ERP` 아래 긴 설명 문구가 붙어 있고, 현재 폭 `264px` 안에서 줄바꿈이 쉽게 발생한다.
- `apps/web/src/shared/config/navigation.ts`는 1차 메뉴가 많고 모두 같은 깊이로 평평하게 나열되어 있어, 사용자는 `지금 운영 흐름상 어디에 있는지`보다 `메뉴 목록을 훑는 일`에 더 주의를 쓰게 된다.
- `Topbar`는 작업 문맥을 잘 보여주지만, 각 화면 헤더까지 긴 설명이 반복되면서 상단 밀도가 높아진다.

### 2. 같은 계층의 화면인데도 보조 내비게이션 패턴이 제각각이다

- `reference-data`는 `Tabs`
- `operations`, `admin`, `settings`는 `outlined Button` 묶음

같은 “상위 영역 안의 하위 화면 전환”인데 패턴이 달라 화면군마다 다른 제품처럼 느껴질 여지가 있다.

### 3. `PageHeader`가 너무 많은 설명을 떠안고 있다

- `apps/web/src/shared/ui/page-header.tsx`는 현재 `eyebrow`, `title`, `description`, `primaryAction` 정도만 처리한다.
- 실제 각 페이지에서는 헤더 설명문이 길어지고, 화면별로 높이와 밀도가 달라진다.
- 헤더 안에서 `현재 상태`, `선택한 기간`, `핵심 CTA`, `보조 액션`, `문맥 요약`을 일정하게 표현하는 규칙이 없다.

### 4. 쓰기 화면과 읽기 화면의 밀도 규칙이 섞여 있다

현재 큰 화면들은 `요약 카드 + 설명 카드 + 표 + 차트 + 폼 + 상태 메시지`가 한 페이지에 함께 쌓이는 경우가 많다.
특히 아래 화면은 한 화면 안의 책임이 크다.

- `accounting-periods-page.tsx`: `496` lines
- `accounting-periods-page.sections.tsx`: `477` lines
- `vehicles-page.tsx`: `473` lines
- `financial-statements-page.tsx`: `463` lines
- `journal-entries-page.tsx`: `423` lines
- `dashboard-page.tsx`: `394` lines
- `plan-items-page.tsx`: `392` lines
- `forecast-page.tsx`: `389` lines
- `recurring-rules-page.tsx`: `382` lines
- `transactions-page.sections.tsx`: `379` lines

코드 길이 자체가 문제는 아니지만, 현재는 `한 화면이 한 번에 너무 많은 역할을 품고 있다`는 신호로 볼 수 있다.

### 5. 요약 카드 사용 기준이 화면마다 다르다

- 운영 허브, 차량 운영, 전망, 재무제표, 예외/알림/상태 화면에서 `SummaryCard`가 적극적으로 쓰인다.
- 반면 실제 입력과 확정이 중요한 화면에서도 상단 설명과 상태 카드가 커져 본문 작업 영역보다 시선을 더 가져가는 구간이 있다.

ERP 화면은 화려한 요약보다 `현재 대상`, `상태`, `다음 행동`, `주요 리스트`가 먼저 읽혀야 한다.

## 목표 상태

정리 후의 목표 상태는 아래와 같다.

### 1. 전역 구조

- 좌측 사이드바는 `짧은 브랜드 + 그룹화된 메뉴`로 정리한다.
- 상단 바는 작업 문맥 중심으로 유지하되, 페이지 헤더와 정보 중복을 줄인다.
- 모든 주요 화면은 같은 상단 리듬으로 시작한다.

### 2. 페이지 구조

모든 대시보드 라우트 화면은 아래 구조 중 하나로 수렴시킨다.

1. `헤더`
2. `국소 내비게이션`
3. `핵심 상태/선택 바`
4. `주 작업 영역`
5. `보조 정보 영역`

핵심 원칙은 `주 작업 영역이 화면의 시선 중심`이 되게 만드는 것이다.

### 3. 화면 역할

- 쓰기 화면: 표/폼/액션이 먼저
- 읽기 화면: 요약/추이/해석이 먼저
- 허브 화면: 링크보다 우선순위가 먼저
- 기준 데이터 화면: 준비 상태와 관리 기능을 분리

### 4. 텍스트 밀도

- 헤더 설명은 한두 문장 안으로 제한한다.
- 긴 도메인 설명은 `도메인 도움말`, 보조 카드, 문서로 보낸다.
- 사이드바와 보조 내비게이션의 라벨은 가능한 한 짧게 유지한다.

## 공통 설계 원칙

### 1. 한 화면의 1순위 작업은 하나만 둔다

- `월 운영`: 월 시작/마감/재오픈
- `계획 항목`: 생성과 상태 확인
- `수집 거래`: 입력/보완/확정 준비
- `전표 조회`: 공식 전표 확인과 조정

나머지 기능은 같은 화면 안에 있더라도 `보조 영역`, `드로어`, `탭`, `세부 패널`로 낮춰 배치한다.

### 2. 헤더는 설명문이 아니라 조작 기준점이어야 한다

공통 헤더는 최소 아래 요소를 안정적으로 담아야 한다.

- 영역명
- 페이지 제목
- 한 줄 설명
- 상태 칩 또는 선택 상태
- 기본 액션 1개
- 필요 시 보조 액션 1개

### 3. 로컬 내비게이션은 한 패턴으로 통일한다

- 상위 영역 내부 전환은 `Tabs` 또는 같은 수준의 세그먼트 패턴 하나로 통일한다.
- `operations`, `admin`, `settings`, `reference-data`는 같은 UX 규칙을 쓴다.

### 4. 요약 카드는 최대 4개까지, 읽기 화면 위주로 쓴다

- 쓰기 화면에서는 0~2개만 허용하거나 생략한다.
- 읽기 화면에서도 첫 줄 요약 카드가 본문보다 강해지지 않게 한다.
- KPI가 없으면 카드 대신 간단한 상태 바나 인라인 메타 정보를 쓴다.

### 5. 설명성보다 조작성을 우선한다

- 표가 핵심인 화면은 표를 먼저 보여준다.
- 생성/수정은 드로어나 다이얼로그로 유지하되, 현재 선택 대상과 결과를 분명히 연결한다.
- 긴 안내 문구보다 `막히는 조건`, `지금 가능한 액션`, `다음 이동 경로`를 짧게 준다.

## 공통 컴포넌트 정리 방향

### 1. App Shell

대상 파일:

- `apps/web/src/shared/layout/sidebar-nav.tsx`
- `apps/web/src/shared/layout/topbar.tsx`
- `apps/web/src/shared/config/navigation.ts`

정리 방향:

- 브랜드 보조 문구를 더 짧게 바꾸거나 제거한다.
- 1차 메뉴를 그룹화한다.
- 그룹 예시:
  - 운영 준비: 작업 문맥, 관리자, 운영 지원, 기준 데이터
  - 월 실행: 월 운영, 반복 규칙, 계획 항목, 업로드 배치, 수집 거래, 전표 조회
  - 자산/계약: 보험 계약, 차량 운영
  - 보고/판단: 재무제표, 차기 이월, 기간 전망, 대시보드

### 2. Page Header

대상 파일:

- `apps/web/src/shared/ui/page-header.tsx`

정리 방향:

- 긴 설명을 강제하지 않는 더 얇은 헤더로 바꾼다.
- `상태 칩`, `보조 액션`, `헤더 메타 행`을 지원하는 `PageHeader v2`를 만든다.
- 헤더 높이와 타이포 리듬을 고정해 화면별 흔들림을 줄인다.

### 3. Section Navigation

신규 공통화 후보:

- `shared/ui/section-tabs.tsx` 또는 동등한 공통 컴포넌트

정리 방향:

- `operations`, `admin`, `settings`, `reference-data`의 하위 전환을 공통 컴포넌트로 통일한다.
- 모바일에서 가로 스크롤 또는 2줄 배치를 같은 기준으로 처리한다.

### 4. 카드 계열

대상 파일:

- `apps/web/src/shared/ui/section-card.tsx`
- `apps/web/src/shared/ui/summary-card.tsx`
- `apps/web/src/shared/ui/data-table-card.tsx`
- `apps/web/src/shared/ui/chart-card.tsx`

정리 방향:

- 카드 제목, 설명, 액션의 간격 규칙을 더 단단히 통일한다.
- `DataTableCard`는 리스트 중심 화면의 기본 뼈대로 삼고, 필요 시 상단 툴바 슬롯을 더 명확히 만든다.
- `SummaryCard`는 read-heavy 화면 전용으로 사용 기준을 문서화한다.

## 화면군별 정리 계획

아래 계획 항목은 `예정 작업 목록`이면서 동시에 `현재 반영 상태를 점검하는 체크리스트`로 사용한다.

## 1. P0: 월 운영 핵심 쓰기 흐름

대상 화면:

- `월 운영`
- `계획 항목`
- `수집 거래`
- `전표 조회`

### 월 운영

현재 문제:

- 시작, 현재 상태, 마감, 재오픈, 기간 목록이 한 흐름 안에 묶여 있다.
- 운영 lifecycle과 관리 이력이 동시에 보여 정보 계층이 두껍다.

정리 방향:

- 상단은 `현재 운영 월 상태`와 `주 액션`만 둔다.
- 본문은 `시작`, `마감`, `재오픈`, `기간 이력`을 명확히 분리한다.
- `현재 운영 월 카드 + 하단 탭/섹션` 구조로 재정렬한다.

### 계획 항목

현재 문제:

- 기간 선택, 생성 액션, 확정 피드백, 요약, 연결 전표 추적이 같은 높이로 섞여 있다.

정리 방향:

- 상단 툴바에서 `대상 기간 선택 + 생성`을 처리한다.
- 본문 1순위는 `계획 항목 표`로 둔다.
- 연결 수집 거래/전표 상태는 표 내부 또는 우측 요약 패널로 제한한다.

### 수집 거래

현재 문제:

- 운영 월 상태, readiness, 필터, 안내, 액션이 모두 상단에 몰려 있다.
- 핵심인 거래 목록 진입 전 인지 부담이 크다.

정리 방향:

- 상단은 `현재 월 / readiness / 등록 버튼`만 보여준다.
- 필터는 표 바로 위 툴바로 내린다.
- 생성/수정은 드로어 유지, 안내 문구는 축약한다.

### 전표 조회

현재 문제:

- 전표를 카드 단위로 길게 펼쳐 보여 읽는 비용이 크다.
- 조회와 조정 액션이 같은 카드 안에서 길게 반복된다.

정리 방향:

- `전표 목록 + 선택 상세` 구조로 바꾼다.
- 기본 화면은 목록 중심, 선택한 전표의 상세 라인과 조정 버튼은 우측 또는 하단 상세 패널에 둔다.
- 반전/정정은 상세 패널 안에서만 노출한다.

## 2. P1: 기준 데이터/운영 지원/관리 화면 정리

대상 화면:

- `운영 지원`
- `기준 데이터`
- `작업 문맥/설정`
- `관리자`

### 운영 지원

현재 문제:

- 허브 카드가 8개로 많고 모두 같은 강도로 보인다.
- `오늘 처리`, `월 마감`, `시스템 상태`의 우선순위 차이가 약하다.

정리 방향:

- `오늘 우선 확인`, `월 마감`, `운영 도구`, `시스템/감사` 그룹으로 분해한다.
- 첫 화면은 링크 모음이 아니라 `지금 처리할 것` 중심 허브로 바꾼다.

### 기준 데이터

현재 문제:

- readiness와 관리 범위는 잘 나뉘어 있으나 전체 영역의 정보 설계가 더 단단해질 필요가 있다.

정리 방향:

- `준비 상태`는 read-only 점검 화면으로 더 간결하게 유지한다.
- `관리`는 실제 편집 화면답게 표/편집 중심으로 재정렬한다.
- 로컬 탭 패턴은 다른 영역과 통일한다.

### 작업 문맥/설정, 관리자

현재 문제:

- 하위 화면 전환 방식이 다른 영역과 일관되지 않다.
- 정보 카드와 설정 폼의 비중이 화면마다 다르다.

정리 방향:

- 같은 탭/세그먼트 구조로 맞춘다.
- 개요성 카드보다 실제 설정 폼 또는 표가 먼저 보이도록 정리한다.

## 3. P1: 운영 기준 데이터 화면 정리

대상 화면:

- `차량 운영`
- `보험 계약`
- `반복 규칙`

### 차량 운영

현재 문제:

- 차량 요약, 차트, 차량 목록, 주유 기록, 정비 기록이 한 화면에 함께 있어 가장 과밀하다.

정리 방향:

- 최소 `차량`, `주유`, `정비` 3개 하위 화면 또는 탭으로 분리한다.
- 첫 화면은 차량 운영 개요 또는 차량 목록 하나만 중심으로 보여준다.
- 차트는 개요 탭에서만 유지하고, 기록 화면은 표 중심으로 바꾼다.

### 보험 계약 / 반복 규칙

현재 문제:

- 기준 데이터 화면인데 설명과 보조 상태가 길어질 수 있다.

정리 방향:

- 목록 우선 구조로 정리한다.
- 선택 상세와 수정 액션은 드로어/상세 패널로 집중시킨다.
- “계획 항목으로 어떻게 이어지는지”는 헤더가 아니라 보조 도움말로 뺀다.

## 4. P2: 보고/판단 화면 정리

대상 화면:

- `대시보드`
- `기간 전망`
- `재무제표`
- `차기 이월`

### 공통 방향

- 이 화면군은 read model 성격이 강하므로 요약 카드와 차트를 허용한다.
- 대신 `한 페이지 안의 카드 종류 수`를 줄여 시선 이동을 통제한다.

### 대시보드 / 기간 전망

정리 방향:

- KPI는 최대 4개까지 유지한다.
- 차트는 핵심 1개만 우선 노출한다.
- 해석 카드와 경고 카드는 수를 줄이고, 필요한 경우 접거나 묶는다.

### 재무제표

현재 문제:

- 대상 선택, 생성, 비교, 보고서 본문, 기준선 정보가 한 화면에 몰려 있다.

정리 방향:

- `대상 선택/생성`을 1단계
- `요약 숫자`를 2단계
- `보고서 상세`를 3단계

이 순서로 읽히게 구조를 재정렬한다.

## 구현 단계

### Phase 0. 기준선 고정

- 현재 화면 목록과 책임을 문서로 고정한다.
- 대표 화면 캡처 또는 수동 체크 기준을 만든다.
- `DESIGN_SYSTEM.md`에 이번 재정리 원칙을 반영할 준비를 한다.

### Phase 1. 공통 셸과 헤더 정리

- 사이드바 브랜드/메뉴 그룹 정리
- 로컬 내비게이션 공통 컴포넌트 도입
- `PageHeader v2` 도입
- 카드/표/헤더 간격 규칙 고정

현재 상태:

- 완료. `PageHeader`, `SectionTabs`, `DataTableCard.toolbar`, 사이드바 그룹 구성이 코드에 반영됨.

### Phase 2. 핵심 쓰기 화면 개편

- 월 운영
- 계획 항목
- 수집 거래
- 전표 조회

이 단계가 끝나면 메인 운영 루프의 체감 품질이 가장 크게 올라가야 한다.

현재 상태:

- 완료. 월 운영, 계획 항목, 수집 거래, 전표 조회를 `현재 기준 -> 작업대/목록 -> 상세` 흐름으로 재배치함.

### Phase 3. 운영 기준/관리 화면 개편

- 운영 지원
- 기준 데이터
- 설정
- 관리자
- 차량 운영
- 보험 계약
- 반복 규칙

현재 상태:

- 완료. 운영 지원 전체, 기준 데이터, 설정, 관리자, 차량 운영, 보험 계약, 반복 규칙이 공통 패턴으로 정리됨.

### Phase 4. 보고 화면 개편

- 대시보드
- 기간 전망
- 재무제표
- 차기 이월

현재 상태:

- 완료. 보고 화면군은 `헤더 -> 기준선/대상 선택 -> 요약/차트/해석` 구조로 정리됨.

### Phase 5. 검증과 문서 동기화

- `SCREEN_FLOW_GUIDE.md` 갱신
- `DESIGN_SYSTEM.md` 갱신
- 주요 화면 스모크 또는 E2E 보완
- 변경 후 용어와 화면 순서 재점검

현재 상태:

- 완료. `SCREEN_FLOW_GUIDE.md`, `DESIGN_SYSTEM.md`를 현재 코드 구조 기준으로 동기화했고, 이 문서는 완료 문서로 보관한다.

## 완료 기준

아래 조건을 만족하면 1차 UI 정리 작업이 완료된 것으로 본다.

1. 사이드바 브랜드와 메뉴가 줄바꿈 없이 안정적으로 읽힌다.
2. 상위 영역 하위 전환 패턴이 하나로 통일된다.
3. 모든 핵심 화면에서 헤더 높이와 구조가 비슷하게 유지된다.
4. 핵심 쓰기 화면은 첫 화면에서 `지금 해야 할 작업`이 즉시 드러난다.
5. 읽기 화면은 KPI, 차트, 해석 카드 수가 과도하지 않다.
6. 한 화면 안에서 동시에 경쟁하는 주요 액션 수가 줄어든다.
7. `작업 문맥 -> 기준 데이터 -> 월 운영 -> 계획/수집 -> 전표 -> 보고` 흐름이 UI만 봐도 자연스럽게 이어진다.

## 이번 단계에서 제외할 범위

- 새로운 도메인 기능 추가
- 새로운 관리자 권한 체계 추가
- 전면적인 브랜딩 리디자인
- Figma 우선 작업만 하고 코드 기준을 비워 두는 방식

이번 계획의 초점은 `현재 있는 기능을 더 읽기 쉽고 덜 복잡하게 다시 배열하는 것`이다.

## 바로 다음 액션 제안

실행 순서는 아래처럼 시작하는 것이 가장 효율적이다.

1. `App Shell + PageHeader + SectionNav` 공통 패턴부터 정리한다.
2. 그다음 `월 운영`, `수집 거래`, `전표 조회`를 먼저 개편한다.
3. 이후 `차량 운영`과 `재무제표`처럼 가장 과밀한 화면을 정리한다.

이 순서를 따르면 화면 수는 많아도, 사용자 체감 품질은 초반 단계에서 빠르게 개선될 가능성이 높다.
