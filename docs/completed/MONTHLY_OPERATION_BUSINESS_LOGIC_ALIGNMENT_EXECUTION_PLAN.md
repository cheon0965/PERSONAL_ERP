# 월별 운영 비즈니스 로직 정합성 개선 실행계획

> 보관 상태: `2026-04-23` 기준 P0-P3 구현, 전체 회귀 테스트, 문서 동기화를 완료한 뒤 `docs/completed/`로 이동했다. 현재 운영 기준은 `docs/API.md`, `docs/CURRENT_CAPABILITIES.md`, `docs/DEMO_GUIDE.md`, `docs/VALIDATION_NOTES.md`, `docs/domain/business-logic-draft.md`, `docs/domain/core-entity-definition.md`를 우선한다.

## 배경

현재 프로젝트는 소상공인이 한 달 단위로 `기준 데이터 준비 -> 월 운영 시작 -> 계획/수집 -> 전표 -> 마감 -> 보고 -> 차기 이월`을 닫아 가는 재무 운영 ERP다. 최근 정책은 운영 중에는 하나의 최신 진행월만 열어 두고, 업로드 배치의 운영월 자동 생성은 운영 시작 전 기초데이터 적재 또는 신규 계좌/카드 기초 업로드 예외로 제한하는 방향이다.

코드 점검 결과 API 핵심 가드는 이 방향으로 상당 부분 정리되어 있지만, Web 화면의 가능 상태 계산과 안내 문구, 일부 보조 도메인의 추적/수명주기 로직이 이전의 다중 열린 월 모델을 일부 유지하고 있었다. 이 문서는 발견사항을 모두 개선하기 위한 실행 순서와 완료 기준을 고정한다.

## 목표

1. 월 운영 정책을 "하나의 최신 진행월" 기준으로 API, Web, 문서, 테스트에 일관되게 반영한다.
2. 업로드 배치의 월 자동 생성은 명시적인 기초데이터 적재와 신규 계좌/카드 bootstrap 예외에만 허용한다.
3. 차량 운영비 회계 연동의 남은 후속 항목을 이 계획으로 편입해, 기존 차량 실행계획은 완료 문서로 보관한다.
4. 거래 출처, 감사, 이월, import parser의 미완료 지점을 제품 정책과 구현 중 하나로 확정한다.

## 실행 원칙

- 월별 운영의 쓰기 진입점은 최신 진행월 하나로 수렴한다.
- 과거 잠금월과 과거 열린월은 조회와 보고의 대상이며, 쓰기 대상이 아니다.
- 예외 흐름은 화면에서 먼저 설명하고, API에서 최종 방어한다.
- 차량/보험/반복계획 같은 보조 도메인은 표준 수집거래와 전표 파이프라인을 재사용한다.
- 기존 사용자 데이터에 다중 열린 월이 남아 있을 수 있으므로, UI는 낡은 기간을 읽기 전용으로 보여 주되 저장은 최신 진행월로만 유도한다.

## Phase 0. 기준선 고정

1. 현재 문서와 테스트 기준선을 먼저 확인한다.
   - `docs/API.md`
   - `docs/DEMO_GUIDE.md`
   - `docs/domain/business-logic-draft.md`
   - `apps/api/test/accounting-periods.request-api.test.ts`
   - `apps/api/test/import-batches.request-api.collect.test.ts`
   - `apps/api/test/transactions.request-api.create.test.ts`
   - `apps/api/test/transactions.request-api.journal-adjustments.test.ts`
2. 월 운영 정책 용어를 통일한다.
   - 최신 진행월: 수집/전표 입력이 가능한 기간 중 가장 최근 기간
   - 잠금월: 마감 완료되어 쓰기 금지인 기간
   - 기초 업로드: 운영 시작 전 또는 신규 계좌/카드 등록 직후에만 허용되는 초기 데이터 적재
3. 다중 열린 월이 있는 기존 데이터는 새 정책에서 예외/복구 대상으로 취급한다.

완료 기준:

- 새 정책 용어가 API, Web copy, 문서에서 같은 의미로 사용된다.
- 기존 완료 문서의 과거 정책과 현재 정책이 혼동되지 않도록 링크가 정리된다.

## Phase 1. 최신 진행월 정책 정렬

### 1. 월 재오픈

작업:

1. Web `buildAccountingPeriodReopenEligibility`가 최신 기간 여부를 함께 판단하게 만든다.
2. 최신 기간이 아닌 잠금월은 `재오픈 불가`로 표시하고, API와 같은 이유를 안내한다.
3. `/periods/close` 또는 재오픈 섹션의 설명에서 "차기 이월/다음 월 오프닝 스냅샷" 조건만 언급하던 문구를 최신월 조건까지 포함하도록 수정한다.
4. API/Web 테스트에 "이전 잠금월은 화면 eligibility와 API 모두 재오픈 불가" 시나리오를 추가한다.

완료 기준:

- 과거 잠금월은 버튼이 비활성화된다.
- 최신 잠금월은 종속 산출물이 없을 때만 재오픈 가능하다.
- API 거절 사유와 Web 안내가 같은 정책을 말한다.

### 2. 월 오픈

작업:

1. 월 오픈 화면에서 "다음 운영 월을 미리 열어 둘 수 있습니다" 문구를 제거한다.
2. 최신 운영 기간이 `LOCKED`가 아니면 다음 월 오픈 버튼을 비활성화하거나 명시 경고를 보여 준다.
3. 첫 월 오픈과 후속 월 오픈의 안내를 분리한다.
4. `openAccountingPeriod` API 오류를 화면의 사전 검증과 같은 문구 체계로 정리한다.

완료 기준:

- 운영 중에는 하나의 최신 진행월만 열린다는 기준이 화면에서 먼저 보인다.
- 열린 최신월이 있으면 다음 월 오픈 요청을 UI가 유도하지 않는다.

### 3. 거래/전표 입력

작업:

1. Web 기간 선택 헬퍼를 아래처럼 분리한다.
   - 조회용: 기간 목록 전체 또는 필터 선택
   - 수집거래 쓰기용: 최신 수집 가능월 1개
   - 조정 전표 쓰기용: 최신 전표 입력 가능월 1개
2. 수집 거래 생성/수정 폼의 날짜 검증을 최신 수집 가능월 기준으로 바꾼다.
3. 수집 거래 목록 필터는 과거 기간 조회를 허용하되, 수정/삭제/전표 확정 가능 여부는 API 정책과 같은 기준으로 계산한다.
4. 전표 반전/정정 다이얼로그의 날짜 검증을 최신 전표 입력 가능월 기준으로 바꾼다.
5. legacy 다중 열린 월 데이터가 있을 때 과거 열린월은 읽기 전용 안내를 보여 준다.

완료 기준:

- Web에서 통과한 거래/전표 날짜가 API의 기간 가드에서 다시 막히지 않는다.
- 다중 열린 월이 남아 있어도 쓰기는 최신 진행월 하나로만 유도된다.

### 4. 계획 항목 생성

작업:

1. `plan-items/generate`가 대상 기간이 최신 진행월인지 확인한다.
2. 잠금월뿐 아니라 과거 열린월, 리뷰/마감 진행 중이지만 최신이 아닌 기간에도 생성을 막는다.
3. Web 계획 항목 화면의 생성 대상도 최신 진행월로 고정한다.

완료 기준:

- 반복 규칙 전개와 계획 기반 수집거래 생성이 최신 진행월 밖에서 발생하지 않는다.

진행 기록:

- 완료. API `plan-items/generate`는 대상 기간이 최신 진행월인지 확인하고, 잠금월뿐 아니라 최신이 아닌 열린/검토/마감 진행월도 거절한다.
- 완료. Web 계획 항목 생성 화면은 생성 대상 후보를 최신 진행월 하나로 고정한다.

## Phase 2. 업로드 배치와 신규 계좌/카드 bootstrap 엄격화

### 1. 신규 계좌/카드 판정 모델

작업:

1. `Account`에 bootstrap 상태를 추가하는 방안을 우선 검토한다.
   - 예: `bootstrapStatus = NOT_REQUIRED | PENDING | COMPLETED`
   - 기존 계좌는 migration에서 `COMPLETED` 또는 `NOT_REQUIRED`로 둔다.
   - 새 `BANK`/`CARD`는 기본 `PENDING`으로 만들되, 사용자가 기초 업로드를 건너뛸 수 있게 한다.
2. bootstrap 허용 조건을 명시한다.
   - 계좌 상태가 `ACTIVE`
   - 계좌 유형이 `BANK` 또는 `CARD`
   - bootstrap 상태가 `PENDING`
   - 해당 계좌의 기존 수집거래, import batch, journal line, opening/closing snapshot line이 없음
   - 대상 월이 최초 운영월이거나 최근 잠금월의 바로 다음 월
3. 첫 수집 성공 또는 사용자의 완료 처리로 bootstrap 상태를 `COMPLETED`로 전환한다.

완료 기준:

- "기존 거래가 0건인 오래된 계좌"는 신규 계좌 bootstrap으로 오인되지 않는다.
- 새 계좌/카드 등록 직후 기초 업로드 예외만 명확히 열린다.

진행 기록:

- 완료. `Account.bootstrapStatus`를 `NOT_REQUIRED | PENDING | COMPLETED`로 추가하고, 새 `BANK`/`CARD`는 `PENDING`, `CASH`는 `NOT_REQUIRED`로 생성한다.
- 완료. 신규 계좌/카드 기초 업로드 예외는 `ACTIVE`, `BANK/CARD`, `PENDING`, 기존 수집거래/import batch/전표 line/opening·closing snapshot line 없음, 최근 잠금월의 바로 다음 월 조건을 모두 통과해야 열린다.
- 완료. 첫 수집 성공 또는 기준 데이터 화면의 직접 완료 처리로 `PENDING -> COMPLETED` 전환이 가능하다.

### 2. 업로드 배치 화면

작업:

1. 업로드 배치에서 현재 운영월 collect와 기초 업로드 collect를 모드로 구분한다.
2. 기초 업로드 모드는 대상 계좌가 bootstrap 가능할 때만 노출한다.
3. 자동 운영월 생성 안내에는 `INITIAL_SETUP`과 `NEW_FUNDING_ACCOUNT` 사유를 명확히 표시한다.
4. collect preview와 batch job 결과에도 "최신 진행월 collect"와 "신규 계좌/카드 기초 업로드"를 구분해 보여 준다.

완료 기준:

- 일반 운영 중 업로드는 최신 진행월 안의 거래만 수집한다.
- 새 계좌/카드의 기초 업로드만 예외적으로 다음 월 운영월을 만들 수 있다.

진행 기록:

- 완료. collect preview 응답과 일괄 등록 Job 결과에 `INITIAL_SETUP`과 `NEW_FUNDING_ACCOUNT` 자동 생성 사유를 구분해 표시한다.
- 완료. 업로드 행 등록 화면의 자금수단 선택지와 자동 판정 요약에서 신규 계좌/카드 기초 업로드 대기 상태를 확인할 수 있다.

## Phase 3. 차량 운영비 후속 계획 편입

기존 `VEHICLE_OPERATING_EXPENSE_ACCOUNTING_EXECUTION_PLAN.md`의 1차 구현은 완료된 것으로 보고, 남은 후속 항목은 이 Phase로 편입한다.

### 1. 차량 수집거래 출처 추적

작업:

1. `CollectedTransactionSourceKind`에 `VEHICLE_LOG`를 추가하거나, 별도 `sourceOwner` 구조를 추가한다.
2. 수집거래 read model에 `linkedFuelLogId`, `linkedMaintenanceLogId`, `vehicleId`, `vehicleName` 중 필요한 최소 정보를 포함한다.
3. 목록/상세 UI에서 차량 출처 badge와 원본 차량 로그 이동 링크를 제공한다.
4. 기존 직접 수정/삭제 차단 메시지도 차량 출처와 원본 위치를 함께 안내한다.

완료 기준:

- 차량에서 생성된 거래가 더 이상 수동 거래처럼 보이지 않는다.
- 사용자는 거래 화면에서 원본 차량 로그를 확인할 수 있다.

진행 기록:

- 완료. `CollectedTransactionSourceKind`에 `VEHICLE_LOG`와 `sourceVehicleLog` 원본 정보를 추가했다.
- 완료. 수집 거래 목록/상세에서 차량 연료/정비 출처와 원본 차량 로그 이동 링크를 제공한다.

### 2. 차량 로그 삭제

작업:

1. 연료/정비 로그 삭제 API를 추가한다.
2. 연결 수집거래가 없으면 로그만 삭제한다.
3. 연결 수집거래가 미확정이면 같은 트랜잭션에서 연결 수집거래를 삭제하거나 연결 해제한다.
4. 연결 수집거래가 `POSTED`, `CORRECTED`, `LOCKED`이면 삭제를 거절하고 전표 반전/정정 안내를 제공한다.
5. Web 목록과 상세에서 삭제 버튼, 확인 다이얼로그, 불가 사유를 추가한다.

완료 기준:

- 미확정 차량 비용은 차량 화면에서 정리할 수 있다.
- 확정된 회계 사실은 차량 삭제로 흔들리지 않는다.

진행 기록:

- 완료. 연료/정비 로그 삭제 API를 추가하고 Owner/Manager 권한, 204/404/409 응답 흐름을 고정했다.
- 완료. 연결 수집거래가 없으면 로그만 삭제하고, 미확정 연결 수집거래가 있으면 같은 transaction에서 연결 수집거래를 함께 삭제한다.
- 완료. 연결 수집거래가 전표 확정/정정/잠금 상태이면 삭제를 거절하고 Web에서 전표 반전/정정 안내와 불가 사유를 보여 준다.
- 완료. Web 연료/정비 목록에 삭제 버튼과 확인 다이얼로그를 추가했다.

### 3. 확정 후 보정 편의

작업:

1. 확정된 차량 로그 편집 차단 안내에 전표 상세/반전/정정 이동 링크를 넣는다.
2. 차량 로그에서 연결 전표 번호와 상태를 직접 확인할 수 있게 한다.
3. 회계 조정 완료 후 차량 로그는 과거 사실로 유지한다는 안내를 통일한다.

완료 기준:

- 차량 사용자는 수정 차단 후 어디서 회계 조정을 해야 하는지 즉시 알 수 있다.

진행 기록:

- 완료. 차량 연료/정비 목록의 회계 연동 컬럼에서 연결 전표 번호와 상태를 직접 확인하고 전표 상세로 이동할 수 있게 했다.
- 완료. 확정된 차량 로그 편집 차단 안내를 "차량 로그는 과거 사실로 유지, 회계 조정은 전표 반전/정정" 문구로 통일했다.

### 4. 차량 기본값과 템플릿

작업:

1. 차량별 기본 자금수단/카테고리 프리셋을 추가한다.
2. 연료/정비 입력 시 기본값을 제안하되 사용자가 변경할 수 있게 한다.
3. 보험/차량/정비 운영비 템플릿화는 반복 규칙과 계획 항목 모델을 재사용한다.

완료 기준:

- 반복 입력 비용이 줄고, 표준 회계 분류가 흔들리지 않는다.

진행 기록:

- 완료. `Vehicle`에 기본 자금수단, 연료 기본 카테고리, 정비 기본 카테고리, 차량 운영비 계획 opt-in 필드를 추가했다.
- 완료. 차량 등록/수정 화면에서 기본값을 관리하고, 연료/정비 입력 시 선택 차량의 기본값을 제안하되 사용자가 바꿀 수 있게 했다.
- 완료. 보험/차량/정비 운영비 템플릿화는 새 모델을 만들지 않고 반복 규칙과 계획 항목 모델을 재사용하는 방향으로 유지했다.

### 5. 차량 운영비 계획/백필

작업:

1. 차량 운영비 계획 항목 자동 생성/매칭은 opt-in으로 설계한다.
2. 기존 차량 로그 일괄 백필 도구는 미확정 기간/미연결 로그만 대상으로 제한한다.
3. 백필 결과는 dry-run summary를 먼저 보여 준다.

완료 기준:

- 기존 차량 로그를 무리하게 회계 숫자에 섞지 않고, 사용자가 검토한 범위만 연결한다.

진행 기록:

- 완료. 차량 운영비 계획 자동 생성/매칭의 진입 조건을 차량별 `operatingExpensePlanOptIn`으로 설계하고 프로필 필드에 반영했다.
- 완료. 백필은 확정 전표가 없는 미확정 기간과 미연결 차량 로그만 대상으로 삼고, 실제 반영 전 dry-run summary를 먼저 보여 주는 설계로 고정했다.

## Phase 4. 남은 미완료 범위 정리

### 1. 스캔 PDF import

작업:

1. 현재 텍스트 레이어 없는 PDF는 명시적인 미지원 상태로 유지한다.
2. OCR을 포함할 경우 공급자, 개인정보 보관, 실패율, 비용을 별도 ADR로 결정한다.
3. OCR 미도입 상태라면 import 화면에서 "텍스트 PDF만 지원"을 업로드 전 안내한다.

완료 기준:

- 스캔 PDF 실패가 파싱 오류처럼 보이지 않고, 지원 범위가 화면과 API에서 명확하다.

진행 기록:

- 완료. IM뱅크 PDF 파서는 텍스트 레이어가 없는 스캔/이미지 PDF를 `SCANNED_PDF_TEXT_LAYER_MISSING` 코드로 명시 차단한다.
- 완료. 업로드 화면에는 "텍스트 레이어가 있는 IM뱅크 원본 PDF만 지원" 안내를 추가했고, OCR은 별도 ADR 전까지 미지원으로 문서화했다.

### 2. 외부 감사 저장소와 중앙 로그

작업:

1. workspace 감사 이벤트, period status history, import job 결과, 전표 조정 이벤트를 외부 sink로 내보내는 포트를 정의한다.
2. 로컬 개발에서는 noop 또는 file sink, 운영에서는 central log sink를 붙일 수 있게 한다.
3. 실패해도 핵심 회계 트랜잭션을 깨지 않는 outbox 또는 비동기 전송 방식을 검토한다.

완료 기준:

- 중요한 운영 이벤트가 중앙에서 추적 가능하다.
- 감사 sink 장애가 월 운영 쓰기 흐름을 불필요하게 막지 않는다.

진행 기록:

- 완료. `OperationalAuditSinkPort`와 기본 noop sink, 비동기 `OperationalAuditPublisher`를 추가했다.
- 완료. workspace 감사 이벤트, 기간 상태 이력, 업로드 일괄 등록 Job 완료, 전표 reverse/correct 이벤트가 외부 sink 경계로 발행되도록 연결했다.
- 완료. sink 실패는 경고 로그만 남기고 핵심 회계 트랜잭션을 롤백하지 않도록 테스트로 고정했다.

### 3. Carry-forward journal entry

작업:

1. 이월을 전표로 남길지, opening balance snapshot 전용 산출물로 유지할지 회계 정책을 확정한다.
2. 전표로 남긴다면 `CARRY_FORWARD` source kind와 posting policy를 사용해 `createdJournalEntryId`를 채운다.
3. 스냅샷 전용으로 결정한다면 계약과 문서에서 `createdJournalEntryId` null 가능성을 의도된 설계로 명시한다.
4. 재오픈/이월 취소 시 전표 또는 스냅샷 산출물 정리 순서를 테스트로 고정한다.

완료 기준:

- `createdJournalEntryId`가 null인 상태가 미완료인지 설계인지 더 이상 애매하지 않다.

진행 기록:

- 완료. 현재 정책은 opening balance snapshot 전용 이월로 확정했다.
- 완료. `CarryForwardRecord.createdJournalEntryId`는 의도적으로 `null`이며 별도 `CARRY_FORWARD` 전표를 생성하지 않는다고 계약 주석, API, 도메인 문서에 명시했다.
- 완료. 이월 생성/취소 테스트에서 snapshot 정리와 `CARRY_FORWARD` 전표 미생성을 검증한다.
