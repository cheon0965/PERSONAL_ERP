# 차량 운영비 회계 연계 실행 완료 기록

> 완료 처리: 2026-04-22 기준 1차 구현 범위는 완료된 실행 기록으로 보관한다. 남은 후속 단계는 [`MONTHLY_OPERATION_BUSINESS_LOGIC_ALIGNMENT_EXECUTION_PLAN.md`](./MONTHLY_OPERATION_BUSINESS_LOGIC_ALIGNMENT_EXECUTION_PLAN.md)의 `Phase 3. 차량 운영비 후속 계획 편입`으로 이동했고, 2026-04-23 기준 완료 처리했다.

## 배경

차량의 연료 기록과 정비 이력은 운영 참고 데이터로 시작했지만, 현재 구현에서는 사용자가 회계 연동을 켠 경우 표준 수집거래(`CollectedTransaction`)와 연결된다. 연결된 수집거래를 전표(`JournalEntry`)로 확정하면 차량 운영비는 기존 기간 전망과 재무제표 흐름에 반영된다.

이 문서의 목적은 차량 로그를 기존 회계 파이프라인에 안전하게 연결하는 기준과, 확정 이후 이력 변경으로 회계 숫자가 흔들리지 않도록 하는 관리 규칙을 고정하는 것이다.

## 현재 구현 상태

- Prisma 스키마와 마이그레이션에 연료/정비 로그의 `linkedCollectedTransactionId` 관계를 추가했다.
- 계약 타입과 DTO에 `accountingLink` 요청 구조와 `linkedCollectedTransaction` 응답 요약을 추가했다.
- 차량 서비스/저장소는 연료/정비 저장 시 선택적으로 표준 지출 수집거래를 생성하고, 미확정 연결은 수정/해제 시 함께 동기화한다.
- 차량에서 생성한 연결 수집거래는 거래 화면에서 직접 수정/삭제할 수 없고, 차량 화면에서만 관리한다.
- 연결 수집거래가 `POSTED`, `CORRECTED`, `LOCKED`이면 차량 로그 수정은 거절한다.
- 차량 Web 폼과 목록에는 회계 연동 옵션, 자금수단/카테고리 입력, 연동 상태 표시가 반영되어 있다.

## 목표

1. 연료 기록과 정비 이력을 저장할 때 선택적으로 수집거래를 함께 생성한다.
2. 연결된 수집거래가 확정되면 기존 `수집거래 -> 전표 -> 마감 -> 재무제표/전망` 흐름을 그대로 탄다.
3. 확정 이후에는 차량 로그와 회계 데이터가 서로 어긋나지 않도록 수정 규칙을 엄격하게 적용한다.

## 범위

### 1차 구현 범위

- 연료 기록/정비 기록에 `linkedCollectedTransactionId`를 추가한다.
- 저장 시 사용자가 원하면 지출(`EXPENSE`) 수집거래를 함께 만든다.
- 연결된 수집거래가 아직 미확정이면 차량 로그 수정 시 수집거래도 함께 갱신한다.
- 연결된 수집거래가 아직 미확정이면 차량 로그 수정 화면에서 연동을 끄는 방식으로 연결을 해제할 수 있다.
- 연결된 수집거래가 `POSTED`, `CORRECTED`, `LOCKED` 상태면 차량 로그 수정을 막는다.
- 차량 로그에서 생성된 수집거래는 거래 화면에서 직접 수정/삭제하지 못하게 막고, 차량 화면에서만 관리한다.
- 차량 화면에 회계 연동 상태를 보여준다.

### 이번 단계에서 제외

- 연료/정비 로그 삭제 기능
- 확정된 차량 로그를 자동으로 정정분개까지 이어주는 보정 플로우
- 차량별 기본 자금수단/카테고리 프리셋
- 이미 존재하는 과거 차량 로그 일괄 백필

## 핵심 정책

### 1. 단일 관리면 원칙

연동된 수집거래의 수정/삭제 진입점은 차량 로그 화면 하나로 제한한다. 거래 화면에서 직접 수정되면 차량 로그와 회계 데이터가 쉽게 어긋나므로, 차량 로그에 연결된 수집거래는 거래 화면에서 직접 수정/삭제하지 못하게 막는다.

### 2. 확정 이후 불변 원칙

다음 상태의 연결 수집거래가 있으면 차량 로그 수정 자체를 막는다.

- `POSTED`
- `CORRECTED`
- `LOCKED`

이 경우 사용자는 차량 로그를 과거 사실 기록으로 보존하고, 회계 조정은 기존 전표 정정/반전 절차로 처리한다.

### 3. 미확정 구간 동기화 원칙

연결 수집거래가 아래 상태에 있으면 차량 로그 수정 시 함께 갱신한다.

- `COLLECTED`
- `REVIEWED`
- `READY_TO_POST`

차량 로그에서 회계 연동을 해제하면 연결 수집거래를 삭제한다. 다만 삭제 가능 상태일 때만 허용한다.

### 4. 회계 파이프라인 재사용 원칙

차량 운영비 전용 전표 경로를 새로 만들지 않는다. 차량 로그는 표준 수집거래를 생성하고, 이후 처리도 기존 흐름을 그대로 따른다.

- 차량 로그 저장
- 선택적 수집거래 생성
- 수집거래 확정
- 전표 생성
- 마감 스냅샷 반영
- 재무제표/전망 반영

## 데이터 모델 변경

### 연료 기록

- `FuelLog.linkedCollectedTransactionId: string | null`

### 정비 이력

- `VehicleMaintenanceLog.linkedCollectedTransactionId: string | null`

### 수집거래 역참조

수집거래에서 어떤 차량 로그가 소유자인지 역으로 추적할 수 있도록 1:1 관계를 둔다. 이 관계는 거래 화면 직접 수정/삭제 차단과 무결성 검사에 사용한다.

## 생성/수정/해제 규칙

### 생성

사용자가 회계 연동을 켜고 저장하면 다음을 함께 만든다.

- 차량 로그
- 연결 수집거래

수집거래 기본 규칙은 아래와 같다.

- 유형: `EXPENSE`
- 거래일: 연료 기록 `filledOn`, 정비 기록 `performedOn`
- 금액: 차량 로그의 `amountWon`
- 자금수단: 사용자 선택
- 카테고리: 사용자 선택 또는 미지정
- 제목/메모: 차량명과 로그 세부정보로 자동 생성
- 기간: 거래일이 최신 진행월 범위 안에 있을 때만 연결 수집거래 생성

### 수정

연결 수집거래가 미확정 상태면 아래를 함께 갱신한다.

- 거래일
- 금액
- 자금수단
- 카테고리
- 제목/메모
- 최신 진행월 기준 운영 기간(`periodId`)

연결 수집거래가 확정 상태면 차량 로그 수정은 거절한다.

### 해제

편집 화면에서 회계 연동을 끄면 연결 수집거래를 삭제한다. 다만 아래 조건을 모두 만족해야 한다.

- 연결 수집거래가 존재한다.
- 연결 수집거래가 미확정 상태다.
- 현재 운영 기간 규칙상 삭제 가능한 날짜다.

## 직접 수정 차단 정책

차량 로그에 연결된 수집거래는 거래 화면에서 직접 수정/삭제를 차단한다.

이 정책이 필요한 이유:

- 차량 로그와 수집거래의 금액/일자/분류가 서로 어긋나는 것을 방지
- 사용자가 어느 화면을 기준으로 고쳐야 하는지 명확화
- 확정 이후 조정은 회계 도메인의 정정 절차로 유도

## 재무 반영 방식

차량 로그가 재무제표와 전망에 반영되는 경로는 별도 구현이 아니라 기존 경로 재사용이다.

1. 차량 로그 저장
2. 연결 수집거래 생성
3. 수집거래 확정 시 `JournalEntry` 생성
4. 마감 시 `ClosingSnapshot` 반영
5. 재무제표 생성 시 posted journal lines 반영
6. 월간 전망 조회 시 posted journal lines 반영

즉, 이번 구현 이후 차량 운영비가 재무 숫자에 들어오려면 여전히 "수집거래 확정"은 필요하다.

## 추가 고려사항

### 1. 기간 통제

- 차량 로그에 회계 연동이 켜져 있으면 저장 시점에 해당 일자가 최신 진행월 범위 안인지 확인해야 한다.
- 날짜 수정 시에도 최신 진행월 범위 밖으로 벗어나면 연결 수집거래 동기화를 막고, 범위 안이면 연결 수집거래의 `periodId`도 같이 갱신해야 한다.

### 2. 금액 0 처리

- 운영 참고용 로그는 0원 허용이 가능하지만, 회계 연동 수집거래는 0원일 수 없다.
- 따라서 회계 연동이 켜진 로그는 금액을 양수로 제한한다.

### 3. 카테고리 미지정

- 카테고리를 비워두면 수집거래는 `REVIEWED` 상태로 생성된다.
- 이후 차량 로그 편집에서 카테고리를 보완해 `READY_TO_POST` 상태로 올릴 수 있다.

### 4. 차량명 변경

- 차량명은 운영 기준 데이터다.
- 과거 수집거래 제목은 생성 시점 또는 최신 차량명 기준으로 갱신될 수 있으나, 회계 사실 자체는 금액/일자/분류가 핵심이므로 제목 불일치는 우선순위가 낮다.
- 1차 구현에서는 차량 로그 수정 시점에 현재 차량명을 기준으로 제목을 다시 만든다.

### 5. 계획 항목 매칭

- 차량 로그에서 생성한 수집거래를 이번 단계에서 자동으로 `PlanItem`에 매칭하지 않는다.
- 계획 항목 자동 매칭까지 넣으면 삭제/해제 시 되돌림 규칙과 사용자 기대가 훨씬 복잡해진다.

### 6. 감사와 운영 안내

- 차량 로그 화면에는 "미연결 / 검토됨 / 전표 준비 / 확정" 같은 회계 연동 상태를 바로 보여준다.
- 확정된 연결은 차량 화면에서 더 이상 수정되지 않는다는 안내를 넣는다.

3. 차량 서비스/저장소에 연료/정비 저장과 수집거래 동기화 로직을 넣는다. `완료`

5. 차량 웹 폼에 회계 연동 옵션과 자금수단/카테고리 입력을 추가한다. `완료`
6. 차량 목록 화면에 회계 연동 상태를 노출한다. `완료`
