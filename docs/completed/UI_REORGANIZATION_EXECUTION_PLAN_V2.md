# UI 화면 분리 실행계획

기준일: `2026-04-15`

문서 상태: `completed`

완료일: `2026-04-16`

연결 문서:

- `docs/DEMO_GUIDE.md`
- `docs/completed/phase-1/UI_REORGANIZATION_EXECUTION_PLAN.md` (1차 실행계획)

## 문서 목적

이 문서는 2차 화면 분리 작업을 시작할 때 사용한 기준을, 현재 코드와 문서 기준으로 다시 정리한 현재 기준선이다.
즉, `무엇을 나눌지`를 정하던 초안이 아니라 `무엇이 실제로 반영되었고 앞으로도 어떤 원칙으로 유지할지`를 기록하는 문서로 본다.

1차 UI 재정리 이후 전역 셸, 사이드바 그룹, 공통 헤더, 운영 지원/관리자/설정/기준 데이터의 하위 화면 구조는 많이 안정됐다.
하지만 실제 운영 화면 기준으로 다시 보면, 일부 화면은 아직도 `한 화면 안에 2개 이상 작업 흐름`이 들어 있고, 몇몇 화면은 `너무 긴 단일 페이지` 또는 `탭에 과도하게 의존하는 구조`를 가지고 있다.

이번 계획의 목적은 새 기능을 추가하는 것이 아니라 아래 두 가지를 확실하게 만드는 것이다.

- 사용자가 첫 진입 5초 안에 `이 화면의 1순위 작업`을 이해할 것
- 한 화면에서 `입력`, `검토`, `이력`, `분석`, `관리`가 동시에 경쟁하지 않게 만들 것

즉, 이번 문서는 `화면 예쁘게 다듬기`가 아니라 `화면 책임 분리와 라우트 재조정`에 초점을 둔다.

## 적용 상태 요약

- 라우트 분리 대상이던 `월 운영`, `업로드 배치`, `전표 조회`, `차량 운영`, `재무제표`, `차기 이월`은 모두 현재 하위 라우트 기준으로 반영되었다.
- 책임이 완전히 분리된 하위 화면은 탭 안에만 두지 않고, DB 메뉴 트리 `WorkspaceNavigationMenuItem` / `WorkspaceNavigationMenuRole`와 `/navigation/tree` 기준으로 사이드바 보조 메뉴에도 노출된다.
- 공통 UI는 `PageHeader`, `SectionTabs`, `DataTableCard`, `SectionCard`, `Topbar` 기준으로 정리되었고, 긴 설명은 화면 본문보다 앞세우지 않고 `도메인 가이드`로 이동하는 방향을 따른다.
- 현재 기준 문서인 `docs/DEMO_GUIDE.md`, `docs/API.md`, `docs/DESIGN_SYSTEM.md`, `docs/VALIDATION_NOTES.md`는 이 구조를 기준으로 유지한다.

## 이번 검토에서 본 기준

이번 계획은 아래 기준으로 전체 화면을 다시 살폈다.

- 현재 운영 흐름: `docs/DEMO_GUIDE.md`
- 현재 내비게이션: DB 메뉴 트리 `WorkspaceNavigationMenuItem` / `WorkspaceNavigationMenuRole`와 `/navigation/tree`
- 실제 화면 구현: `apps/web/src/features/*`
- 이미 완료된 1차 UI 정리 기준: `docs/completed/UI_REORGANIZATION_EXECUTION_PLAN.md`

### 분리 판단 기준

아래 중 2개 이상에 해당하면 `라우트 분리 우선 후보`로 본다.

1. 한 화면에 서로 다른 작업 단계가 3개 이상 같이 있다.
2. 탭이 4개 이상이거나, 탭이 없더라도 사실상 별도 하위 화면처럼 읽힌다.
3. 단일 페이지 파일이 대략 `550 lines` 이상이거나, 같은 화면군 프런트 표면이 `1500 lines` 이상으로 커졌다.
4. `목록`, `상세`, `생성/수정`, `이력/분석`이 한 화면에서 모두 경쟁한다.
5. 상단 설명과 상태 카드가 길어져 실제 작업 표나 폼이 첫 시야의 중심이 되지 못한다.

### 판단 레벨

- `유지`: 현재 라우트 구조를 유지하고 내부 컴포넌트만 정리한다.
- `내부 분리`: 라우트는 유지하되 카드/패널/폼/표를 재배치하고 파일을 더 쪼갠다.
- `라우트 분리`: 같은 기능군 안에서 하위 화면 또는 상세 라우트를 만든다.

## 이번 검토 결론

전체 화면을 다시 보면 결론은 단순하다.

1. `운영 지원`, `관리자`, `설정`, `기준 데이터`는 이미 하위 화면 분리가 잘 되어 있어서 2차 분리의 주 대상이 아니다.
2. 실제 분리 우선순위는 `월 운영`, `차량 운영`, `전표 조회`, `업로드 배치`, `재무제표`, `차기 이월`에 집중해야 한다.
3. `반복 규칙`, `보험 계약`, `계획 항목`, `수집 거래`, `기간 전망`, `대시보드`는 당장 라우트를 늘리기보다 `내부 책임 경계`를 먼저 더 분명하게 만드는 편이 낫다.
4. 인증/온보딩 화면은 길이가 다소 있더라도 다단 작업 화면이 아니라서 이번 분리 작업의 우선순위는 낮다.

## 목표 정보구조

이번 2차 분리에서 중요한 것은 `메뉴를 늘리는 것`이 아니라 `화면 책임을 나누는 것`이다.

### 전역 규칙

1. 사이드바 1차 메뉴는 지금 수준을 크게 늘리지 않는다.
2. 사이드바 구조는 프런트 하드코딩이 아니라 DB에 저장된 트리 메뉴를 기준으로 한다.
3. 자주 오가는 동급 화면만 사이드바 또는 `SectionTabs`에 노출한다.
4. 하위 라우트로 분리된 화면 중 책임이 완전히 독립된 작업 화면은 탭 안에만 두지 않고 사이드바 보조 메뉴로도 노출한다.
5. 메뉴별 노출 권한은 `OWNER / MANAGER / EDITOR / VIEWER` 역할 단위로 관리한다.
6. 선택한 항목의 상세, 생성, 결과 확인은 `컨텍스트 하위 라우트`로 보낸다.
7. 쓰기 화면은 `작업대 -> 표/폼 -> 결과` 순서를 지킨다.
8. 읽기 화면은 `기준 선택 -> 요약 -> 상세 본문` 순서를 지킨다.
9. 긴 설명은 화면 첫 시야를 차지하지 않도록 `PageHeader`보다 `도메인 가이드`와 보조 섹션으로 보낸다.

### 화면 분리 방식

- `허브형`: 현재 상태와 빠른 진입점만 보여준다.
- `목록형`: 표 또는 카드 목록이 주인공이 된다.
- `상세형`: 선택한 단일 대상의 상세와 후속 액션만 다룬다.
- `작업형`: 생성, 마감, 승격, 조정처럼 한 번의 의도적인 작업만 수행한다.

## 기능별 목표 화면 구조

### 월 운영

- `/periods`
  현재 운영 월 상태, readiness, 다음 작업만 보여주는 홈
- `/periods/open`
  새 운영 월 시작과 기초 잔액 입력 전용
- `/periods/close`
  월 마감, 차단 사유, 마감 전 점검 전용
- `/periods/history`
  기간 이력, 잠금 월, 재오픈 진입점 전용

`재오픈`은 빈도가 낮으므로 1차 메뉴가 아니라 `history` 또는 `close` 안의 문맥 액션으로 두는 것이 낫다.

### 차량 운영

- `/vehicles`
  운영 개요, 비용 차트, 최근 기록만 보여주는 홈
- `/vehicles/fleet`
  차량 마스터 목록과 차량 등록/수정
- `/vehicles/fuel`
  주유/충전 기록 목록과 등록/수정
- `/vehicles/maintenance`
  정비 기록 목록과 등록/수정

현재 `overview` 탭의 차트와 요약 카드는 유지하되, 실제 CRUD 목록은 각각의 하위 화면으로 보내야 한다.

### 업로드 배치

- `/imports`
  배치 목록, 현재 운영 월, readiness, 신규 업로드 진입점
- `/imports/[batchId]`
  선택 배치의 행 목록, 승격 준비, 승격 실행

이렇게 나누면 `배치 선택`과 `행 처리`가 서로 시선을 뺏지 않는다.

### 전표 조회

- `/journal-entries`
  전표 목록과 필터
- `/journal-entries/[entryId]`
  선택 전표 상세, 라인, 계보, 관련 거래 링크
- 조정 액션
  상세 화면 안에서만 반전/정정 전표 진입

현재 쿼리스트링 기반 선택 방식은 초기 호환용으로 유지하고, 점진적으로 상세 라우트 링크로 바꾸는 것이 좋다.

### 재무제표 / 차기 이월

- `생성 화면`
  기간 선택, 생성 권한, 생성 버튼, 직전 기준선 안내
- `결과 화면`
  생성된 스냅샷 또는 이월 결과 상세

지금은 생성과 결과 해석이 한 화면에 같이 있어서, 사용자는 무엇이 `입력`이고 무엇이 `결과`인지 구분하는 데 시간을 쓴다.

## 단계별 실행 결과

아래 단계는 모두 현재 코드와 현재 기준 문서에 반영된 상태다.

### Phase 0. 기준선 고정

- 이 문서를 2차 화면 분리 기준선으로 확정했다.
- 화면별 deep link, 현재 라우트, 새 라우트 초안을 표로 고정했다.
- 기존 링크 호환 방식을 먼저 정했다.
  - 예: `?entryId=...` 호환을 유지하면서 새 상세 라우트와 병행했다.

### Phase 1. 공통 라우트 뼈대 준비

- 화면군 내부 하위 라우트를 위한 공통 레이아웃과 `SectionTabs` 패턴을 반영했다.
- 상위 라우트에서 쓰는 보조 내비게이션 패턴을 통일했다.
- `상세 화면 뒤로가기`, `현재 선택 대상`, `상위 화면 복귀` 흐름을 feature 공통 패턴으로 맞췄다.

### Phase 2. P0 쓰기 화면 분리

우선순위:

1. `월 운영`
2. `업로드 배치`
3. `전표 조회`
4. `차량 운영`

현재 상태:

- `월 운영`, `업로드 배치`, `전표 조회`, `차량 운영`은 모두 하위 라우트 중심 구조로 분리되었다.
- 실제 사용 흐름도 `열기 -> 수집 -> 확정 -> 보조 운영` 순서가 화면 구조에 드러나도록 정리되었다.

### Phase 3. P1 기준/연결 화면 정리

- `보험 계약`
- `반복 규칙`
- `계획 항목`
- `내 계정 / 보안`
- `관리자 - 멤버`

현재 상태:

- `보험 계약`, `반복 규칙`, `계획 항목`, `내 계정 / 보안`, `관리자 - 멤버`는 라우트를 불필요하게 늘리지 않고 내부 책임 경계를 더 선명하게 정리했다.
- 화면 설명 밀도는 줄이고, 긴 안내는 `도메인 가이드`와 보조 패널로 이동했다.

### Phase 4. P1 보고 화면 분리

- `재무제표`
- `차기 이월`
- `기간 전망`
- `대시보드`

현재 상태:

- `재무제표`, `차기 이월`은 생성/선택 화면과 결과 상세 화면으로 분리되었다.
- `기간 전망`, `대시보드`는 라우트를 유지하되 `요약 -> 드라이버/체크포인트 -> 해석` 구조로 재배치되었다.

### Phase 5. 문서/테스트 동기화

- `docs/DEMO_GUIDE.md`의 흐름과 새 라우트를 맞췄다.
- DB 기본 메뉴 트리, 사이드바, 로컬 내비게이션 문구를 동기화했다.
- `docs:check:surface` 기준으로 현재 Web/API surface와 기준 문서를 함께 유지한다.

## 검증 기준

현재 구현은 아래 기준을 충족하도록 정리되어 있으며, 이후 UI 변경도 같은 기준으로 판단한다.

아래 조건을 만족하면 이번 2차 분리 작업이 제대로 끝난 것으로 본다.

1. 한 화면의 첫 시야에 `핵심 작업`이 하나만 보인다.
2. 로컬 탭이 `4개 이상` 필요한 화면은 하위 라우트로 내려간다.
3. 상세를 보기 위해 목록과 상세를 같은 화면에서 억지로 동시에 유지하지 않는다.
4. 쓰기 화면은 표나 폼이, 읽기 화면은 기준선과 결과가 먼저 보인다.
5. 기존 deep link가 새 구조에서도 끊기지 않는다.
6. `계획 항목 -> 수집 거래 -> 전표`, `월 운영 -> 재무제표 -> 차기 이월 -> 전망` 흐름이 화면만 봐도 자연스럽게 이어진다.

## 이번 작업에서 의도적으로 제외하는 것

- 도메인 규칙 자체 변경
- 새 회계 기능 추가
- 사이드바 1차 메뉴 대폭 확장
- 전면적인 브랜딩 리디자인

이번 계획의 핵심은 `이미 있는 기능을 더 짧은 화면 책임으로 다시 배치하는 것`이다.

## 현재 유지 원칙

1. 새 복합 화면을 추가할 때는 먼저 `독립 라우트`, `SectionTabs`, `사이드바 보조 메뉴` 중 어떤 책임 분리가 맞는지부터 판단한다.
2. `PageHeader`는 짧게 유지하고, 긴 도메인 설명은 `도메인 가이드` 또는 별도 문서로 보낸다.
3. 메뉴 구조나 화면 경로가 바뀌면 `DB 기본 메뉴 트리`, `docs/DEMO_GUIDE.md`, `docs/API.md`, `docs/VALIDATION_NOTES.md`를 같은 변경에서 함께 맞춘다.
