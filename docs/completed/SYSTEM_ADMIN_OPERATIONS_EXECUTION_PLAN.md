# 전체 관리자 운영 필수 기능 실행 계획

완료 상태: `2026-04-19` 기준으로 전체 사용자 관리, 사업장 관리, 지원 모드, 보안 위협 로그, 운영 상태, 관련 문서/검증 반영까지 마무리되어 완료 문서로 이관합니다.

## 목적

이 문서는 개인 프로젝트 수준의 `PERSONAL_ERP`에서 전체 관리자가 갖춰야 할 최소 운영 기능을 구현하기 위한 실행 계획이다.

전체 관리자 기능의 목표는 대형 SaaS 운영센터를 만드는 것이 아니다.
이 프로젝트의 핵심 방향은 여전히 `1인 사업자와 소상공인의 월별 재무 운영 사이클을 끝까지 닫는 것`이다.
따라서 전체 관리자 기능은 본 업무 흐름을 대체하지 않고, 문제가 생겼을 때 개발자 겸 운영자가 빠르게 확인하고 복구하는 보조 도구로 제한한다.

## 배경

현재 코드베이스는 이미 아래 기반을 갖추고 있다.

- `INITIAL_ADMIN_*` seed 기반 전체 관리자 계정
- 전체 관리자 전용 플랫폼 메뉴
- 전체 사업장/사용자/멤버 조회
- 멤버 초대, 역할 변경, 상태 변경, 제거
- 메뉴 / 권한 관리와 권한 정책 요약
- 전체 감사 로그
- 별도 보안 위협 로그
- 사용자 본인 계정의 프로필, 비밀번호, 세션, 보안 이벤트 화면
- 운영 지원 영역의 시스템 상태, 체크리스트, 알림, 메모, 내보내기 화면

다만 개인 프로젝트를 실제로 운영해 보려면 다음 네 가지가 아직 가장 큰 공백으로 남아 있다.

1. 전체 사용자 관리 강화
2. 사업장 관리 화면
3. 전체 관리자 사업장 전환 화면
4. 운영 상태 화면

## 적용 원칙

### 개인 프로젝트에 맞춘 범위

- 한 명 또는 소수 운영자가 직접 문제를 확인하고 복구할 수 있으면 충분하다.
- 복잡한 승인 워크플로우, 티켓 시스템, 감사 보존 정책 UI는 만들지 않는다.
- 운영자가 원인을 빨리 찾고 안전하게 되돌릴 수 있는 기능을 먼저 만든다.
- 보안상 위험한 기능은 편의보다 안전을 우선한다.

### 제품 방향성과의 연결

- 일반 사용자의 월별 재무 운영 화면은 최대한 건드리지 않는다.
- 전체 관리자 화면은 `플랫폼 관리` 메뉴 아래에 분리한다.
- 사업장 운영 메뉴는 선택한 지원 문맥이 있을 때만 열리도록 한다.
- 모든 전체 관리자 쓰기 작업은 감사 로그에 남긴다.
- 사용자 개인정보와 보안 정보는 필요한 만큼만 노출한다.

### 구현 기준

- API는 `apps/api/src/modules/admin` 안에서 시작한다.
- Web은 `apps/web/src/features/admin` 아래에 전체 관리자 전용 화면으로 분리한다.
- 계약 타입은 `packages/contracts/src/admin.ts`에 먼저 정의한다.
- 새 DB 변경이 필요하면 migration-first로 진행한다.
- 새 화면은 화면 도움말 3탭 기준을 유지한다.
- 모든 파일 조회/추가/수정/삭제는 UTF-8 기준으로 진행한다.

## 최종 목표 화면

### 전체 관리자 메뉴

플랫폼 관리 메뉴는 아래 구조를 목표로 한다.

- 관리자 허브
- 전체 사용자 관리
- 사업장 관리
- 사업장 전환 / 지원 모드
- 보안 위협 로그
- 로그관리
- 운영 상태
- 메뉴 / 권한 관리
- 권한 정책 요약

`메뉴 / 권한 관리`와 `권한 정책 요약`은 선택한 사업장 문맥이 있을 때만 현재처럼 사업장 기준으로 동작한다.

### 관리자 허브

관리자 허브는 모든 기능을 길게 설명하지 않고 아래만 요약한다.

- 운영 위험: 높은 보안 위협, 최근 실패 로그, 준비 안 된 사업장 수
- 계정 위험: 잠금 사용자, 미인증 사용자, 활성 세션 많은 사용자
- 사업장 위험: 중지/아카이브 사업장, 기본 장부 누락 사업장
- 바로 가기: 사용자 관리, 사업장 관리, 지원 모드, 운영 상태

## Phase 1. 전체 사용자 관리 강화

### 목표

전체 관리자가 사용자 계정 문제를 안전하게 확인하고, 최소한의 복구 작업을 수행할 수 있게 한다.

### 필요한 데이터 모델

현재 `User`에는 `isSystemAdmin`, `emailVerifiedAt`, `createdAt`은 있지만 계정 상태 필드가 없다.

추가 후보:

- `UserStatus`
  - `ACTIVE`
  - `LOCKED`
  - `DISABLED`
- `User.lockedReason`
- `User.lockedAt`

개인 프로젝트 수준에서는 `DISABLED`까지 필요하지 않다면 `LOCKED`만으로 시작해도 된다.
다만 로그인 차단과 운영 상태 표시를 명확히 하려면 enum을 두는 편이 좋다.

### API 계획

- `GET /admin/users`
  - 현재 목록 API를 확장한다.
  - 세션 수, 활성 세션 수, 멤버십 수, 이메일 인증 여부, 계정 상태를 포함한다.
- `GET /admin/users/:userId`
  - 사용자 상세, 멤버십 목록, 최근 세션, 최근 보안 위협 이벤트 요약을 반환한다.
- `PATCH /admin/users/:userId/status`
  - `ACTIVE`, `LOCKED`, `DISABLED` 중 허용 범위로 상태를 변경한다.
  - 자기 자신 잠금은 차단한다.
  - 마지막 전체 관리자 잠금은 차단한다.
- `POST /admin/users/:userId/revoke-sessions`
  - 해당 사용자의 모든 활성 세션을 만료한다.
  - 현재 전체 관리자 본인의 현재 세션은 기본적으로 제외하거나 별도 확인을 요구한다.
- `PATCH /admin/users/:userId/system-admin`
  - 전체 관리자 권한 부여/회수.
  - 마지막 전체 관리자 회수는 차단한다.
- `PATCH /admin/users/:userId/email-verification`
  - 운영 보정용으로 이메일 인증 상태를 수동 확인 처리할 수 있게 한다.
  - 인증 해제는 기본 범위에서 제외한다.

### Web 계획

새 화면:

- `/admin/users`
- 화면명: `전체 사용자 관리`

주요 영역:

- 요약 카드
  - 전체 사용자
  - 미인증 사용자
  - 잠금 사용자
  - 전체 관리자
  - 활성 세션
- 사용자 목록
  - 이메일
  - 이름
  - 상태
  - 이메일 인증
  - 전체 관리자 여부
  - 멤버십 수
  - 활성 세션 수
  - 생성일
- 상세 Drawer
  - 기본 정보
  - 멤버십 목록
  - 세션 요약
  - 최근 보안 위협 이벤트
  - 작업 버튼

작업 버튼:

- 계정 잠금
- 잠금 해제
- 모든 세션 만료
- 이메일 인증 처리
- 전체 관리자 권한 부여/회수

### 감사 로그

모든 쓰기 작업은 `WorkspaceAuditEvent`가 아니라 platform-level 성격이므로 현재는 `SecurityEventLogger.log('audit.action_succeeded')` 패턴으로 남긴다.
추후 필요하면 `PlatformAuditEvent` 모델로 분리한다.

필수 이벤트:

- `admin_user.status_updated`
- `admin_user.sessions_revoked`
- `admin_user.email_marked_verified`
- `admin_user.system_admin_granted`
- `admin_user.system_admin_revoked`

### 검증 기준

- 일반 사용자는 전체 사용자 API에 접근할 수 없다.
- 전체 관리자는 사용자 목록과 상세를 조회할 수 있다.
- 잠금된 사용자는 로그인할 수 없다.
- 전체 관리자가 특정 사용자 세션을 강제 만료하면 refresh가 실패한다.
- 마지막 전체 관리자는 잠금/권한 회수할 수 없다.
- 모든 쓰기 작업은 감사 이벤트로 추적된다.

## Phase 2. 사업장 관리 화면

### 목표

전체 관리자가 사업장 상태와 기본 장부 문제를 확인하고, 운영에 필요한 최소 상태 변경을 수행할 수 있게 한다.

### 필요한 데이터 모델

현재 `Tenant.status`는 이미 존재한다.

상태 후보:

- `ACTIVE`
- `TRIAL`
- `SUSPENDED`
- `ARCHIVED`

새 모델 추가 없이 시작한다.
필요하면 나중에 `TenantAdminNote` 또는 `PlatformAuditEvent`를 추가한다.

### API 계획

- `GET /admin/tenants`
  - 현재 목록 API를 확장한다.
  - 기본 장부, 멤버 수, 활성 멤버 수, 소유자 수, 최근 접속, 최근 감사 이벤트 요약을 포함한다.
- `GET /admin/tenants/:tenantId`
  - 사업장 상세, 기본 장부, 장부 목록, 멤버 요약, 최근 감사 로그, 최근 보안 위협 요약을 반환한다.
- `PATCH /admin/tenants/:tenantId/status`
  - 사업장 상태를 변경한다.
  - 기본 장부가 없는 사업장은 `ACTIVE` 전환을 차단하거나 경고한다.
- `PATCH /admin/tenants/:tenantId/default-ledger`
  - 기본 장부를 변경한다.
  - 개인 프로젝트에서는 다중 장부 관리가 핵심이 아니므로 필요 시에만 구현한다.

### Web 계획

새 화면:

- `/admin/tenants`
- 화면명: `사업장 관리`

주요 영역:

- 요약 카드
  - 전체 사업장
  - 활성 사업장
  - 중지 사업장
  - 기본 장부 누락
  - 소유자 없는 사업장
- 사업장 목록
  - 이름
  - 슬러그
  - 상태
  - 기본 장부
  - 활성 멤버 수
  - 소유자 수
  - 최근 변경일
- 상세 Drawer
  - 사업장 기본 정보
  - 기본 장부
  - 멤버 요약
  - 최근 감사 로그
  - 지원 모드 진입 버튼

작업 버튼:

- 상태 변경
- 지원 모드로 열기
- 감사 로그로 보기

### 접근 제어 영향

사업장 `SUSPENDED` 또는 `ARCHIVED` 상태를 실제 업무 화면에서 어떻게 다룰지 결정해야 한다.

기본 정책:

- 일반 사용자는 `ACTIVE`, `TRIAL` 사업장만 업무 화면 접근 가능.
- 전체 관리자는 지원 모드에서 `SUSPENDED` 사업장을 읽을 수 있다.
- `ARCHIVED` 사업장은 원칙적으로 읽기 전용.

### 검증 기준

- 전체 관리자만 사업장 목록과 상세를 볼 수 있다.
- 사업장 상태 변경은 감사 이벤트로 남는다.
- `SUSPENDED` 사업장의 일반 사용자 업무 접근 정책이 일관되게 동작한다.
- 기본 장부가 없는 사업장은 지원 모드 진입 시 명확한 안내를 표시한다.

## Phase 3. 전체 관리자 사업장 전환 화면

### 목표

전체 관리자가 문제를 재현하거나 운영 확인을 위해 특정 사업장 문맥을 명시적으로 선택할 수 있게 한다.

이 기능은 완전한 impersonation이 아니다.
전체 관리자가 다른 사용자로 가장하는 것이 아니라, `SYSTEM_ADMIN` 권한으로 특정 사업장과 장부를 지원 문맥으로 선택하는 것이다.

### 현재 구조와 갭

현재 인증 문맥은 활성 멤버십 중 역할 우선순위와 가입일로 `currentWorkspace`를 선택한다.
전체 관리자도 사업장 멤버십이 있으면 일반 업무 메뉴를 함께 볼 수 있다.

하지만 운영 관점에서는 아래 문제가 있다.

- 어떤 사업장 문맥을 보고 있는지 명시적으로 선택하기 어렵다.
- 전체 관리자가 멤버십이 없는 사업장을 지원하기 어렵다.
- 사업장 전환이 로컬 UI 상태인지 서버 상태인지 기준이 없다.

### 데이터 모델 선택

개인 프로젝트 기준 추천은 `AuthSession`에 지원 문맥을 저장하는 것이다.

추가 후보:

- `AuthSession.supportTenantId`
- `AuthSession.supportLedgerId`
- `AuthSession.supportStartedAt`

이 방식의 장점:

- 브라우저를 새로고침해도 지원 문맥이 유지된다.
- 세션을 만료하면 지원 문맥도 함께 사라진다.
- 사용자 전역 설정보다 임시 운영 모드 성격에 맞다.

대안:

- `UserSetting`에 마지막 선택 사업장 저장
- Web localStorage에만 저장

대안은 구현은 쉽지만 서버 접근통제 기준과 엇갈릴 수 있으므로 후순위로 둔다.

### API 계획

- `GET /admin/support-context`
  - 현재 세션의 지원 문맥을 반환한다.
- `POST /admin/support-context`
  - `tenantId`, `ledgerId`를 받아 현재 세션 지원 문맥으로 설정한다.
  - 전체 관리자만 가능하다.
- `DELETE /admin/support-context`
  - 지원 문맥을 해제한다.

인증 문맥 반영:

- `AuthenticatedWorkspaceResolver`는 전체 관리자이고 세션에 지원 문맥이 있으면 해당 tenant/ledger를 `currentWorkspace`로 반환한다.
- 반환된 membership은 실제 멤버십이 아니라 `SYSTEM_ADMIN` 지원 문맥으로 표시한다.
- 업무 API의 `RequiredWorkspaceContext.systemRole`은 `SYSTEM_ADMIN`으로 유지한다.

### Web 계획

새 화면:

- `/admin/support-context`
- 화면명: `사업장 전환 / 지원 모드`

주요 영역:

- 현재 지원 문맥
- 사업장 검색/선택
- 장부 선택
- 지원 모드 시작
- 지원 모드 해제
- 선택한 사업장의 핵심 상태 요약

상단바/사이드바 반영:

- 지원 모드가 켜져 있으면 상단바에 작게 `지원 모드` 배지를 표시한다.
- 사업장 운영 메뉴는 지원 문맥이 있을 때 표시한다.
- 지원 문맥 없음 안내 행은 유지하되 설명은 짧게 한다.

### 안전 기준

- 일반 사용자로 가장하지 않는다.
- 모든 지원 모드 시작/해제는 감사 로그에 남긴다.
- 지원 모드에서 수행한 쓰기 작업은 actorRole을 `SYSTEM_ADMIN`으로 남긴다.
- 지원 모드 해제 버튼은 항상 쉽게 접근 가능해야 한다.

### 검증 기준

- 전체 관리자가 지원 문맥을 선택하면 `/auth/me`의 currentWorkspace가 해당 사업장으로 바뀐다.
- 새로고침 후에도 같은 세션에서는 지원 문맥이 유지된다.
- 지원 모드 해제 후 사업장 운영 메뉴가 사라지거나 기본 문맥으로 돌아간다.
- 일반 사용자는 지원 문맥 API를 사용할 수 없다.

## Phase 4. 운영 상태 화면

### 목표

전체 관리자가 프로젝트 운영 상태를 한 화면에서 확인하고, 장애 원인을 빠르게 좁힐 수 있게 한다.

이미 사업장 운영 메뉴에 `운영 지원 > 시스템 상태`가 있다.
이번 계획의 운영 상태 화면은 전체 관리자용 플랫폼 상태 화면으로 분리한다.

### API 계획

- `GET /admin/operations/status`
  - 전체 관리자 전용
  - API 상태
  - DB 연결 상태
  - Prisma migration 상태 요약
  - 메일 설정 상태
  - 최근 감사 실패/거부 이벤트
  - 최근 보안 위협 이벤트
  - 최근 로그인 실패/가입 제한 이벤트

가능하면 기존 `GET /health/ready` 결과를 내부에서 재사용한다.
Prisma migration 상태는 운영 DB에 직접 위험한 명령을 실행하지 말고, 현재 적용된 migration 테이블 조회 또는 제한된 read-only 점검으로 시작한다.

### Web 계획

새 화면:

- `/admin/operations`
- 화면명: `운영 상태`

주요 영역:

- 상태 카드
  - API
  - DB
  - Prisma migration
  - 메일 발송 설정
  - 보안 위협
  - 감사 실패/거부
- 최근 이벤트
  - 최근 보안 위협 로그
  - 최근 실패/거부 감사 로그
  - 최근 메일 발송 실패
- 바로 가기
  - 보안 위협 로그
  - 감사 로그
  - 사업장 관리
  - 전체 사용자 관리

### 개인 프로젝트 기준 운영 상태

운영 상태 화면은 `모든 문제를 자동 해결하는 화면`이 아니다.
아래 질문에 1분 안에 답할 수 있으면 충분하다.

- API가 살아 있는가?
- DB 연결이 되는가?
- migration이 어긋나 보이는가?
- 메일 설정이 비어 있거나 실패 중인가?
- 최근 보안 위협이 급증했는가?
- 특정 요청번호로 감사 로그를 따라갈 수 있는가?

### 검증 기준

- 전체 관리자만 운영 상태 API에 접근할 수 있다.
- DB가 정상일 때 상태가 `ok`로 표시된다.
- DB 점검 실패 시 화면이 전체 실패하지 않고 문제 카드만 경고로 표시된다.
- 보안 위협과 감사 실패 요약이 각 로그 화면으로 이어진다.

## 리스크와 대응

### 리스크 1. 전체 관리자 권한이 너무 강해짐

대응:

- 모든 쓰기 작업 감사 로그 남김
- 자기 자신 잠금/권한 회수 제한
- 마지막 전체 관리자 보호
- 관리자 직접 비밀번호 변경 제외

### 리스크 2. 지원 모드가 impersonation처럼 오해됨

대응:

- 화면과 로그에서 `지원 모드`, `SYSTEM_ADMIN`을 명확히 표시
- 다른 사용자 ID로 가장하지 않음
- actorUserId는 실제 전체 관리자 ID 유지

### 리스크 3. 사업장 상태 변경이 일반 업무 API와 충돌

대응:

- 상태별 접근 정책을 먼저 문서화
- `AuthenticatedWorkspaceResolver`와 guard 기준을 함께 수정
- SUSPENDED/ARCHIVED 상태의 쓰기 차단 테스트 추가

### 리스크 4. 운영 상태 화면이 과도하게 커짐

대응:

- 처음에는 read-only 요약만 제공
- 자동 복구 버튼은 추가하지 않음
- 상세 분석은 로그 화면으로 연결
