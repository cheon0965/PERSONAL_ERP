## 문서 목적

이 문서는 현재 프론트 화면을 다시 정리할 때 사용할 `실행 기준선`이다.

이번 작업의 방향은 두 가지다.

1. `꼭 한 화면이어야 하는 경우`를 제외하면, 하나의 화면 안에 함께 들어 있는 다른 책임을 별도 화면으로 분리한다.
2. 화면과 메뉴에 붙어 있는 설명형 문구를 걷어내고, `실제 메뉴명`, `실제 기능명`, `핵심 상태값`만 남긴다.

즉, 이번 계획은 단순한 문구 수정이 아니라 `화면 책임 분리`와 `표면 텍스트 슬림화`를 동시에 다룬다.

## 이번 작업의 확정 원칙

### 1. 화면 분리 원칙

아래 중 하나라도 명확하면 별도 화면으로 분리한다.

- 사용자가 서로 다른 목적의 작업을 한 화면에서 수행해야 한다.
- 같은 화면 안에서 `생성`, `수정`, `상태 추적`, `이력 조회`, `보안 관리`, `기준값 관리`가 함께 경쟁한다.
- 한 화면 안의 내부 섹션이 사실상 사이드바 메뉴 한 개씩에 해당하는 독립 기능이다.
- 나중에 메뉴명으로 바로 열어도 어색하지 않은 영역이 한 화면 안에 2개 이상 있다.

반대로 아래에 해당하면 한 화면 유지 대상으로 본다.

- 같은 엔터티의 단일 CRUD 작업을 보조 패널과 함께 처리하는 경우
- 허브 화면으로서 `요약 + 바로가기`만 제공하는 경우
- 상세 화면에서 같은 대상의 보조 액션만 이어지는 경우

### 2. 설명문구 정리 원칙

기본적으로 아래 텍스트는 제거 대상이다.

- `PageHeader.description`
- `SectionCard.description`
- `SummaryCard.subtitle`
- 사이드바와 관리자 메뉴 트리의 메뉴 설명
- 화면 상단의 “이 화면은 무엇을 하는지”를 길게 풀어 쓰는 안내 문구
- 같은 내용을 카드마다 반복 설명하는 보조 문장

다만 아래는 제거 대상이 아니다.

- 검증 에러 메시지
- 입력 필드의 필수 안내와 파괴적 작업 경고
- 빈 상태와 오류 상태를 설명하는 최소 문구
- 감사 로그, 메모, 차량 정비 내용처럼 도메인 데이터 자체의 설명 필드
- 도메인 설명을 옮겨 둘 `도메인 가이드` 내용

### 한 화면 유지 대상

아래는 현재 기준으로 `한 기능 화면`에 더 가깝기 때문에 라우트 분리보다 내부 정리와 설명 제거만 수행한다.

- `수집 거래`
- `반복 규칙`
- `보험 계약`
- `운영 지원 허브`
- `관리자 허브`
- `대시보드`
- `기간 전망`
- `사업장 설정`

이 화면들은 설명형 문구를 걷어내고, 카드 수를 줄이거나 버튼과 표 중심으로 재배치한다.

## 허브 화면 정리 방식

`운영 지원`, `관리자`, `대시보드`, `기간 전망`은 별도 기능 화면을 묶어 주는 허브 성격이 강하므로 유지한다.

대신 아래 기준으로 화면을 얇게 만든다.

- 헤더 설명 문단 제거
- 그룹 설명 문장 제거
- 카드 설명 제거
- 남는 정보는 `숫자`, `상태`, `버튼`, `링크` 중심으로 재배치
- 긴 배경 설명은 `도메인 가이드`로 이동

즉, 허브 화면은 `설명 화면`이 아니라 `현재 상태 + 다음 이동` 화면으로 고정한다.

## 단계별 실행 계획

### Phase 0. 기준선 고정

- 이 문서를 현재 기준 실행계획으로 추가한다.
- 분리 대상, 유지 대상, 설명문구 제거 예외 범위를 확정한다.
- 현재 deep link와 메뉴 구조를 표 기준으로 고정한다.

### Phase 1. 공통 설명문구 차단

대상:

- `PageHeader`
- `SectionCard`
- `SummaryCard`
- `SidebarNav`
- 관리자 메뉴 권한 화면

작업:

- route-level 화면에서 `description` 기본 사용을 중단한다.
- 메뉴 설명 렌더링을 제거한다.
- 설명이 꼭 필요하면 `Alert`, `helperText`, `도메인 가이드`로 이동한다.

완료 기준:

- 상단 헤더는 `제목 + 배지 + 메타데이터 + 액션`만 남는다.

### Phase 2. P0 화면 분리

대상:

1. `기준 데이터 관리`
2. `내 계정 / 보안`

작업:

- 새 라우트와 메뉴 트리를 만든다.

- 상위 허브 또는 기존 경로는 새 화면으로 자연스럽게 연결한다.

완료 기준:

- 각 화면이 하나의 작업만 담당한다.
- 메뉴명만 보고도 들어갈 화면이 명확하다.

### Phase 3. P1 화면 분리

대상:

- `계획 항목`

작업:

- `생성`과 `목록/진행 추적`을 화면으로 분리한다.
- 현재 표 액션에서 사용하는 링크를 새 구조에 맞춰 정리한다.

완료 기준:

- 계획 생성 커맨드와 계획 추적 표가 같은 시야를 강하게 경쟁하지 않는다.

### Phase 4. 전체 화면 설명문구 일괄 정리

- 기존 경로에서 새 경로로의 이동 정책을 문서화한다.

## 검증 기준

아래 조건을 만족하면 이번 작업이 끝난 것으로 본다.

1. 메뉴는 `이름`만으로도 이동 목적이 분명하다.
2. 독립 기능 2개 이상이 한 route 화면 안에 공존하지 않는다.
3. 화면 상단 설명 문단 없이도 첫 시야에서 해야 할 작업이 보인다.
4. 카드와 요약 블록은 기능명, 상태값, 버튼만으로 읽힌다.
5. 검증 메시지와 파괴적 작업 경고는 그대로 남는다.
6. 도메인 배경 설명은 `도메인 가이드`로만 이동해도 사용 흐름이 깨지지 않는다.
7. 기존 주요 링크는 새 구조에서도 끊기지 않는다.

## 이번 작업에서 제외하는 것

- 새로운 도메인 기능 추가
- 회계 규칙 변경
- API 계약 변경이 필요한 대규모 백엔드 리팩터링
- 메모, 정비 내용, 전표 라인 설명 같은 실제 업무 데이터 삭제
- 유효성 검증 문구와 에러 문구 제거

## 최종 유지 원칙

1. 메뉴명은 `실제 기능명`만 쓴다.
2. 허브 화면은 `요약 + 이동`만 담당한다.
3. 기능 화면은 한 화면당 한 작업 책임만 가진다.
4. 설명은 화면이 아니라 `도메인 가이드`에 둔다.
5. 새 화면을 추가할 때도 먼저 `이게 독립 기능인가`를 판단한 뒤 라우트를 만든다.
