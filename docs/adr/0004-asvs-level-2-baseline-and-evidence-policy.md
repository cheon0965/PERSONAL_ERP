# ADR 0004. ASVS Level 2 Baseline and Evidence Policy

## 상태

승인

## 배경

이 프로젝트는 1인 사업자와 소상공인의 사업 운영 재무 데이터와 현금흐름을 다루므로, 최종적으로 `OWASP ASVS Level 2` 수준을 목표로 삼는 것이 타당하다.

다만 현재 프로젝트는 다음 특성을 가진다.

- 1인 개발 중심이다.
- 포트폴리오 설명력도 중요하다.
- 모든 보안 요구사항을 한 번에 구현하기보다, 현재 구조에 맞는 순서로 올리는 편이 현실적이다.

그래서 `ASVS를 전면 체크리스트로 바로 강제`하는 대신, 현재 코드에 맞춘 기준선과 증적 정책을 먼저 고정할 필요가 있다.

## 결정

다음 원칙을 채택한다.

1. 이 프로젝트의 보안 목표 기준선은 `OWASP ASVS 5.0.0 Level 2`로 둔다.
2. 구현 범위가 없는 기능은 `N/A` 또는 `deferred`로 분리한다.
3. 보안 작업은 항상 아래 넷 중 최소 둘 이상의 증적을 남긴다.
   - 코드 변경
   - 테스트
   - 문서 또는 ADR
   - CI/운영 로그
4. 현재 기준선과 갭은 `docs/ASVS_L2_BASELINE_MATRIX.md`에서 관리한다.
5. 구현 우선순위는 아래 순서를 따른다.
   - 인증/세션 하드닝
   - 브라우저/API 경계 보안
   - CI 보안 게이트
   - 보안 이벤트 로깅
   - 접근통제/데이터 보호 증적 보강

## 이유

- 지금 가장 큰 리스크는 브라우저 토큰 저장과 세션 하드닝 부족이다.
- 입력 검증, workspace 접근통제, env 검증, request-id 같은 기반은 이미 있으므로, 높은 효과의 갭부터 메우는 편이 효율적이다.
- 포트폴리오 관점에서도 `기준을 세우고 단계적으로 강화했다`는 설명이 가능해진다.
- 기능이 없는 영역까지 억지로 구현하지 않게 만들어 과설계를 줄인다.

## 결과

좋은 점:

- 보안 작업의 범위와 우선순위가 흔들리지 않는다.
- 이후 변경이 `ASVS L2 목표와 얼마나 연결되는지` 설명하기 쉬워진다.
- 문서, 테스트, CI를 함께 남기는 습관을 강제할 수 있다.

감수하는 점:

- 초기에는 완전한 control-by-control 매핑보다 실용적 매핑을 우선한다.
- MFA, 비밀번호 재설정, 외부 결제/SMS처럼 아직 없는 기능 영역은 지금 당장 구현하지 않고 `N/A` 또는 `deferred`로 관리한다.
- 이 ADR 작성 이후 회원가입/이메일 인증과 업로드 배치 파일첨부는 구현되었으므로, 현재 판정은 `docs/ASVS_L2_BASELINE_MATRIX.md`의 회원가입/파일 업로드 보안 항목을 우선한다.

## 후속 작업

1. `docs/ASVS_L2_EXECUTION_PLAN.md`의 P1을 시작한다.
2. 세션 저장 정책과 refresh lifecycle을 재설계한다.
3. 보안 회귀 테스트와 CI 보안 게이트를 단계적으로 추가한다.
