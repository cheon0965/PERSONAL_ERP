# 디자인 시스템

## 목표

- 화면이 늘어나도 같은 톤과 밀도로 확장되게 만든다.
- 도메인 기능과 공통 UI를 명확히 분리한다.
- 리스트, 카드, 폼, 차트 패턴을 재사용 가능하게 유지한다.
- 월 운영 중심 ERP 흐름이 화면 구조만 봐도 자연스럽게 읽히게 만든다.

## 레이아웃 구조

- `shared/layout/app-shell.tsx`
- `shared/layout/sidebar-nav.tsx`
- `shared/layout/topbar.tsx`

대시보드 라우트는 모두 같은 App Shell 안에서 동작합니다.

현재 App Shell 원칙은 아래와 같습니다.

- 사이드바 브랜드는 짧게 유지하고, 긴 보조 설명은 넣지 않습니다.
- 1차 메뉴는 `운영 준비 / 월 실행 / 운영 자산 / 보고 / 판단` 그룹으로 나눕니다.
- Topbar는 현재 사업장과 장부 기준을 보여주고, 페이지 헤더는 현재 화면 기준과 다음 액션을 보여주는 역할로 분리합니다.

## 공통 UI 컴포넌트

- `PageHeader`
- `SectionTabs`
- `SummaryCard`
- `SectionCard`
- `DataTableCard`
- `ChartCard`
- `StatusChip`
- `QueryErrorAlert`

원칙:

- feature에서 반복되는 조각만 공통화합니다.
- 도메인별 폼이나 컬럼 구성은 feature 안에 둡니다.
- 상위 영역 내부 전환은 `SectionTabs` 한 패턴으로 통일합니다.

## 핵심 페이지 패턴

현재 주요 화면은 아래 구조 중 하나로 수렴시키는 것을 기본 규칙으로 삼습니다.

1. `PageHeader`
2. `SectionTabs` 또는 같은 수준의 국소 내비게이션
3. `핵심 상태/선택 바`
4. `주 작업 영역`
5. `보조 정보 영역`

핵심 원칙은 `주 작업 영역이 화면의 시선 중심`이 되게 만드는 것입니다.

## 화면 역할 분류

새 화면을 만들거나 기존 화면을 정리할 때는 먼저 이 화면이 어떤 역할인지부터 결정합니다.

- `쓰기 화면`
  - 계획 항목, 수집 거래, 전표 조회, 업로드 배치처럼 입력·검토·확정이 핵심인 화면입니다.
  - 요약 카드보다 `목록`, `폼`, `액션`, `선택 상세`가 먼저 보여야 합니다.
- `읽기 화면`
  - 대시보드, 재무제표, 자금수단별 현황, 기간 전망처럼 해석과 비교가 핵심인 화면입니다.
  - 기준 선택 뒤에 `핵심 수치`, `추이`, `해석`이 이어져야 합니다.
- `허브 화면`
  - 운영 지원, 관리자 홈처럼 후속 화면 진입과 우선순위 확인이 핵심인 화면입니다.
  - 링크 나열보다 `지금 우선 확인`, `후속 작업`, `바로 이동`이 먼저 보여야 합니다.
- `기준 관리 화면`
  - 기준 데이터, 설정처럼 운영 기준을 점검·수정하는 화면입니다.
  - 현재 기준과 편집 대상을 먼저 보여주고, 긴 설명은 보조 영역으로 내립니다.

## 페이지 조합 패턴

일반적인 대시보드형 화면은 다음 순서를 권장합니다.

1. `PageHeader`
2. 국소 내비게이션
3. 상태/선택 요약 또는 표 툴바
4. 표 또는 차트
5. 보조 카드, 상세 패널, 드로어

화면 유형별 기본 원칙은 아래와 같습니다.

- 쓰기 화면: 표, 폼, 액션이 먼저 보여야 합니다.
- 읽기 화면: 요약, 추이, 해석이 먼저 보여야 합니다.
- 허브 화면: 링크 모음보다 `지금 처리할 것`이 먼저 보여야 합니다.
- 기준 데이터 화면: 준비 상태와 관리 화면을 분리합니다.

## PageHeader 규칙

`PageHeader`는 설명문 블록이 아니라 조작 기준점으로 사용합니다.

- `eyebrow`, `title`, `description`은 항상 짧게 유지합니다.
- `description`은 가능하면 1~2문장 안에서 끝냅니다.
- `badges`에는 현재 상태, 선택 기간, readiness 같은 짧은 상태만 넣습니다.
- `metadata`에는 대상 월, 권한, 규모, 최근 기준처럼 인라인으로 읽을 정보를 넣습니다.
- `primaryAction`, `secondaryAction`은 현재 화면의 다음 행동만 남깁니다.

헤더에서 다 설명하려 하지 말고, 긴 도메인 안내는 `도메인 가이드`나 보조 섹션으로 보냅니다.
실제 구현에서는 `PageHeader`와 `Topbar`의 `도메인 가이드`가 같은 도움말 문맥을 엽니다.

## 카드/표 규칙

- `DataTableCard`는 리스트 중심 화면의 기본 뼈대입니다.
- 표 중심 화면에서는 가능하면 `toolbar`를 사용해 필터, 상태 요약, 짧은 운영 문맥을 표 바로 위에 둡니다.
- `DataTableCard`는 데스크톱에서는 DataGrid pagination을 사용하고, 모바일에서는 카드 목록으로 전환하되 5/10/20개 단위 페이지네이션을 유지합니다.
- 모바일 카드 목록을 직접 새로 만들기보다 `mobileCard` 슬롯으로 카드 본문만 바꾸고, 선택/빈 상태/페이지네이션은 공통 컴포넌트에 맡깁니다.
- `SectionCard`는 상태 요약, 후속 화면 안내, 설명 블록을 묶을 때 씁니다.
- `SummaryCard`는 read-heavy 화면에서만 제한적으로 씁니다.
- 읽기 화면에서도 첫 줄 KPI 카드는 최대 4개까지만 허용합니다.
- 쓰기 화면에서는 KPI 카드 대신 인라인 메타, 칩, 표 툴바를 우선합니다.

표 중심 ERP 화면에서는 아래 순서를 기본값으로 둡니다.

1. `헤더에서 현재 기준 확인`
2. `표 툴바에서 필터와 짧은 상태 요약 확인`
3. `목록에서 대상 선택`
4. `상세 패널/드로어에서 생성·수정·보정`

즉, `목록 먼저, 편집은 나중`이 기본 패턴입니다.

## 드로어/다이얼로그 규칙

- 생성/수정은 가능하면 드로어 또는 다이얼로그 안으로 밀어 넣습니다.
- 목록 화면에서는 `목록 확인 -> 선택 대상 결정 -> 드로어 열기` 흐름이 먼저 읽혀야 합니다.
- 드로어 설명은 짧게 유지하고, 현재 대상과 변경 영향만 분명히 알려줍니다.

## 스타일 원칙

- MUI theme는 `shared/theme`에서 관리합니다.
- 시각 토큰은 `tokens.ts`를 기준으로 유지합니다.
- 공통 컴포넌트는 도메인 용어보다 구조적 역할 이름을 사용합니다.
- 사이드바, 탭, 헤더, 표 툴바의 라벨은 가능한 한 짧게 유지합니다.
- 경고/상태 메시지는 화면 상단에 과도하게 쌓지 않고, 핵심 작업을 가리지 않는 위치에 배치합니다.

## 확장 원칙

- 새로운 feature가 기존 컴포넌트를 2곳 이상에서 재사용할 때만 `shared` 승격을 검토합니다.
- 현재는 디자인 검증 전용 `/design-system` 페이지를 별도로 유지하지 않습니다.
- 대신 공통 컴포넌트와 레이아웃 검증은 실제 대시보드 화면과 feature 조합 안에서 계속 확인합니다.
- 새 화면을 만들 때는 먼저 `PageHeader + SectionTabs + DataTableCard/SectionCard` 조합으로 해결 가능한지 검토합니다.
