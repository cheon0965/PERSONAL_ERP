# 화면 기준 운영 흐름 가이드

## 문서 목적

이 문서는 현재 Web 화면을 기준으로 PERSONAL_ERP의 월별 재무 운영 메인 흐름을 빠르게 이해하고, 데모 계정/데모 데이터로 한 사이클을 짧게 체험하기 위한 가이드다.

- 도메인 규칙의 상위 기준은 `docs/domain/business-logic-draft.md`와 `docs/domain/core-entity-definition.md`를 우선한다.
- 이 문서는 그 도메인 규칙이 실제 화면에서 어떤 순서로 보이고, 사용자가 어떤 흐름으로 따라가면 되는지 설명하는 운영용 안내서다.

## 한 줄로 보는 메인 흐름

`작업 문맥/설정 확인 -> 운영 지원 허브 점검 -> 기준 데이터 준비 -> 월 운영 시작 -> 보험 계약/차량 운영 정리 -> 반복 규칙 정리 -> 계획 항목 생성 -> 수집 거래 입력 또는 업로드 배치 처리 -> 전표 확정/반전/정정 -> 월 마감 -> 재무제표 생성 -> 차기 이월 -> 기간 전망 확인`

핵심은 거래 한 건을 입력하는 것이 아니라, `월 단위 운영 사이클`을 끝까지 닫는 것이다.
즉, 범용 소상공인 ERP 전반보다 월별 재무운영 흐름을 정확히 수행하는 쪽에 더 가깝다.

## 현재 UI를 읽는 공통 규칙

최근 화면 정리 이후 주요 화면은 같은 리듬으로 읽으면 된다.

1. `헤더`
2. `하위 화면 탭 또는 보조 메뉴`
3. `현재 기준 또는 짧은 상단 상태`
4. `주 작업 목록 또는 결과 본문`
5. `보조 정보와 후속 화면`

쓰기 화면은 `목록 먼저`, 읽기 화면은 `기준 선택 후 결과`, 허브 화면은 `지금 우선 확인`이 먼저 보이도록 맞춰져 있다.
긴 설명은 헤더에 쌓지 않고 `도메인 가이드`에서 이어서 보는 것을 기본으로 한다.

## 5분 데모 체험 루트

가장 먼저 아래 루트를 한 번 따라가면 전체 구조가 빠르게 잡힌다.
이 루트는 모든 기능을 깊게 쓰는 튜토리얼이 아니라, 데모 데이터로 `계획 -> 수집 거래 -> 전표 -> 마감 -> 재무제표 -> 차기 이월 -> 전망`이 이어지는 감각을 확인하는 빠른 시승 코스다.

### 체험 전 준비

로컬에서 실제 API와 DB를 붙여 체험한다면 README의 빠른 시작을 먼저 완료한다.

```bash
npm run db:up
npm run db:migrate
npm run db:seed
npm run dev
```

Web은 `http://localhost:3000`에서 열고, 데모 계정은 아래 값을 사용한다.

| 항목     | 값                 |
| -------- | ------------------ |
| Email    | `demo@example.com` |
| Password | `Demo1234!`        |

데모 seed에는 `사업 운영 통장`, `비용 예비 통장`, `사업용 카드`, 기본 카테고리, 2026년 4월 기준 반복 규칙, 보험 계약, 차량 운영 기록이 준비되어 있다.
따라서 5분 체험에서는 보험/차량/반복 규칙을 새로 만들기보다, 이미 있는 기준을 확인하고 바로 월 운영으로 넘어가는 편이 좋다.

새 계정을 만들 때는 `/register`에서 이름, 이메일, 비밀번호를 입력한 뒤 인증 메일의 링크를 열어 `/verify-email` 흐름을 완료한다.
사업장 초대를 받은 기존 사용자는 초대 메일의 `/accept-invitation` 링크로 멤버십을 활성화한 뒤 로그인한다.
인증이 끝난 사용자는 기본 workspace, 기본 장부, OWNER membership이 준비된 상태로 `/login`에서 로그인한다.

### 5분 안에 한 사이클 보기

아래 순서대로만 따라가면 보통 5분 안에 월 운영 한 사이클의 감각을 확인할 수 있다.

| 화면                         | 할 일                                                                                                                 | 확인할 것                                              |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| `로그인`                     | `demo@example.com` / `Demo1234!`로 로그인한다.                                                                        | 로그인 후 현재 작업 문맥과 대시보드가 열린다.          |
| `작업 문맥`                  | Tenant, 장부, 권한이 보이는지 확인한다.                                                                               | 데모 사용자가 현재 장부의 Owner 권한을 가진다.         |
| `운영 지원`                  | 운영 허브와 체크리스트를 열어 위험 신호를 확인한다.                                                                   | 예외, 마감 차단, 업로드 현황, 상태/알림 진입점을 본다. |
| `기준 데이터`                | readiness 요약과 자금수단/카테고리를 확인한다.                                                                        | 자금수단과 수입/지출 카테고리가 준비되어 있다.         |
| `월 운영`                    | 열린 기간이 없으면 `2026-04` 월을 `/periods/open`에서 연다.                                                           | 첫 월이면 오프닝 잔액 라인 1건 이상이 필요하다.        |
| `계획 항목`                  | `2026-04` 기간을 선택하고 `계획 항목 생성`을 누른다.                                                                  | 반복 규칙에서 계획 항목과 연결 수집 거래가 생긴다.     |
| `계획 항목` 또는 `수집 거래` | `전표 준비` 상태 항목을 모두 `바로 전표 확정` 또는 `전표 확정`으로 확정한다.                                          | 전표 번호가 연결되고 계획 항목 상태가 확정으로 바뀐다. |
| `전표 조회`                  | 방금 생성된 전표를 열어 본다.                                                                                         | 필요 시 반전 전표/정정 전표를 만들 수 있음을 확인한다. |
| `월 운영`                    | `/periods/close`에서 현재 열린 월을 마감한다.                                                                         | 미확정 수집 거래가 남아 있으면 먼저 확정해야 한다.     |
| `재무제표`                   | `/financial-statements`에서 잠금된 `2026-04`를 선택해 생성하고, `/financial-statements/[periodId]`에서 보고서를 본다. | 마감된 전표 기준 공식 보고 스냅샷이 생긴다.            |
| `차기 이월` / `기간 전망`    | `/carry-forwards`에서 생성한 뒤 `/carry-forwards/[periodId]` 결과를 보고, `기간 전망`에서 다음 달 전망을 본다.        | 다음 월 오프닝 기준과 전망 read model이 이어진다.      |

첫 월을 여는 경우 오프닝 잔액 라인은 예를 들어 `현금및예금` 계정과목, `사업 운영 통장` 자금수단, `2,450,000`원으로 입력하면 데모 seed의 자금수단과 자연스럽게 맞는다.
이미 열린 운영 기간이 있다면 새 월을 열지 말고 현재 열린 기간으로 바로 `계획 항목 생성`부터 진행한다.

### 체험 중 막히면 먼저 볼 것

| 증상                           | 먼저 확인할 화면         | 처리 방법                                                                            |
| ------------------------------ | ------------------------ | ------------------------------------------------------------------------------------ |
| 월 운영 시작이 막힌다          | `기준 데이터`, `월 운영` | readiness 부족을 확인하고, 첫 월이면 오프닝 잔액 라인을 1건 이상 입력한다.           |
| 계획 항목이 생성되지 않는다    | `반복 규칙`, `월 운영`   | 열린 기간이 있는지, 활성 반복 규칙이 해당 월에 걸리는지 확인한다.                    |
| 전표 확정 버튼이 보이지 않는다 | `계획 항목`, `수집 거래` | 연결 수집 거래가 `전표 준비` 상태인지 확인하고, 부족한 카테고리/자금수단을 보완한다. |
| 월 마감이 실패한다             | `수집 거래`              | 미확정 수집 거래가 남아 있으면 전표 확정하거나 정리한 뒤 다시 마감한다.              |
| 재무제표/이월 생성이 안 된다   | `월 운영`                | 해당 기간이 `잠금` 상태인지 확인한다. 재무제표와 이월은 마감 이후에 실행한다.        |

업로드 배치까지 보고 싶다면 5분 체험을 한 번 마친 뒤 `업로드 배치 -> 수집 거래 승격 -> 수집 거래에서 전표 확정`만 별도 루트로 확인하는 것을 권장한다.
업로드는 원본 보존과 자동 판정 preview까지 함께 보게 되므로, 첫 체험에서는 계획 항목이 자동으로 만든 수집 거래를 먼저 따라가는 편이 이해가 빠르다.

## 현재 사이드바 메뉴 그룹

현재 Web 사이드바는 프런트 하드코딩 목록이 아니라 DB에 저장된 `WorkspaceNavigationMenuItem` 트리를 `/navigation/tree`에서 읽어 구성한다.
메뉴 권한은 `WorkspaceNavigationMenuRole`에 역할별로 저장하며, `/admin/navigation`의 `메뉴 / 권한` 화면에서 소유자가 조정한다.

기본 트리는 운영 흐름 기준으로 아래처럼 접히는 드롭다운 구조를 가진다.

### 운영 준비

- `작업 문맥`: `사업장 설정`, `기본 정보`, `비밀번호`, `세션`, `보안 이벤트`
- `관리자`: `회원 관리`, `메뉴 / 권한`, `감사 로그`, `권한 정책`
- `플랫폼 관리`(전체 관리자 전용): `관리자 허브`, `전체 사용자 관리`, `사업장 관리`, `사업장 전환 / 지원 모드`, `보안 위협 로그`, `로그관리`, `운영 상태`, `메뉴 / 권한 관리`, `권한 정책 요약`
- `운영 지원`: `체크리스트`, `예외 처리함`, `월 마감 지원`, `업로드 현황`, `시스템 상태`, `알림 센터`, `백업 / 내보내기`, `운영 메모`
- `기준 데이터`: `자금수단`, `카테고리`, `공식 참조값`

### 월 실행

- `월 운영`: `월 운영 시작`, `월 마감 / 재오픈`, `기간 이력`
- `반복 규칙`
- `계획 항목`
- `업로드 배치`
- `수집 거래`
- `전표 조회`

### 운영 자산

- `보험 계약`
- `차량 운영`: `차량 목록`, `연료 기록`, `정비 이력`

### 보고 / 판단

- `재무제표 생성 / 선택`
- `차기 이월 생성 / 선택`
- `기간 전망`
- `대시보드`

현재 사이드바는 허브 화면과 책임이 분리된 하위 화면을 함께 보여준다.
즉, 탭 안에서만 이동하지 않고 `월 운영 시작`, `월 마감 / 재오픈`, `차량 목록`, `연료 기록`, `정비 이력` 같은 실제 작업 화면을 바로 메뉴에서 열 수 있다.
상위 메뉴를 숨기거나 역할을 제한하면 하위 메뉴도 함께 노출되지 않으며, 직접 URL 접근도 같은 메뉴 권한 기준으로 한 번 더 차단한다.

`대시보드`는 사이클 중간중간 확인하는 운영 현황 화면이라, 실제 작업 순서를 방해하지 않도록 사이클 말미의 상시 점검 화면으로 둡니다.

## 화면 기준 메인 로직 요약

### 1. 작업 문맥

`작업 문맥` 화면은 지금 내가 어떤 Tenant, Membership, Ledger 문맥 안에서 작업하는지 확인하는 시작점이다.

- 여기서 확인한 Tenant, 권한, Ledger가 이후 모든 화면의 조회/쓰기 기준이 된다.
- 데이터가 이상하게 보일 때는 가장 먼저 이 화면을 확인하는 것이 좋다.
- 사업장명, 슬러그, 기본 장부명, 기준 통화, 시간대 수정은 `/settings/workspace`에서 Owner/Manager가 수행한다.
- 내 계정 이름은 `/settings/account/profile`, 비밀번호는 `/settings/account/password`, 활성 세션은 `/settings/account/sessions`, 최근 보안 이벤트는 `/settings/account/events`에서 관리한다.

### 2. 관리자

`관리자` 화면은 기본적으로 현재 사업장 안에서 멤버와 감사 로그를 확인하는 운영 관리 화면이다. `INITIAL_ADMIN_*`로 주입된 전역 관리자 계정은 여기에 더해 별도의 `플랫폼 관리` 메뉴에서 모든 사업장과 사용자를 횡단 조회하는 범위로 동작한다.

- 회원관리는 현재 Tenant 멤버의 역할과 상태를 확인하고, 소유자가 초대와 역할/상태 변경을 수행하는 곳이다.
- 메뉴/권한 관리는 DB에 저장된 사이드바 트리와 메뉴별 허용 역할을 조정하는 곳이다.
- 로그관리는 회원관리 명령과 권한 거부 같은 workspace 감사 이벤트를 requestId 기준으로 추적하는 곳이다.
- 권한 정책 요약은 `/admin/policy`에서 DB 메뉴 기준 OWNER / MANAGER / EDITOR / VIEWER의 화면 노출 기준을 확인하는 곳이다.
- 일반 소유자/관리자는 현재 사업장 문맥 안에서만 동작한다.
- 전체 관리자는 `/admin` 허브에서 위험 요약과 바로가기를 보고, `/admin/users`, `/admin/tenants`, `/admin/security-threats`, `/admin/operations`에서 플랫폼 단위 운영 상태를 확인한다.
- 전체 관리자는 `/admin/support-context`에서 특정 사업장/장부를 명시적으로 선택한 뒤 지원 모드로 업무 화면을 연다.
- 지원 모드가 켜지면 상단바에 `지원 모드` 배지가 보이고, 사업장 운영 메뉴는 선택한 지원 문맥 기준으로 열린다.

### 2-1. 운영 지원

`운영 지원` 화면은 회계 본류를 새로 만들지 않고 운영자가 당장 볼 위험 신호와 후속 작업을 한곳에 모으는 보조 허브다.

- 운영 허브는 `오늘 우선 확인 -> 월 마감 준비 -> 운영 도구 -> 시스템/감사` 순서로 읽히게 구성한다.
- 체크리스트, 예외 처리함, 업로드 현황, 월 마감, 알림, 시스템 상태는 모두 `헤더 기준 -> 짧은 상단 상태 -> 실제 처리 목록` 패턴으로 맞춘다.
- 예외 처리함, 월 마감, 업로드 운영 현황은 기존 기준 데이터/기간/수집/업로드/보고 화면으로 이동하는 deep link를 제공한다.
- 시스템 상태/헬스와 알림 센터는 DB readiness, 메일 경계, 감사 이벤트, 마감 차단 사유를 운영자가 먼저 볼 수 있게 모은다.
- 백업/내보내기는 기준 데이터, 수집 거래, 전표, 재무제표 스냅샷을 수동 UTF-8 CSV로 반출하고 감사 로그를 남긴다.
- 운영 메모/인수인계는 월 마감, 예외 처리, 알림 후속 조치 메모를 현재 workspace/ledger 문맥에 남긴다.

### 3. 기준 데이터

`기준 데이터` 화면은 입력 화면들이 공통으로 참조하는 공식 기준값을 보여준다.

- 자금수단, 카테고리, 계정과목, 거래유형이 여기에 있다.
- `수집 거래`, `반복 규칙`, `업로드 배치`, `전표`는 이 화면의 기준 데이터를 바탕으로 움직인다.
- 이 영역은 `준비 상태`, `자금수단`, `카테고리`, `공식 참조값`으로 분리해 보여준다.
- `준비 상태`는 read-only 점검 화면이고, 나머지 화면은 자금수단 관리, 카테고리 관리, 공식 참조값 확인을 각각 담당한다.
- 자금수단과 수입/지출 카테고리는 Owner/Manager가 직접 생성, 이름 수정, 비활성화/재활성화를 수행할 수 있다.
- 자금수단은 현재 범위에서 `ACTIVE/INACTIVE/CLOSED` lifecycle을 제한적으로 관리한다.
- `CLOSED` 전환은 비활성 자금수단에서만 가능하고, 종료 후에는 기존 기록 보존용 읽기 전용 상태로 유지한다.
- 자금수단의 잔액 직접 수정은 아직 지원하지 않는다.
- 계정과목, 거래유형은 `system-managed` 기준 데이터로 유지되며 운영자는 존재 여부와 활성 상태를 점검한다.

### 4. 월 운영

`운영 기간 관리` 화면은 월별 운영 사이클의 시작과 끝을 담당한다.

- 상단 `현재 운영 월` 카드에서 현재 월 상태와 다음 작업을 먼저 확인한다.
- `/periods/open`에서 `운영 시작`과 오프닝 잔액 입력을 처리한다.
- `/periods/close`에서 `월 마감 / 재오픈`을 처리한다.
- `/periods/history`에서 최근 월 상태와 기초 잔액 기준을 확인한다.

이 프로젝트에서 많은 화면은 `현재 열린 운영 기간`이 있어야 제대로 동작한다.

### 5. 보험 계약

`보험 계약` 화면은 보험료와 보장/만기 같은 운영 계약 기준을 관리한다.

- 보험 계약은 반복 규칙과 연결되어 월별 계획 지출의 원천이 될 수 있다.
- 계약 자체는 전표가 아니며, 계획과 수집 거래를 더 정확히 만들기 위한 운영 기준 데이터다.
- 활성/비활성 상태를 정리하면 이후 계획 생성과 검토 흐름에서 혼동을 줄일 수 있다.

### 6. 차량 운영

`차량 운영` 화면은 차량 기본 정보, 연료 이력, 정비 이력을 관리한다.

- 차량 기본 프로필과 운영 이력을 분리해 관리한다.
- 주유비, 정비비 같은 차량 관련 수집 거래를 분류하고 검토하는 참고 맥락이 된다.
- 현재 화면은 차량 비용을 직접 전표로 확정하는 화면이 아니라, 이후 반복 규칙/계획 항목/수집 거래를 더 정확히 해석하기 위한 운영 기록 화면이다.

### 7. 반복 규칙

`반복 규칙` 화면은 미래 계획의 입력 기준을 정의한다.

- 월세, 보험료, 구독료, 차량 관련 정기 비용 같은 반복 지출/수입 규칙을 만든다.
- 보험 계약이나 차량 운영에서 정리한 기준은 반복 규칙을 만들고 설명하는 보조 근거가 된다.
- 아직 회계 확정은 아니다.
- 이 화면의 결과는 다음 단계인 `계획 항목`으로 연결된다.
- 현재 UI는 `규칙 목록 -> 관리 기준 -> 보험 연동/후속 화면` 순서로 읽히게 정리되어 있다.

### 8. 계획 항목

`계획 항목` 영역은 반복 규칙을 현재 운영 기간의 PlanItem으로 생성하고, 생성된 계획을 추적하는 화면이다.

- `/plan-items/generate`에서 반복 규칙을 현재 월의 계획 데이터로 펼친다.
- 아직 실제 거래도 아니고 공식 전표도 아니다.
- “이번 달에 어떤 거래가 예정되어 있는가”를 운영 관점에서 보는 단계다.
- 현재 구현에서는 계획 항목 생성 시 연결된 반복성 수집 거래도 함께 만들어진다.
- `/plan-items`에서는 매칭된 수집 거래 제목/상태와 생성된 전표 번호까지 바로 추적할 수 있다.
- 전표 준비 상태의 연결 수집 거래는 계획 항목 화면에서도 바로 확정할 수 있지만, 실제 회계 확정은 여전히 연결된 수집 거래의 전표 확정 경로를 사용한다.

### 9. 업로드 배치

`업로드 배치` 화면은 파일이나 붙여넣기 원본을 파싱해서 ImportedRow로 보관하는 수집 경계다.

- 원본 파일과 파싱 결과를 보존한다.
- 필요한 행만 골라 `수집 거래`로 승격한다.
- 승격 전에 계획 항목 매칭, 카테고리 보완, 다음 상태, 중복 보류 여부를 preview로 먼저 확인한다.
- 이미 승격한 행은 어떤 수집 거래로 연결됐는지와 그 상태를 다시 추적할 수 있다.
- 즉, 업로드 배치는 `실제 거래 확정`이 아니라 `수집 후보 관리` 화면이다.
- 현재 UI는 `/imports` 목록 화면과 `/imports/[batchId]` 작업대 화면을 분리해 `배치 선택 -> 배치 작업대 -> 업로드 행` 순서로 읽히게 정리되어 있다.
- 배치와 행은 모두 표 중심으로 확인하고, 승격·보완은 선택한 배치 기준으로 이어서 처리한다.

화면 흐름으로 보면 다음과 같다.

`업로드 배치 -> ImportedRow 검토 -> 수집 거래 승격 -> 수집 거래 화면에서 확인/확정`

### 10. 수집 거래

`수집 거래` 화면은 실제로 발생한 거래를 운영 기간 안에 수집하는 화면이다.

- 수기 등록으로 직접 입력할 수 있다.
- 업로드 배치에서 승격된 거래도 여기에 모인다.
- 계획 항목 생성으로 자동 등록된 예정 거래도 수집 거래에 모인다.
- 이 단계도 아직 최종 회계 확정 전이다.
- 수집·검토·전표 준비 상태를 구분해 본 뒤, 전표 준비 상태의 거래를 전표로 확정할 수 있다.
- 연결된 원계획이 있으면 이 화면에서도 바로 따라갈 수 있다.
- 현재 UI는 `운영 월과 필터 -> 거래 목록 -> 선택 작업`보다 `거래 목록`이 먼저 읽히게 구성되어 있다.

### 11. 전표 조회

`전표 조회` 화면은 공식 회계 확정 결과를 검토하는 화면이다.

- 수집 거래를 확정하면 JournalEntry가 생성된다.
- 전표는 회계적 진실의 단일 원천이다.
- 필요하면 반전 전표나 정정 전표를 추가로 생성한다.
- 확정된 수집 거래는 삭제/취소로 되돌리지 않고, 전표 조정 이력을 남기는 방식으로 처리한다.
- 현재 UI는 `/journal-entries` 목록과 `/journal-entries/[entryId]` 상세를 분리해 `전표 목록 -> 상세 -> 라인/조정 이력` 구조로 읽힌다.

즉, `계획 항목 -> 수집 거래 -> 전표`로 갈수록 공식성이 높아진다.

### 12. 재무제표

`재무제표 스냅샷` 화면은 잠금된 운영 기간의 공식 보고 결과를 보여준다.

- `/financial-statements`에서는 잠금된 기간을 선택하고 생성 여부를 확인한다.
- `/financial-statements/[periodId]`에서는 ClosingSnapshot과 JournalEntry를 바탕으로 생성된 공식 보고 스냅샷을 읽는다.
- 대시보드/전망과 달리 이 화면은 공식 보고 계층이다.

### 13. 차기 이월

`이월 기준 생성` 화면은 잠금된 기간의 결과를 다음 월 오프닝 기준으로 넘긴다.

- `/carry-forwards`에서는 잠금된 기간을 기준으로 CarryForward를 만든다.
- `/carry-forwards/[periodId]`에서는 다음 월의 OpeningBalance 기준이 어떻게 이어지는지 확인한다.
- 월 운영 사이클을 다음 달로 연결하는 마무리 단계다.

### 14. 기간 전망

`기간 운영 전망` 화면은 현재 월과 다음 달의 운영 판단을 돕는 read model이다.

- 열린 월의 확정 전표와 남은 계획을 함께 읽어 현재 기간말 전망을 본다.
- 차기 이월 이후에는 다음 달 전망까지 확인해 다음 월 운영 준비 상태를 점검한다.
- 공식 보고 계층은 아니며, 재무제표/차기 이월 이후 운영 판단을 이어가는 보조 화면이다.
- 현재 UI는 `요약 수치 -> 전망 드라이버 -> 공식 비교/참고사항` 순서로 읽히게 정리되어 있다.

## 화면 순서로 이해하는 운영 루프

## 현재 화면 리듬

대부분의 주요 화면은 아래 리듬으로 읽히도록 정리되어 있다.

1. `PageHeader`
2. `하위 화면 탭 또는 보조 메뉴`
3. `현재 기준` 또는 `지금 우선 확인`
4. `표 / 차트 / 작업 목록`
5. `후속 화면` 또는 `보조 설명`

즉, 긴 설명 카드보다 `현재 기준`, `다음 행동`, `주요 리스트`가 먼저 보이도록 맞춘 상태다.
도메인 규칙 설명이 더 필요하면 헤더의 `도메인 가이드`에서 이어서 본다.

### 추천 기본 순서

1. `작업 문맥`에서 현재 Tenant, Ledger, 권한을 확인하고 필요하면 `사업장 설정`, `내 계정/보안`, `권한 정책`을 점검한다.
2. `운영 지원`에서 운영 체크리스트, 예외 처리함, 월 마감/업로드 현황, 알림 센터를 먼저 확인한다.
3. `기준 데이터`에서 readiness 요약을 먼저 보고, 자금수단·수입/지출 카테고리·계정과목·거래유형이 모두 준비됐는지 확인한다.
4. `월 운영`에서 현재 월을 연다.
5. `보험 계약`과 `차량 운영`에서 반복 비용과 운영 참고 데이터를 정리한다.
6. `반복 규칙`을 준비하거나 점검한다.
7. `계획 항목`에서 현재 월의 계획 데이터를 생성한다.
8. 실제 거래가 파일/붙여넣기 기반이면 먼저 `업로드 배치`에서 자동 판정 preview를 확인하고 승격한다.
9. 직접 입력이나 보완이 필요한 거래는 `수집 거래`에서 정리한다.
10. 계획 항목 또는 수집 거래 화면에서 검토가 끝난 수집 거래를 전표로 확정한다.
11. `전표 조회`에서 확정 결과와 반전/정정 전표 필요 여부를 확인한다.
12. `월 운영`으로 돌아가 마감을 실행한다.
13. `재무제표`에서 공식 보고 스냅샷을 생성한다.
14. `차기 이월`에서 다음 월 오프닝 기준을 만든다.
15. `기간 전망`에서 다음 달 전망과 남은 계획을 확인한다.
16. 필요한 경우 `운영 지원`의 `백업/내보내기`와 `운영 메모/인수인계`로 반출과 후속 기록을 남긴다.

## 실무적으로 가장 많이 쓰게 될 두 가지 경로

### A. 수기 입력 중심 경로

`작업 문맥 -> 운영 지원 -> 기준 데이터 -> 월 운영 -> 수집 거래 -> 전표 조회 -> 월 마감 -> 재무제표 -> 차기 이월 -> 기간 전망`

이 경로는 반복 규칙과 업로드 없이도 기본 회계 운영을 끝까지 수행할 수 있는 최소 루트다.

### B. 계획 + 업로드 중심 경로

`작업 문맥 -> 운영 지원 -> 기준 데이터 -> 월 운영 -> 보험 계약/차량 운영 -> 반복 규칙 -> 계획 항목 -> 업로드 배치 -> 수집 거래 -> 전표 조회 -> 월 마감 -> 재무제표 -> 차기 이월 -> 기간 전망`

이 경로는 이 프로젝트가 의도한 전체 운영 사이클에 더 가깝다.

### C. 5분 데모 체험 경로

`로그인 -> 작업 문맥 -> 운영 지원 -> 기준 데이터 -> 월 운영 시작 -> 계획 항목 생성 -> 바로 전표 확정 -> 전표 조회 -> 월 마감 -> 재무제표 생성 -> 차기 이월 생성 -> 기간 전망`

이 경로는 이미 만들어진 데모 기준 데이터를 최대한 활용해서 전체 사이클을 가장 짧게 경험하는 루트다.
보험 계약, 차량 운영, 반복 규칙은 체험 전에 “이미 준비된 기준”으로 확인만 하고, 실제 쓰기 흐름은 계획 항목부터 시작한다.

## 준비 부족 시 화면이 어떻게 안내하나

1. `월 운영` 화면은 월을 열기 전에 기준 데이터 readiness 부족을 경고하고 `기준 데이터` 화면으로 이동 링크를 제공한다.
2. `수집 거래` 등록 드로어는 자금수단/카테고리 부족 여부와 현재 readiness 부족 항목을 함께 안내한다.
3. `업로드 배치` 승격 드로어는 자동 판정 preview와 별도로 기준 데이터 readiness 부족을 함께 알려 준다.
4. 즉, 현재 제품은 자금수단과 수입/지출 카테고리의 제한적 관리만 제품 안에서 제공하고, 계정과목/거래유형은 system-managed 안내 중심으로 유지한다.

## 대시보드와 전망은 어디서 보나

`월 운영 대시보드`와 `기간 운영 전망`은 메인 쓰기 흐름의 시작점이라기보다, 각 단계 사이에서 상태를 읽는 운영 판단 화면이다.

- `대시보드`: 현재 기간의 운영 숫자와 최근 공식 숫자를 함께 본다.
- `전망`: 남은 계획과 확정 전표를 합쳐 기간말 예상 상태를 보고, 차기 이월 이후 다음 달 전망까지 확인한다.
- 대시보드는 요약과 빠른 체크포인트 중심이고, 세부 해석은 전망/재무제표/전표 화면으로 이어진다.

권장 사용 시점은 다음과 같다.

1. 월을 연 직후 현재 상태 확인
2. 계획 항목 생성 후 계획 대비 확인
3. 업로드 배치 승격 결과와 계획 항목 연결 상태 확인
4. 수집 거래/전표 확정 후 실제 반영 상태 확인
5. 마감 직전 이상 징후 점검
6. 차기 이월 이후 다음 월 운영 준비 상태 확인

## 보험/차량 화면은 메인 흐름에서 어떻게 보나

`보험 계약`과 `차량 운영`은 메인 회계 확정 화면은 아니지만, 현재 사이클에서는 반복 규칙과 계획 항목 앞에서 정리하는 운영 자산 화면으로 본다.

- 보험 계약은 반복 규칙과 계획 지출을 설명하는 계약 기준 데이터다.
- 차량 운영은 차량비/주유비를 더 정확히 분류하고 검토하는 운영 참고 데이터다.
- 두 화면 모두 수집 거래와 전표를 직접 확정하지 않고, 이후 계획/수집/검토를 더 정확히 만드는 앞단 기준으로 사용한다.

즉, 이 화면들은 메인 흐름 바깥이 아니라 `반복 규칙`, `계획 항목`, `수집 거래`를 더 잘 운영하기 위한 선행 컨텍스트다.

## 화면 기준으로 기억하면 좋은 공식성 단계

가볍게 기억하면 아래 순서다.

1. `보험 계약` / `차량 운영`: 반복 비용과 운영 참고 기준
2. `반복 규칙`: 계획 기준
3. `계획 항목`: 기간별 계획 데이터
4. `수집 거래` / `업로드 배치`: 실제 거래 수집과 검토
5. `전표 조회`: 공식 회계 확정
6. `재무제표` / `차기 이월`: 공식 보고와 다음 기간 연결
7. `기간 전망`: 다음 달 운영 판단
