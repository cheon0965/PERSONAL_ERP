# PERSONAL_ERP 포트폴리오 프로젝트 요약

문서 기준일: 2026-05-02

이 문서는 `PERSONAL_ERP`를 포트폴리오에서 짧고 선명하게 설명하기 위한 압축 문서입니다.
작업 이력보다 프로젝트의 목적, 구현 범위, 핵심 도메인, 구조 판단, 검증 근거를 먼저 보여줍니다.

## 1. 한눈에 보는 프로젝트

`PERSONAL_ERP`는 1인 사업자와 소상공인이 매달 반복하는 재무 운영을 하나의 흐름으로 닫아 주는 월별 재무 운영 시스템입니다.

```text
기준 데이터 준비
-> 월 운영 시작
-> 계획 생성
-> 거래 수집
-> 전표 확정
-> 월 마감
-> 재무제표
-> 차기 이월
-> 다음 달 전망
```

단순히 거래를 기록하는 앱이 아니라, 실제 사업 운영에서 필요한 예정 거래, 업로드 거래, 공식 회계 기록, 마감, 보고, 다음 달 판단까지 연결하는 것이 핵심입니다.

기술 기반은 Next.js, NestJS, Prisma, MySQL입니다.
구조는 하나의 저장소와 하나의 배포 단위를 유지하는 모듈러 모놀리스이며, 회계 정합성이 중요한 흐름에만 더 엄격한 유스케이스, 포트, 어댑터 경계를 적용했습니다.

## 2. 포트폴리오 핵심 문장

> PERSONAL_ERP는 1인 사업자와 소상공인의 월별 재무 운영을 기준 데이터 준비부터 거래 수집, 전표 확정, 월 마감, 재무제표, 차기 이월, 다음 달 전망까지 연결하는 실무형 ERP 프로젝트입니다. Next.js, NestJS, Prisma, MySQL 기반으로 구현했고, 공식 회계 흐름은 수집 거래와 전표를 분리해 마감과 보고까지 추적 가능하게 설계했습니다.

## 5. 핵심 흐름

사용자는 먼저 현재 사업장과 장부를 확인합니다.
그다음 기준 데이터를 준비하고, 현재 운영월을 연 뒤 반복 규칙에서 계획 항목을 생성합니다.
실제 발생한 거래는 수기로 입력하거나 업로드 배치에서 검토해 수집 거래로 등록합니다.
검토가 끝난 수집 거래는 전표로 확정되고, 확정된 전표를 기준으로 월 마감, 재무제표, 차기 이월, 다음 달 전망이 이어집니다.

중요한 점은 수집 거래와 전표를 합치지 않았다는 것입니다.
수집 거래는 사용자가 검토할 운영 입력이고, 전표는 공식 회계 확정 결과입니다.
이 분리 덕분에 업로드 오류, 중복 후보, 계획 매칭, 확정 이후 정정을 같은 흐름 안에서 다룰 수 있습니다.

## 6. 공식 회계 경계

현재 프로젝트의 공식 회계 기준은 아래 흐름입니다.

```text
수집 거래
-> 전표 / 전표 라인
-> 마감 스냅샷
-> 재무제표 스냅샷
-> 차기 이월 기록
```

구현 이름으로는 `CollectedTransaction`, `JournalEntry`, `JournalLine`, `ClosingSnapshot`, `FinancialStatementSnapshot`, `CarryForwardRecord`가 이 흐름을 구성합니다.
레거시 Prisma `Transaction` 모델은 현재 공식 회계 기준으로 사용하지 않습니다.

이 선택으로 얻은 효과는 명확합니다.

1. 업로드 원본과 공식 회계 기록을 분리할 수 있습니다.
2. 확정 이후 수정은 원본 덮어쓰기가 아니라 반전/정정 전표로 남길 수 있습니다.
3. 마감, 재무제표, 차기 이월이 스냅샷 기준으로 설명됩니다.

## 8. 아키텍처 요약

이 프로젝트는 작은 팀이나 1인 개발 환경에서도 끝까지 구현, 검증, 배포할 수 있는 구조를 우선했습니다.
그래서 서비스별 분산보다 하나의 저장소와 하나의 실행 흐름을 유지하고, 코드 안에서는 도메인 책임이 섞이지 않도록 경계를 나눴습니다.

```text
apps/web      Next.js 기반 화면
apps/api      NestJS 기반 서버
packages/contracts
              화면과 서버가 공유하는 요청/응답 계약
packages/money
              원화 금액 계산 기준
docs          제품, 도메인, 운영, 검증 문서
```

```text
화면
-> HTTP 요청과 공용 계약
-> 서버
-> Prisma
-> MySQL
```

이 구성을 선택한 이유는 세 가지입니다.

1. 화면과 서버를 분리해 사용자 경험과 비즈니스 규칙을 독립적으로 다룰 수 있습니다.
2. 요청/응답 계약과 금액 계산 기준을 공용 패키지로 분리해 흔들리기 쉬운 경계를 고정했습니다.
3. 서버 내부는 도메인별 모듈로 나누되, 실제 분산 시스템은 도입하지 않아 개발과 배포 복잡도를 통제했습니다.

기준 데이터나 단순 조회는 읽기 쉬운 계층 구조로 두고, 수집 거래, 업로드, 전표, 마감처럼 회계 정합성이 중요한 부분에만 더 엄격한 경계를 세웠습니다.
이 방식은 포트폴리오에서 “패턴을 많이 쓴 프로젝트”가 아니라 “문제에 맞는 구조를 선택한 프로젝트”라는 점을 보여줍니다.

| 금액 처리      | `packages/money`에서 원화 계산 기준 고정     | 금액 직접 연산, 반올림, 배분 오류를 줄임        |

| 오류 표시      | 사용자 안내와 개발자 진단 분리               | 사용자는 해결 방향을 보고, 개발자는 원인을 추적 |
| 모바일 목록    | 표를 카드로 전환하고 페이지 처리 유지        | 작은 화면에서도 긴 목록을 탐색 가능하게 함      |

## 10. 운영과 보안 기준

- 대부분의 서버 기능은 인증된 사용자만 접근합니다.
- 브라우저 요청은 허용된 출처를 기준으로 제한합니다.
- 민감한 응답은 브라우저에 저장되지 않도록 `Cache-Control: no-store`를 사용합니다.
- 모든 서버 응답에는 요청 번호를 붙여 오류 추적에 사용합니다.
- 멤버/권한 변경, 운영 반출, 운영 메모는 감사 이벤트로 남깁니다.
- 사용자가 보는 오류 문구와 개발자가 확인할 오류 코드, 요청 경로, 요청 번호, 원본 응답은 분리합니다.
- 파일 조회, 생성, 수정, 삭제와 CSV 반출은 UTF-8 기준을 유지합니다.
- 외부 비밀값은 저장소 안의 일반 `.env`보다 외부 비밀값 폴더를 우선합니다.

## 11. 검증 체계

대표 검증 명령은 아래와 같습니다.

```bash
npm run check:quick
npm run test
npm run docs:check
npm run money:check
npm run test:e2e:smoke:build
npm run test:e2e
npm run test:prisma
npm run audit:runtime
```

검증은 문서, 타입, 코드 규칙, 금액 계산, 서버 요청, 화면 요청, 브라우저 흐름, 실제 MySQL 경계를 나눠 확인합니다.
특히 `docs:check`는 문서에 적힌 화면 경로와 서버 기능이 실제 코드와 어긋나지 않는지 확인합니다.
최근 문서 기준으로 화면 경로 61개, 서버 기능 136개, 문서의 경로 지도 61개를 대조합니다.

## 12. 포트폴리오에서 강조할 장면

- 레거시 거래 모델을 기준으로 삼지 않고, 공식 원장을 전표 중심으로 정리했습니다.
- 업로드 원본을 곧바로 공식 기록으로 만들지 않고, 검토와 수집 거래 단계를 거치게 했습니다.
- 원화 금액 처리를 공용 패키지로 분리해 반올림과 배분 기준을 고정했습니다.
- 사용자용 오류와 개발자용 진단 정보를 분리해 운영 추적성을 높였습니다.
- 운영 허브, 예외 처리함, 업로드 현황, 운영 메모처럼 실제 운영자가 볼 화면을 함께 구성했습니다.
