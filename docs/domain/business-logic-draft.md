# 비즈니스 로직 설계 초안

## 전표 기반 개인 재무관리 ERP (SaaS 운영 반영)

---

## 1. 문서 목적

이 문서는 본 프로젝트의 **비즈니스 로직의 시작과 끝**, **운영 흐름**, **핵심 도메인 개념**, **주요 회계 정책**, **핵심 유스케이스**, **SaaS 운영 전제**, **아키텍처 선택의 근거**를 하나의 논리로 정리하기 위한 초안이다.

이 문서의 목적은 다음과 같다.

- 프로젝트가 단순한 가계부 앱이 아니라 **전표 기반 개인 재무관리 ERP**라는 점을 명확히 한다.
- 기능 나열이 아니라 **운영 사이클 중심의 설계**로 프로젝트를 정의한다.
- 계획 데이터, 수집 데이터, 확정 데이터, 보고 데이터의 경계를 명확히 한다.
- 전표, 결산, 재무제표, 차기 이월까지 이어지는 전체 흐름을 정리한다.
- SaaS 운영을 전제로 한 **테넌트/사용자/권한 경계**를 명확히 한다.
- 현재 프로젝트 아키텍처가 왜 이렇게 설계되어야 하는지를 설명 가능하게 만든다.

---

## 2. 프로젝트 정의

### 2.1 프로젝트 정체성

**전표 기반 개인 재무관리 ERP**

### 2.2 한 줄 정의

이 프로젝트는 **개인의 월별 재무 운영을 계획하고, 실제 거래를 수집·검토·확정하여 전표로 전기하고, 결산과 재무제표 산출 및 차기 이월까지 수행하는 개인 자산관리 ERP 시스템**이다.

### 2.3 핵심 관점

이 시스템은 단순히 수입/지출을 기록하는 앱이 아니다.  
이 시스템은 다음을 수행하는 **기간 기반 운영 시스템**이다.

- 한 회계기간(월)을 시작한다.
- 반복 규칙을 바탕으로 운영 계획을 생성한다.
- 실제 거래를 수집한다.
- 수집된 거래를 검토하고 확정한다.
- 확정된 거래를 전표로 반영한다.
- 기간을 결산하고 잠근다.
- 확정된 결과로 재무제표를 산출한다.
- 결과를 다음 기간으로 이월한다.

즉, 이 프로젝트의 중심은 **거래 입력 화면**이 아니라 **월별 재무 운영 사이클**이다.

### 2.4 SaaS 운영 전제

본 프로젝트는 단일 설치형 프로그램이 아니라 **SaaS 운영**을 목표로 한다.  
따라서 비즈니스 도메인과 별도로 다음 경계가 반드시 존재해야 한다.

- **Platform User**: SaaS 플랫폼에 로그인하는 사용자
- **Tenant**: 고객 데이터와 과금/운영 경계를 가지는 테넌트(워크스페이스)
- **Tenant Membership**: 특정 사용자가 특정 테넌트에서 어떤 역할을 가지는지 나타내는 관계
- **Tenant Invitation**: 테넌트 멤버 초대 흐름
- **Tenant Subscription**: 플랜/과금/운영 제한을 관리하는 구독 경계
- **Support Access Grant**: 지원 인력의 제한적 고객 접근을 통제하는 허용 경계
- **Actor**: 사용자/지원/시스템 작업의 실행 주체를 감사 가능하게 추적하는 내부 기준
- **Ledger**: 실제 회계/운영 데이터가 귀속되는 장부 단위

핵심 원칙:

- 한 사용자는 여러 Tenant에 속할 수 있다.
- 한 Tenant는 여러 사용자를 가질 수 있다.
- 한 Ledger는 정확히 하나의 Tenant에 속한다.
- 모든 회계/운영 데이터는 Tenant 경계를 넘나들 수 없다.
- 모든 쓰기 작업은 **누가 어떤 역할로 실행했는지**를 남겨야 한다.

## 3. 범위 정의

### 3.1 포함 범위

본 프로젝트는 다음 범위를 포함한다.

- 개인 기준의 월별 재무 운영
- 반복 수입/지출 규칙 관리
- 월 운영 초안 생성
- 카드/계좌 업로드 기반 거래 수집
- 수기 입력 기반 거래 등록
- 수집 거래 검토 및 분류
- 전표 생성 및 전표 기반 집계
- 월 결산
- 기간 마감 및 잠금
- 재무제표 산출
- 차기 이월
- SaaS 기준의 Tenant/사용자/멤버십/권한 경계

### 3.2 제외 범위

본 프로젝트는 범용 회계 플랫폼이 아니며, 다음은 초기 범위에서 제외한다.

- 사용자 자유 계정과목 설계
- 사용자 자유 거래유형 설계
- 사용자 자유 분개 템플릿 설계
- 범용 승인/결재 워크플로
- 부가세/세무 전반
- 외화 회계
- **고객이 직접 다중 장부를 설계/운영하는 기능**
- 범용 ERP 수준의 규칙 엔진

주의:

- 위의 “다중 장부 제외”는 **고객 기능 범위**를 뜻한다.
- 이는 **SaaS 멀티테넌시**와 충돌하지 않는다.
- 즉, 플랫폼은 여러 Tenant를 수용할 수 있지만, 각 Tenant는 초기 범위에서 기본 1 Ledger 운영을 전제로 한다.

즉, 이 프로젝트는 **범용 회계 시스템**이 아니라 **개인 재무 운영 도메인에 특화된 전표 기반 ERP**이다.

## 4. 사용자/권한 모델

### 4.1 사용자 분류

본 프로젝트는 사용자와 내부 실행 주체를 다음 세 범주로 구분한다.

#### A. 플랫폼 역할 (Platform Role)

SaaS 전체 운영 관점의 역할이다.

- `CustomerUser`: 일반 고객 사용자
- `PlatformAdmin`: 플랫폼 운영 관리자
- `SupportReadonly`: 고객 지원용 제한 조회 사용자

#### B. 테넌트 역할 (Tenant Role)

특정 Tenant 내부에서의 역할이다.

- `Owner`: 최종 소유자, 멤버 관리/마감/재오픈/설정 가능
- `Manager`: 운영 관리자, 대부분의 재무 운영 가능
- `Editor`: 거래 입력/업로드/검토 가능, 마감/재오픈은 제한
- `Viewer`: 조회 전용

#### C. 시스템 액터 (System Actor)

플랫폼 사용자 역할과 별도로 동작하는 내부 실행 주체다.

- `System`: 배치/자동화/내부 시스템 액터로, 계획 생성·스냅샷 생성·차기 이월 같은 자동 작업의 실행 주체로 본다.

### 4.2 역할별 핵심 권한

#### Owner

- 자금수단/카테고리/반복 규칙 관리
- 월 운영 시작
- 마감/잠금
- 재오픈/정정 승인
- 멤버십 관리
- 오프닝 잔액 수정

#### Manager

- 자금수단/카테고리/반복 규칙 관리
- 거래 수집/검토/전표 확정
- 결산 수행
- 재무제표 조회
- 월 마감 검토와 결산 준비는 가능하지만, 공식 마감/잠금 실행 권한은 기본 정책에서 가지지 않는다

#### Editor

- 수기 입력
- 업로드
- 거래 검토
- 전표 확정
- 마감/재오픈/멤버 관리 불가

#### Viewer

- 대시보드 조회
- 원장/재무제표 조회
- 쓰기 작업 불가

#### SupportReadonly

- 원칙적으로 고객 데이터 변경 불가
- 조회 시 마스킹/감사 로그가 필요
- 고객 명시 동의 또는 내부 정책 기반 지원 세션에서만 접근 허용

### 4.3 권한 원칙

- 회계 데이터의 쓰기 작업은 반드시 Tenant Membership 기반으로 판정한다.
- Platform Role만으로 고객 회계 데이터를 수정할 수 없다.
- `System`은 Platform User 역할이 아니라 내부 Actor로 취급하며, 정책화된 자동 작업에서만 제한적으로 사용한다.
- 예외적 지원 접근은 별도 정책과 감사 추적을 요구한다.
- 결산/마감/재오픈은 일반 조회/입력보다 더 강한 권한이 필요하다.

### 4.4 구현 시작 전 권한 최종 고정

- 장부/원장/재무제표 조회: `Owner`, `Manager`, `Viewer`
- 거래 입력/업로드/검토/전표 확정: `Owner`, `Manager`, `Editor`
- 결산 점검: `Owner`, `Manager`
- 공식 마감/잠금: `Owner`
- 기간 재오픈: `Owner`
- `SupportReadonly`는 `Support Access Grant`가 있을 때만 제한 조회할 수 있으며, 일반 Membership 권한으로 대체하지 않는다.

## 5. 비즈니스 로직의 시작과 끝

## 5.1 시작

이 프로젝트의 시작은 단순 거래 입력이 아니다.

정확한 시작은 다음 두 단계로 구성된다.

### A. 초기 셋업

최초 사용 시 한 번 준비하는 기준이다.

- Tenant 생성
- 초기 Owner Membership 생성
- 기본 Ledger 생성
- 자금수단 준비
- 카테고리 준비
- 반복 수입/지출 규칙 준비
- 기초 계정 체계 로드
- 최초 오프닝 잔액 입력

### B. 월 운영 시작

특정 월의 운영을 시작하는 행위다.

- 대상 월 오픈
- 전월 이월값 연결
- 월 운영 초안 생성

즉, 이 프로젝트의 시작은 **월 운영 기준을 세우고 계획 데이터를 생성하는 것**이다.

## 5.2 끝

이 프로젝트의 끝은 단순 마감이 아니다.

정확한 끝은 다음과 같다.

- 해당 기간의 거래를 전표로 확정한다.
- 결산을 수행한다.
- 기간을 마감하고 잠근다.
- 마감된 전표를 기준으로 재무제표를 산출한다.
- 차기 이월 기준을 확정한다.

즉, 이 프로젝트의 끝은 **해당 기간의 재무 상태와 성과가 보고 가능한 상태로 확정되는 것**이다.

## 6. 운영 사이클

본 프로젝트의 핵심 운영 흐름은 다음 8단계로 구성된다.

### 6.1 기준 설정

월 운영을 위한 기준을 준비한다.

포함 항목:

- 자금수단 등록
- 카테고리 등록
- 반복 수입/지출 규칙 등록
- 기초 계정 체계 로드
- 최초 오프닝 잔액 입력

권한:

- Owner / Manager

### 6.2 계획 생성

반복 규칙을 바탕으로 해당 월의 운영 초안을 생성한다.

예:

- 월급 예정
- 월세 예정
- 통신비 예정
- 보험료 예정
- 카드 대금 결제 예정

중요:

- 이 단계의 데이터는 **계획 데이터**이다.
- 이 단계의 데이터는 아직 **전표가 아니다**.
- 이 단계의 데이터는 아직 **회계적 확정 사실이 아니다**.
- **카드 사용 거래 자체는 계획 생성의 기본 대상이 아니라 실제 소비 발생 후 수집/확정되는 거래**로 본다.
- **카드 대금 결제는 청구/결제 주기에 따라 계획 생성 대상이 될 수 있는 별도 거래**로 본다.

권한:

- Owner / Manager
- 시스템 배치(System) 가능

### 6.3 실제 거래 수집

실제 거래를 시스템 안으로 가져온다.

수집 경로:

- 카드 엑셀 업로드
- 계좌 내역 업로드
- 수기 거래 입력

중요:

- 업로드 거래는 기본적으로 **검토가 필요한 수집 데이터**이다.
- 수기 입력 거래는 정책에 따라 **즉시 확정 후보** 또는 **검토 대상**이 될 수 있다.
- 업로드 거래는 기본적으로 `Collected` 상태에서 시작한다.
- 직접 입력 거래는 기본적으로 `ReadyToPost` 상태에서 시작하며, 시스템 정책에 따라 `Collected` 또는 `Reviewed` 상태를 거치도록 할 수 있다.
- 수집 데이터는 아직 전표가 아니다.

권한:

- Owner / Manager / Editor

### 6.4 검토 및 분류

수집 거래를 확정 가능한 상태로 정리한다.

포함 항목:

- 중복 검사
- 형식 오류 검사
- 초안과의 매칭
- 카테고리 분류
- 자금수단 확인
- 거래유형 판별
- 전표 생성 가능성 검토

권한:

- Owner / Manager / Editor

### 6.5 전표 확정(전기)

검토 완료된 거래를 전표로 확정한다.

포함 항목:

- 전표 헤더/라인 생성
- 차변/대변 균형 검증
- 계정 반영
- 거래유형별 자동분개 수행
- 이후 집계/보고/결산은 전표 기준으로 처리

중요:

- **전표가 회계적 진실의 단일 원천**이다.

권한:

- Owner / Manager / Editor
- 마감 후 정정전표는 Owner / Manager만 가능

### 6.6 결산

해당 월을 마감하기 전에 최종 검토를 수행한다.

포함 항목:

- 미확정 거래 점검
- 누락 거래 점검
- 잔액 검토
- 중복 전표 점검
- 월 집계 검토
- 마감 가능 여부 확인

중요:

- 결산은 **공식 재무제표 산출 전의 검증/정리 단계**다.
- 결산 단계에서는 가결산/미리보기 리포트가 가능할 수 있다.

권한:

- Owner / Manager

### 6.7 마감 및 잠금

해당 월을 공식 확정 상태로 전환한다.

포함 항목:

- 기간 상태를 `Locked`로 변경
- 일반 수정 금지
- 마감 후 정정 정책 적용
- 차기 이월 기준값 확정

권한:

- Owner

### 6.8 재무제표 산출 및 차기 이월

마감된 전표를 기준으로 재무제표를 산출하고 다음 기간 시작값을 확정한다.

산출 항목:

- 개인 재산상태표
- 월간 손익보고서
- 현금흐름 요약표
- 순자산 변동표

이후:

- 자산/부채/순자산 잔액 이월
- 다음 월 운영 시작 기준 생성

권한:

- 산출: Owner / Manager / Viewer
- 차기 이월 실행: Owner / Manager 또는 System

## 8. 핵심 회계 정책

이 프로젝트는 개인 자산관리 도메인에 맞는 제한적 회계 정책을 가진다.

### 8.1 기본 원칙

- 모든 확정 거래는 전표로 반영된다.
- 전표는 차변/대변 균형을 만족해야 한다.
- 재무제표는 전표를 기준으로 산출한다.
- 계획 데이터와 확정 데이터는 분리한다.
- 손익 거래와 비손익 거래를 구분한다.
- 권한 정보는 회계 결과를 바꾸지 않지만, 회계 변경 가능 범위를 결정한다.

### 8.2 손익 거래

손익에 영향을 주는 거래다.

예:

- 급여 수입
- 부수입
- 일반 소비 지출
- 수수료 비용
- 생활비
- 고정비

### 8.3 비손익 거래

손익에 직접 반영되지 않는 거래다.

예:

- 자산 간 이체
- 카드 대금 결제
- 대출 원금 상환
- 예금 이동
- 투자 계좌 간 자금 이동

원칙:

- 비손익 거래는 손익표에 직접 반영하지 않는다.

### 8.4 카드 사용과 카드 대금 결제

이 프로젝트는 다음 정책을 따른다.

- **카드 사용 시 비용/부채를 인식한다.**
- **카드 대금 결제 시에는 카드미지급금 감소를 처리하며 손익에는 재반영하지 않는다.**

즉:

- 카드 사용 = 비용 발생
- 카드 대금 결제 = 부채 상환

운영 연결 원칙:

- **카드 사용 거래는 실제 소비 발생 시점의 거래로 수집/검토/전표 확정의 대상**이다.
- **카드 대금 결제는 청구/결제 주기에 따라 계획 생성 및 실제 결제 반영의 대상**이다.
- 따라서 카드 사용과 카드 대금 결제는 서로 다른 거래유형이며, 동일한 손익 이벤트로 중복 인식하지 않는다.

### 8.5 이체

자산 간 이체는 자산 재배치로 본다.

예:

- 현금 → 예금
- 예금 → CMA
- 입출금 계좌 간 이동

원칙:

- 이체는 손익으로 보지 않는다.

### 8.6 최초 오프닝 잔액

프로젝트 첫 시작 시에는 다음 데이터를 수기 입력해야 한다.

- 현금 잔액
- 계좌 잔액
- 카드 미지급금
- 대출 잔액
- 투자자산 초기 잔고

원칙:

- 최초 오프닝 잔액은 프로젝트의 시작 전 재무 상태를 시스템 안으로 들여오는 행위다.
- 최초 오프닝 잔액은 월 운영 시작의 기초가 된다.
- 최초 오프닝 잔액 입력은 Owner 권한을 기본으로 한다.

### 8.7 금액/통화/반올림 정책

초기 구현은 KRW 단일 통화 장부를 기준으로 한다.

원칙:

- 영속 금액은 모두 `1원` 단위 정수로 저장한다.
- 계산 중간값은 더 높은 정밀도를 사용할 수 있지만, `JournalLine`, `ClosingSnapshot`, `FinancialStatementSnapshot` 저장 전에는 반올림을 확정한다.
- 기본 반올림 기준은 `HALF_UP`(사사오입)이다.
- 공식 회계 금액은 라인 단위 반올림 후 저장된 값을 기준으로 하며, 합계/마감/재무제표는 해당 확정 금액의 합으로 계산한다.
- 분할/배분 계산에서 잔차가 생기면 마지막 라인 또는 명시적 조정 라인에 잔여 차이를 반영해 총액을 보존한다.
- 화면 표시는 포맷팅만 담당하며, 보고 화면에서 다시 다른 반올림 규칙을 적용해 원장과 불일치시키지 않는다.

## 9. 초안의 생명주기

초안은 단순 예정 데이터가 아니라 운영 관리의 대상이다.

초안 상태는 다음을 가진다.

- `Draft`: 생성된 예정 상태
- `Matched`: 실제 거래와 연결된 상태
- `Confirmed`: 전표 확정까지 완료된 상태
- `Skipped`: 의도적으로 미반영 처리한 상태
- `Expired`: 기간 종료로 효력을 상실한 상태

원칙:

- 초안은 전표가 아니다.
- 초안은 실제 거래 또는 수동 판단을 통해 종료 상태로 이동한다.
- 초안은 기간을 넘겨 자동 이월하지 않는다. 이월이 필요하면 다음 기간에서 새 계획으로 재생성한다.
- 초안의 `Confirmed`는 **해당 초안이 실제 거래와 연결되고, 그 결과 전표 확정까지 완료되어 운영상 종료된 상태**를 뜻한다.
- 거래 상태의 `Posted`는 **전표가 실제로 반영되어 회계적으로 확정된 상태**를 뜻한다.

## 10. 거래 입력 및 확정 정책

### 10.1 외부 유입 거래

예:

- 카드 엑셀 업로드
- 계좌 내역 업로드

정책:

- 기본적으로 검토가 필요하다.
- 중복 검사와 분류 과정을 거친 후 전표 확정 대상으로 간다.
- 업로드 실행자와 검토 실행자는 감사 추적 가능해야 한다.

### 10.2 직접 입력 거래

예:

- 현금 지출
- 현금 수입
- 누락 거래 수기 등록

정책:

- 기본적으로 사용자가 직접 입력한 만큼 즉시 확정 후보로 본다.
- 직접 입력 거래는 기본적으로 `ReadyToPost` 상태에서 시작한다.
- 단, 시스템 정책에 따라 `Collected` 또는 `Reviewed` 상태를 거쳐 검토 대기 상태로 둘 수 있다.
- 수기 입력의 작성자와 수정자는 감사 추적 가능해야 한다.

### 10.3 업로드 최소 구현 범위

- 1차 구현은 `카드 엑셀 1종`과 `계좌 CSV 1종`을 기준 범위로 고정한다.
- 파싱 실패 행은 오류 상태의 `ImportedRow`로 남기고, `CollectedTransaction`으로 승격하지 않는다.
- 원본 파일 메타데이터와 원본 행 payload는 추적 목적으로 보존한다.
- 업로드 거래는 즉시 전표 생성하지 않고 반드시 검토 단계를 거친다.
- 중복 후보 판정은 `sourceFingerprint`(거래일시, 금액, 자금수단, 정규화된 설명, 원본 출처 조합)를 기준으로 수행한다.

역할:

- 마감 결과를 다음 월의 시작값으로 연결

핵심 규칙:

- 자산/부채/순자산 잔액 이월
- 손익 계정은 기간 종료 처리
- 다음 월 오프닝 기준 생성

권한:

- Owner / Manager / System

### 12.12 거래 확정 트랜잭션 경계

하나의 `CollectedTransaction`을 `Posted`로 전환하는 단위는 하나의 write transaction으로 본다.

같이 묶는 항목:

- 대상 `CollectedTransaction` 상태 변경
- `JournalEntry` / `JournalLine` 생성
- 매칭된 `PlanItem` 상태 갱신
- Actor 기반 감사 필드 기록

원칙:

- 위 단계 중 하나라도 실패하면 전체를 롤백한다.
- 하나의 `CollectedTransaction`은 하나의 `JournalEntry`만 생성할 수 있다.
- `ImportBatch`는 파싱/수집 단위이지 회계 확정 단위가 아니다.

### 12.13 기간 마감 트랜잭션 경계

공식 마감/잠금은 하나의 `AccountingPeriod`를 `Locked`로 확정하는 write transaction으로 본다.

같이 묶는 항목:

- 마감 가능성 검증
- `AccountingPeriod` 상태 변경 (`Closing` -> `Locked`)
- `PeriodStatusHistory` 기록
- `ClosingSnapshot` 생성
- `BalanceSnapshotLine` 생성

원칙:

- 위 단계 중 하나라도 실패하면 Period는 `Locked`가 되면 안 된다.
- `CarryForwardRecord` 생성과 다음 기간 오프닝 생성은 마감 성공 후 별도 use case로 수행한다.
- `FinancialStatementSnapshot` 생성도 마감 후 재실행 가능한 후처리로 둔다.

### 12.14 재무제표 스냅샷과 차기 이월 실행 시점

- 공식 `FinancialStatementSnapshot`은 Period가 `Locked`된 뒤 생성한다.
- 스냅샷 생성 실패가 Period 잠금 자체를 되돌리지는 않으며, 재시도 가능해야 한다.
- `CarryForwardRecord`는 `Locked`된 `ClosingSnapshot`과 `BalanceSnapshotLine`을 근거로 별도 실행한다.
- 차기 이월은 마감의 파생 작업이지만, 마감 결과와 불일치하면 안 된다.

### 12.15 업로드 확정 단위

- 1차 구현의 기본 확정 단위는 거래 1건이다.
- `ImportBatch`는 파싱과 후보 수집의 경계이지 자동 일괄 전표 확정 경계가 아니다.
- 운영자는 배치 화면에서 여러 건을 검토할 수 있지만, 실제 `JournalEntry` 생성은 건별 검증을 통과한 거래에 대해서만 수행한다.
- 묶음 확정은 후속 확장 항목으로 둔다.

## 13. 상태 변화 원칙

### 13.1 거래 상태

거래 상태는 다음과 같이 정의한다.

- `Collected`: 외부 유입 또는 정책상 검토가 필요한 상태로 수집된 거래
- `Reviewed`: 형식 검증, 중복 검사, 분류, 초안 매칭 등을 마친 상태
- `ReadyToPost`: 전표 생성이 가능하며 확정 대기 중인 상태
- `Posted`: 전표가 실제로 생성되어 회계적으로 반영된 상태
- `Corrected`: 정정전표 또는 조정전표를 통해 원거래의 효과가 수정된 상태
- `Locked`: 마감 또는 잠금 정책상 일반 변경이 금지된 상태

시작 원칙:

- 업로드 거래는 기본적으로 `Collected`에서 시작한다.
- 직접 입력 거래는 기본적으로 `ReadyToPost`에서 시작한다.
- 단, 정책에 따라 직접 입력 거래도 `Collected` 또는 `Reviewed` 상태를 거칠 수 있다.

### 13.2 기간 상태

기간 상태는 다음과 같이 정의한다.

- `Open`: 월 운영이 시작되어 거래 수집과 확정이 가능한 상태
- `InReview`: 마감 전 검토와 조정이 진행되는 상태
- `Closing`: 결산 점검과 마감 실행 중인 상태
- `Locked`: 결산과 마감이 완료되어 일반 수정이 금지된 최종 상태

원칙:

- 본 프로젝트에서 기간 상태는 `Closed`와 `Locked`를 분리하지 않는다.
- **마감 완료 상태는 곧 `Locked` 상태**로 본다.
- 재오픈이 필요하면 별도 예외 정책으로 처리하며, 기본 운영 상태로는 두지 않는다.

### 13.3 핵심 원칙

- 계획은 계획으로 존재한다.
- 수집은 수집으로 존재한다.
- 확정은 전표로 남는다.
- 마감은 기간을 잠근다.
- 보고는 마감 전표 기준으로 생성된다.
- 모든 상태 전이는 Tenant 경계 안에서만 일어난다.
- 모든 쓰기 전이는 실행 주체(Actor)를 남긴다.

## 14. 구현 시작 기준 시나리오

### 14.1 첫 대표 시나리오

1차 구현의 대표 시나리오는 **신규 Tenant의 첫 월을 열고, 카드 사용 거래 1건을 업로드해 전표 확정한 뒤, 월을 마감하고 재무제표와 차기 이월 기준까지 만드는 흐름**으로 고정한다.

흐름:

1. Owner가 Tenant, Ledger, FundingAccount, Category, RecurringRule, OpeningBalanceSnapshot을 준비한다.
2. Owner 또는 Manager가 대상 월을 `Open`한다.
3. System이 월 초안(`PlanItem`)을 생성한다.
4. Editor가 카드 엑셀을 업로드해 `ImportBatch`, `ImportedRow`, `CollectedTransaction`을 만든다.
5. Editor 또는 Manager가 거래를 검토해 거래유형, 카테고리, 자금수단을 확정하고 필요 시 초안과 매칭한다.
6. 시스템이 해당 거래를 `Posted`로 전환하면서 `JournalEntry`, `JournalLine`을 생성하고, 연결된 초안이 있으면 `Confirmed`로 종료한다.
7. Manager가 결산 점검을 수행한다.
8. Owner가 기간을 `Locked`로 마감한다.
9. 마감 후 System 또는 권한 있는 사용자가 `FinancialStatementSnapshot`을 생성한다.
10. System 또는 권한 있는 사용자가 `CarryForwardRecord`와 다음 월 오프닝 기준을 생성한다.

### 14.2 첫 시나리오의 필수 엔티티

- `Ledger`
- `AccountingPeriod`
- `FundingAccount`
- `Category`
- `RecurringRule`
- `PlanItem`
- `ImportBatch`
- `ImportedRow`
- `CollectedTransaction`
- `JournalEntry`
- `JournalLine`
- `OpeningBalanceSnapshot`
- `ClosingSnapshot`
- `BalanceSnapshotLine`
- `CarryForwardRecord`
- `FinancialStatementSnapshot`

### 14.3 첫 시나리오의 실패 케이스

- 첫 월인데 `OpeningBalanceSnapshot`이 없으면 월 운영 시작을 차단한다.
- 업로드 거래가 기존 `sourceFingerprint`와 충돌하면 중복 후보로 보류하고 자동 전표 확정을 금지한다.
- 손익 거래인데 Category가 비어 있으면 `ReadyToPost`로 올리지 않는다.
- `Owner`가 아닌 사용자가 공식 마감/잠금을 시도하면 거부하고 감사 로그를 남긴다.
- `Locked` 기간의 거래를 overwrite 수정하려고 하면 원본 수정 대신 `Reversal` 또는 `Correction/Adjustment` 전표 흐름으로 유도한다.

## 15. 아키텍처와의 연결 근거

### 14.1 왜 모듈러 모놀리스인가

이 프로젝트의 핵심 흐름은 강하게 연결되어 있다.

- 계획 생성
- 거래 수집
- 검토/분류
- 전표 확정
- 결산
- 마감
- 재무제표 산출
- 차기 이월

이 흐름은 강한 정합성과 일관된 상태 전이를 요구한다.  
초기부터 분산 서비스로 쪼개기보다 **하나의 강한 쓰기 모델과 단일 도메인 경계** 안에서 유지하는 것이 더 합리적이다.

### 14.2 왜 외부 의존성만 강하게 분리하는가

교체 가능성이 큰 것은 외부에 있다.

예:

- 파일 업로드 파서
- 인증
- 현재 시각
- 스토리지
- 알림
- 향후 금융 연동

따라서 외부 의존성만 포트/어댑터/DI로 강하게 분리하고, 내부 도메인은 과도하게 쪼개지 않는다.

### 14.3 왜 읽기 모델을 분리하는가

전표 확정과 결산은 정합성이 핵심이고, 재무제표/대시보드/리포트는 집계와 조회 성능이 핵심이다.

따라서:

- 쓰기 모델은 전표/결산 중심으로 유지하고
- 읽기 모델은 projection/read service 중심으로 분리하는 것이 합리적이다.

즉, 현재 아키텍처는 보기 좋기 위해서가 아니라 **운영 사이클을 버티기 위해 설계된 구조**다.

### 14.4 왜 SaaS 경계를 도메인 밖에 따로 둬야 하는가

이 프로젝트의 핵심 회계 도메인은 Ledger/Period/Journal 중심으로 강하게 묶여 있다.  
반면 사용자 인증, 테넌트, 멤버십, 지원 접근, 구독 상태는 **플랫폼 운영 경계**에 가깝다.

따라서:

- 회계 도메인은 회계 정합성 중심으로 유지하고
- SaaS 도메인은 테넌시/권한/운영 통제 중심으로 분리하는 것이 맞다.

이렇게 해야 회계 모델이 불필요하게 오염되지 않으면서도 SaaS 운영을 안정적으로 수용할 수 있다.

## 16. 최종 정리

### 이 프로젝트의 시작

**월 운영 기준을 세우고 계획 데이터를 생성하는 것**

### 이 프로젝트의 중심

**실제 거래를 검토하고 전표로 확정하는 것**

### 이 프로젝트의 끝

**마감된 전표를 기준으로 재무제표를 산출하고 차기 이월 기준을 확정하는 것**

### 이 프로젝트가 ERP인 이유

- 단순 기록 앱이 아니다.
- 계획, 수집, 검토, 전기, 결산, 마감, 보고가 하나의 흐름으로 이어진다.
- 전표가 단일 진실 원천이다.
- 기간 단위 운영과 잠금이 있다.
- 재무제표까지 산출된다.

### 이 프로젝트가 SaaS로도 성립하는 이유

- 회계 데이터는 Tenant 경계 안에서 격리된다.
- 사용자와 권한은 Membership으로 관리된다.
- 마감/정정/조회 권한이 역할별로 분리된다.
- 플랫폼 운영자와 고객 사용자의 권한이 분리된다.

즉, 본 프로젝트는 **개인의 월별 재무 운영을 전표 기반으로 관리하고 재무제표로 닫는 개인 재무관리 ERP**이면서, 동시에 **테넌트/사용자/권한 경계를 갖춘 SaaS 운영형 시스템**이다.
