# PERSONAL_ERP PPT 발표 노트

이 문서는 PPT 본문에 넣기에는 긴 면접 답변을 30초 단위로 준비하기 위한 메모입니다.
PPT 본문은 제출용 흐름을 보여주고, 상세 설명은 이 노트에서 보강합니다.

## 핵심 발표 흐름

1. 이 프로젝트는 단순 장부가 아니라 월별 재무 운영을 닫는 ERP입니다.
2. 핵심 도메인 결정은 수집 거래와 전표를 분리한 것입니다.
3. 구조는 실무형 modular monolith이며, 핵심 쓰기 모델에만 선택적으로 use case/port/adapter 경계를 적용했습니다.
4. 화면은 기능 수를 보여주기보다 월 운영 사이클에서 각 화면이 맡는 역할을 보여줍니다.
5. 검증은 문서, 타입, 테스트, 보안 게이트, 배포 확인을 함께 관리하는 방식으로 구성했습니다.

## 아키텍처 설명 순서

아키텍처는 폴더 구조부터 설명하지 않고, 왜 그런 구조가 필요했는지부터 설명합니다.

1. 월 운영은 열림, 검토, 확정, 마감, 이월이라는 상태 흐름을 갖습니다.
2. 공식 숫자가 흔들리면 보고와 다음 달 기준이 함께 깨집니다.
3. 업로드, 수기 입력, 반복 계획, 차량/부채 비용처럼 입력 경로가 여러 개입니다.
4. 그래서 원장 쓰기 흐름은 강하게 보호하고, 보고/전망은 읽기 조합으로 분리했습니다.
5. 이 판단을 코드에서는 modular monolith, 선택적 use case/port/adapter, read model/projection, shared contract/money package로 표현했습니다.

## 예상 질문과 30초 답변

### 왜 수집 거래와 전표를 분리했나요?

수집 거래는 사용자가 검토하고 보정하는 운영 입력이고, 전표는 마감과 보고의 기준이 되는 공식 회계 기록입니다.
둘을 합치면 업로드 오류, 중복 후보, 계획 매칭, 확정 이후 정정이 모두 같은 모델에 섞입니다.
그래서 입력 후보는 수집 거래로 남기고, 확정된 결과만 전표와 전표 라인으로 승격했습니다.

### 왜 microservice가 아니라 modular monolith인가요?

이 프로젝트는 1인 개발과 공개 데모 배포까지 끝내는 것이 중요했습니다.
서비스별 DB, broker, gateway를 먼저 넣으면 운영 복잡도가 제품 완성을 밀어냅니다.
대신 하나의 저장소와 배포 단위를 유지하면서, 원장·반복 자동화·보고/전망·운영 지원 같은 책임 경계는 문서와 코드 구조로 설명 가능하게 유지했습니다.

### 어디에는 강한 구조를 적용했고, 어디에는 단순하게 두었나요?

돈, 전표, 마감, 권한처럼 실패 비용이 큰 쓰기 흐름은 application/domain/infrastructure 구조로 승격했습니다.
`collected-transactions`, `accounting-periods`, `import-batches`, `journal-entries`, `auth`, `admin`, `plan-items`, `financial-statements`, `carry-forwards`가 대표입니다.
반대로 기준 데이터나 설정처럼 CRUD 성격이 강한 곳은 controller-service-repository-mapper 구조를 유지했습니다.

### read model은 왜 별도로 설명하나요?

대시보드, 자금수단별 현황, 전망, 운영 허브는 쓰기 규칙을 소유하는 곳이 아닙니다.
여러 컨텍스트의 데이터를 읽어 화면용 요약과 판단 근거를 만드는 영역입니다.
그래서 write domain 패턴을 억지로 적용하지 않고 read service, read repository, projection 성격을 드러내는 방향으로 정리했습니다.

### LLM을 사용했다면 품질 리스크는 어떻게 막았나요?

LLM을 사용한 프로젝트일수록 변경 범위를 작게 나누고 검증 가능한 경계를 두는 것이 중요하다고 봤습니다.
그래서 shared contract, money package, module public boundary, 문서 표면 검사, money 연산 검사, request API 테스트를 함께 사용했습니다.
LLM이 코드를 제안하더라도 최종 기준은 테스트, 문서 정합성, 도메인 규칙이 통과하는지로 판단했습니다.

### 이 프로젝트가 상용 프로젝트와 다른 한계는 무엇인가요?

개인 프로젝트라서 팀 협업, 장기 장애 대응, 실제 고객 트래픽, 백업/모니터링 자동화 경험은 제한적입니다.
대신 그 한계를 숨기지 않고 PPT의 한계/다음 단계에 분리했습니다.
현재 프로젝트가 보여주는 강점은 제품 흐름, 도메인 경계, 검증 가능한 구조, 공개 배포까지 한 사람이 끝까지 닫았다는 점입니다.

### 가장 중요한 설계 결정 3개는 무엇인가요?

첫째, 수집 거래와 전표를 분리해 운영 입력과 공식 기록을 나눴습니다.
둘째, 전체를 microservice로 나누지 않고 modular monolith로 두되 핵심 쓰기 모델만 강하게 구조화했습니다.
셋째, dashboard/forecast 같은 영역은 쓰기 모델이 아니라 read model로 두어 책임을 분리했습니다.

### 왜 모든 service를 interface로 분리하지 않았나요?

외부 기술 경계가 아닌 내부 호출까지 모두 interface로 만들면 코드량과 조립 비용이 늘어납니다.
이 프로젝트의 목표는 DI 사용량을 늘리는 것이 아니라, 변경 위험이 큰 경계를 명확히 하는 것입니다.
그래서 Prisma, 시간, 파일, 메일, 저장소 같은 외부 의존성 중심으로만 port/adapter를 사용했습니다.

## 발표 전 확인 체크리스트

- PPT 재생성 후 슬라이드 수가 33장인지 확인합니다.
- 스크린샷 구간의 화면 캡처가 최신 화면과 맞는지 확인합니다.
- `docs/VALIDATION_NOTES.md`의 최신 검증 수치와 PPT 검증 슬라이드의 수치가 맞는지 확인합니다.
- 발표 직전 `npm run docs:check`, `npm run test`, `npm run audit:runtime` 실행 결과를 최신화합니다.
- Docker가 필요한 Prisma/E2E/보안 검증은 실행 가능 환경과 별도 표시 여부를 확인합니다.

모든 문서/텍스트 파일 작업은 UTF-8 인코딩 기준을 유지합니다.
