# PERSONAL_ERP 포트폴리오 아키텍처 가이드

## 문서 목적

이 프로젝트가 어떤 목적의 포트폴리오인지, 왜 이런 구조 판단을 했는지, 현재 코드가 그 목적에 얼마나 맞는지를 설명합니다.

짧게 말하면 이 프로젝트는 `실무형 모듈러 모놀리스`를 기반으로 하고, `핵심 도메인에만 선택적으로 클린/헥사고날 성향`을 적용한 1인 사업자·소상공인용 월별 재무 운영 시스템 포트폴리오입니다.

## 쉬운 3줄 요약

1. 이 프로젝트는 웹, API, 공용 계약을 나눈 `모듈러 모놀리스`입니다.
2. 모든 곳을 복잡하게 분리하지 않고, `collected-transactions`, `accounting-periods`, `import-batches`, `journal-entries`처럼 회계 정합성이 중요한 쓰기 모델에 경계를 더 엄격하게 세웠습니다.
3. 목표는 “패턴을 많이 쓴 프로젝트”가 아니라 “왜 이 구조를 선택했는지 설명할 수 있는 프로젝트”입니다.

## 이 프로젝트가 보여주려는 것

- 1인 개발로도 끝까지 완성 가능한 구조 판단을 할 수 있다.
- 협업을 고려해 기능 경계, 계약, 테스트, 문서 우선순위를 설계할 수 있다.
- 클린 아키텍처를 맹목적으로 적용하지 않고, 필요한 곳에만 선택적으로 적용할 수 있다.
- 외부 의존성은 명확히 다루되, 프로젝트 내부까지 과하게 의존성 주입하지 않는 절제된 설계를 할 수 있다.

## 이 프로젝트의 핵심 판단 원칙

## 용어를 쉬운 말로 바꾸면

- `모듈러 모놀리스`
  하나의 저장소와 하나의 배포 단위 안에 있지만, 기능별 경계가 분명한 구조
- `공용 계약`
  웹과 API가 같은 요청/응답 타입을 함께 쓰는 방식
- `선택적 클린/헥사고날 적용`
  모든 모듈을 똑같이 바꾸지 않고, 중요한 곳에만 유스케이스/포트/어댑터 경계를 도입하는 방식
- `외부 의존성 경계`
  데이터베이스, JWT, argon2, 시간 공급처럼 “프로젝트 바깥 기술”과 만나는 지점

## MSA 대비 컨텍스트 맵

중요한 점은 이것입니다.
이 컨텍스트 맵은 `서비스 분리 설계도`가 아니라 `책임 지도`입니다.

즉, 지금도 여전히 이 프로젝트는 아래 기준을 유지합니다.

- 하나의 저장소
- 하나의 배포 단위
- 하나의 데이터베이스
- 빠른 로컬 개발 루프

대신 “나중에 분리한다면 어디를 기준으로 나눌 수 있는가”를 지금부터 설명 가능한 상태로 고정합니다.

```text
[ 플랫폼과 공용 계약 ] -> 모든 컨텍스트의 계약/공통 인프라 지원
[ 인증과 접근 제어 ] -> 모든 컨텍스트의 인증/요청 주체 기준선

[ 반복 자동화 ] ------ 참조만 ------> [ 원장 ]
[ 원장 ] --------------- 읽기 -------> [ 보고와 전망 ]
[ 반복 자동화 ] --------- 읽기 -------> [ 보고와 전망 ]
[ 운영 자산/보장/부채 ] -- 수집/상환 연계 --> [ 원장 ]
[ 운영 자산/보장/부채 ] ---- 읽기 -------> [ 보고와 전망 ]
[ 운영 지원 ] ------------ 읽기 -------> [ 원장/사업장 운영 관리 ]
```

| 컨텍스트            | 현재 모듈                                                                                                                                                                                                                                   | 소유 책임                                                             | 읽기/쓰기 성격   | 나중에 분리한다면                                 |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ---------------- | ------------------------------------------------- |
| 인증과 접근 제어    | `auth`, `common/auth`                                                                                                                                                                                                                       | 로그인, 토큰, 요청 주체 인증, 회원가입, 이메일 인증, 계정 보안 경계   | 횡단 관심        | 공통 인증 계층                                    |
| 사업장 운영 관리    | `workspace-settings`, `admin`, `navigation`, `common/infrastructure/operational`                                                                                                                                                            | 사업장 설정, 멤버/권한 정책, 감사 이벤트, 데이터베이스 메뉴 트리      | 쓰기 + 운영 관리 | 독립 도메인 후보                                  |
| 원장                | `funding-accounts`, `categories`, `account-subjects`, `ledger-transaction-types`, `reference-data-readiness`, `accounting-periods`, `collected-transactions`, `journal-entries`, `import-batches`, `financial-statements`, `carry-forwards` | 기준 데이터, 월 운영, 수집/전표, 업로드, 공식 보고, 차기 이월         | 핵심 쓰기        | 가장 강한 분리 후보                               |
| 반복 자동화         | `recurring-rules`, `plan-items`                                                                                                                                                                                                             | 반복 규칙 정의, 계획 항목 생성과 추적성                               | 쓰기 + 원장 참조 | 원장 다음 후보                                    |
| 운영 자산/보장/부채 | `vehicles`, `insurance-policies`, `liabilities`                                                                                                                                                                                             | 운영비 성격의 자산/보장 도메인, 부채 상환 일정과 원금/이자 분리 경계  | 쓰기 + 원장 연계 | 별도 도메인 후보                                  |
| 운영 지원           | `operations-console`                                                                                                                                                                                                                        | 운영 체크리스트, 예외, 마감/업로드 현황, 시스템 상태, 알림, 반출/메모 | 읽기/운영 보조   | 읽기 서비스 보조 후보                             |
| 보고와 전망         | `dashboard`, `funding-account-status`, `forecast`                                                                                                                                                                                           | 여러 컨텍스트를 읽어 요약, 자금수단별 분석, 예측을 만든다             | 읽기/조합 중심   | 프런트엔드 전용 백엔드(BFF) 또는 읽기 서비스 후보 |
| 플랫폼과 공용 계약  | `packages/contracts`, `packages/money`, env, Prisma, health, 공통 외부 의존성 조립                                                                                                                                                          | 계약 원천, 금액 값 기준, 런타임 기반선                                | 지원 계층        | 공통 플랫폼 성격                                  |

여기서 `원장(Ledger)`은 현재 코드베이스가 회계 기준선을 읽을 때 사용하는 핵심 컨텍스트 이름입니다.
현재 API 모듈명은 `funding-accounts`, `accounting-periods`, `insurance-policies`, `liabilities`, `vehicles`, `recurring-rules`, `plan-items`, `collected-transactions`, `import-batches`, `journal-entries`, `financial-statements`, `carry-forwards`, `operations-console`, `dashboard`, `funding-account-status`, `forecast` 등이 있고, 웹 화면/라우트는 사용자 경험 관점의 축약 이름으로 `transactions`, `periods`, `imports`, `reference-data`, `insurances` 등을 함께 사용합니다.
회계 도메인의 상세 엔티티/상태/권한 기준은 `docs/domain/business-logic-draft.md`, `docs/domain/core-entity-definition.md`를 우선합니다.

면접이나 포트폴리오 설명에서는 이렇게 정리하면 됩니다.

- 지금은 `모듈러 모놀리스`가 맞다.
- 하지만 경계는 이미 `원장`, `반복 자동화`, `보고와 전망` 기준으로 읽히게 설계했다.
- 따라서 실제 MSA로 가지 않더라도 `MSA 대비 수준`은 설명할 수 있다.

## 여기서 고정하는 금지선

### 허용되는 의존 방향

- 웹은 HTTP와 `packages/contracts`를 통해서만 API와 연결합니다.
- 인증과 접근 제어는 인증과 요청 주체 기준선만 제공하고, 다른 비즈니스 도메인의 규칙 소유자가 되지 않습니다.
- 반복 자동화는 원장의 계정/카테고리 참조 상태를 읽을 수 있습니다.
- 보고와 전망은 원장, 반복 자동화, 운영 자산/보장/부채를 읽어 요약과 예측을 만듭니다.
- 플랫폼과 공용 계약은 공통 인프라와 계약을 제공하지만 도메인 규칙을 소유하지 않습니다.

### 금지되는 직접 참조

- `dashboard`, `funding-account-status`, `forecast`는 `collected-transactions`, `recurring-rules`의 생성/검증 규칙을 직접 소유하지 않습니다.
- 어떤 모듈도 다른 모듈의 저장소, 어댑터, 컨트롤러를 직접 가져다 쓰는 것을 기본 규칙으로 삼지 않습니다.
- `packages/contracts`에는 앱 전용 구현 코드나 비즈니스 로직을 넣지 않습니다.
- 새로운 메시지 브로커, 아웃박스, 게이트웨이, 서비스 분리는 별도 ADR 없이 도입하지 않습니다.
- 설명 가치가 없는 `shared domain`, `common business` 폴더로 서로 다른 도메인 규칙을 섞지 않습니다.

### 왜 이 금지선이 중요한가

- 경계가 문서에만 있고 코드에서는 무너지는 상황을 막을 수 있습니다.
- 과한 MSA 흉내 없이도 “어디까지 준비돼 있는가”를 분명하게 보여줄 수 있습니다.
- 이후 P1, P2 작업이 `의미 있는 경계 강화`인지 `보기 좋은 추상화`인지 쉽게 판정할 수 있습니다.

## 현재 코드가 실제로 어떻게 구성되어 있는가

### 1. 웹은 `app / features / shared` 구조를 사용한다

- `app/`은 라우팅과 페이지 진입점을 담당합니다.
- `features/`는 기능별 화면과 API 호출 로직을 묶습니다.
- `shared/`는 공용 UI, 인증 상태, 조회 도우미 함수, 레이아웃, 테마를 담당합니다.

이 구조는 프런트엔드에서 지나친 아키텍처 실험보다 `기능 응집도`와 `재사용 가능한 공통 계층`을 우선한 선택입니다.

### 2. API는 기본적으로 기능별 모듈 구조를 사용한다

대부분의 모듈은 아래 흐름을 따릅니다.

```text
컨트롤러 -> 서비스 -> 저장소 -> 변환기/계산기
```

이 구조는 `funding-accounts`, `categories`, `reference-data-readiness`, `vehicles`, `workspace-settings`, `navigation` 같은 영역에서 여전히 유효합니다.
특히 기본 생성·조회·수정·삭제 성격이 강하거나 계산 로직만 분리하면 충분한 영역에서는 이 방식이 더 읽기 쉽고 유지보수도 쉽습니다.

### 3. 핵심 쓰기 모델과 운영 핵심 모듈만 더 엄격한 구조로 승격했다

`collected-transactions`와 `recurring-rules`를 시작으로, `accounting-periods`, `import-batches`, `journal-entries`, `auth`, `admin`, `insurance-policies`, `plan-items`, `financial-statements`, `carry-forwards`까지 핵심 모듈을 아래 흐름으로 승격했습니다.

```text
컨트롤러 -> 유스케이스 -> 포트 -> 어댑터
```

그리고 내부 규칙은 별도 `domain` 파일이나 순수 함수로 분리합니다.

이렇게 한 이유는 단순합니다.

- 이 모듈들은 프로젝트의 핵심 쓰기 모델이거나 운영 관리의 핵심 경계입니다.
- 접근 범위 검증, 입력 규칙, 생성 흐름, 상태 전이, 보호 정책이 중요합니다.
- 테스트와 설명 가치가 높습니다.
- 다른 모든 모듈에 같은 복잡도를 강요할 필요는 없습니다.

`auth`는 회원가입, 이메일 인증, 로그인, 세션, 계정 보안 등 10개 유스케이스로 분리했고, `admin`은 초대/역할/상태/제거 4개 유스케이스로 분리했습니다.
`auth`와 `admin`은 완전한 포트/어댑터 분리 대신 유스케이스 단위로 로직을 나누고 `SupportService`를 통해 공통 로직을 공유하는 `얇은 헥사고날` 구조를 채택했습니다.

### 4. 대상 모듈은 `public.ts`를 공식 공개 경계로 둔다

`accounting-periods`, `carry-forwards`, `collected-transactions`, `dashboard`, `financial-statements`, `forecast`, `funding-account-status`, `import-batches`, `journal-entries`, `liabilities`, `navigation`, `plan-items`, `recurring-rules`는 이제 모듈 바깥에서 직접 내부 파일을 가져다 쓰지 않고 각 모듈의 `public.ts`를 통해서만 접근합니다.

이 선택의 의미는 단순합니다.

- `collected-transactions.module.ts`, `dashboard-read.service.ts` 같은 파일은 내부 구현입니다.
- 모듈 조립이나 미래의 확장 포인트는 `public.ts`를 기준으로 설명합니다.
- 아직 퍼사드, 인터페이스, 토큰을 과하게 늘리지 않고도 “공개 경계”를 코드에서 보여줄 수 있습니다.

## 의존성 주입은 어디까지 하고 어디서 멈췄는가

이 프로젝트는 “의존성 주입을 많이 썼다”가 목표가 아닙니다.
`외부 의존성을 깔끔하게 주입하고, 내부는 단순하게 유지한다`가 목표입니다.

### 명확하게 분리한 것

- 데이터베이스 접근 경계
- 접근 범위 확인 경계
- 시간 공급 경계
- 일부 모듈의 저장소 접근 경계

### 일부러 과하게 분리하지 않은 것

- 컨트롤러가 유스케이스를 부르는 단순 내부 호출
- 유스케이스가 순수 정책 함수와 도우미 함수를 호출하는 흐름
- 변환기, 형식화 함수, 검증 도우미 함수
- 모든 서비스/저장소 쌍의 1:1 인터페이스화

### 이 판단이 중요한 이유

초보자도 읽을 수 있고, 협업자도 의도를 이해할 수 있으며, 1인 개발자가 끝까지 유지할 수 있어야 하기 때문입니다.

## 왜 전체를 순수 클린/헥사고날로 가지 않았는가

이 프로젝트는 처음부터 끝까지 모든 모듈을 완전히 클린/헥사고날 구조로 바꾸지 않았습니다.
그 이유는 아래와 같습니다.

- 현재 저장소는 이미 건강한 모듈형 레이어드 구조를 갖고 있습니다.
- 모든 모듈을 같은 강도로 분리하면 코드량과 조립 복잡도가 빠르게 늘어납니다.
- 포트폴리오에서 더 중요한 것은 “패턴 사용량”보다 “판단의 적절성”입니다.
- 실제 제품 완성과 운영 준비를 밀어내면 오히려 실무형 포트폴리오로서 약해집니다.

즉, 이 프로젝트는 `무조건 클린 아키텍처`를 보여주는 프로젝트가 아니라 `상황에 맞는 구조 선택`을 보여주는 프로젝트입니다.

즉, 이 프로젝트가 말하는 `외부 의존성은 분리하되, 내부는 과하게 추상화하지 않는다`는 원칙이 테스트에서도 드러나도록 만든 증거입니다.

## 현재 프로젝트는 목적과 판단 원칙에 얼마나 부합하는가

### 전체 판정

`전반적으로 매우 잘 부합합니다.`

### 잘 부합하는 이유

- 웹, API, 공용 계약 경계가 명확합니다.
- 핵심 쓰기 모델 두 개에만 선택적으로 더 엄격한 구조를 적용했습니다.
- 대상 모듈의 모듈 간 import는 `public.ts` 진입점으로 고정해 공개 경계가 코드에서도 드러납니다.
- `dashboard`, `forecast`는 `읽기 서비스`, `읽기 저장소`, `투영` 네이밍으로 보고와 전망 컨텍스트 성격을 코드에서 설명할 수 있습니다.
- 모든 API 응답에 `x-request-id`를 붙이고 `health/ready`를 제공해 최소 운영 신호도 설명할 수 있습니다.
- 외부 의존성 경계만 분리하고 내부 과잉 의존성 주입을 피했습니다.
- 테스트가 구조 의도를 뒷받침합니다.
- `https://personalerp.theworkpc.com` 공개 HTTPS 도메인으로 Web/API 단일 도메인 배포를 검증했습니다.
- 운영, 계약, 개발, ADR 문서 체계가 이미 갖춰져 있습니다.
- 예외 처리와 최소 로깅 기준도 짧은 규칙 문서로 고정해 두었습니다.

### 아직 완전히 끝나지 않은 부분

- GitHub 저장소 보호 규칙은 문서화되어 있지만, 비공개 저장소 플랜이나 실제 설정 상태에 따라 서버 강제가 바로 적용되지 않을 수 있습니다.
- 운영 레벨 세부 정책은 공개 Docker/Caddy 배포까지 검증했지만, 장기 백업/모니터링/장애 복구 자동화는 별도 운영 과제로 남아 있습니다.

이 부분들은 “실패”가 아니라 `현재 단계에서 의도적으로 남겨 둔 현실적인 한계`에 가깝습니다.

## 이 프로젝트에서 의도적으로 하지 않은 것

- 모든 모듈의 일괄적인 클린 아키텍처 전환
- 모든 서비스/저장소 쌍의 인터페이스화
- 프런트엔드 전체의 억지스러운 헥사고날화
- 조기 CQRS, 이벤트 버스, 마이크로서비스 전환
- 보기 좋은 이름만 붙인 과도한 포트/어댑터 추가

이 문서에서 중요한 것은 “무엇을 했는가”만이 아니라 “무엇을 하지 않았는가”도 설명하는 것입니다.

## 포트폴리오 설명 예시

- 이 프로젝트는 Next.js, NestJS, Prisma, MySQL 기반의 1인 사업자·소상공인용 월별 재무 운영 시스템입니다.
- 기본 구조는 모듈러 모놀리스이며, 웹/API 계약은 `packages/contracts`로 통합했습니다.
- 모든 모듈을 같은 방식으로 과하게 추상화하지 않고, 핵심 쓰기 모델인 `collected-transactions`와 `recurring-rules`에만 유스케이스/포트/어댑터 경계를 도입했습니다.
- 의존성 주입은 외부 기술 경계에만 최소한으로 적용했고, 내부 도우미 함수와 변환 함수는 순수 함수로 유지했습니다.
- 이 선택 덕분에 테스트성과 설명력은 높이고, 1인 개발 복잡도는 통제할 수 있었습니다.
