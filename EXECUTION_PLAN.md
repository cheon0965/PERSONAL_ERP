# 프로젝트 실행 계획

## 목적

이 문서는 이미 끝난 정비 내역을 다시 적는 문서가 아닙니다.
현재 시점에 남아 있는 작업 중에서 `협업 안정성`, `형상관리 안정성`, `프로젝트 구조의 확정성`, `배포 신뢰성`에 직접 영향을 주는 과제만 관리합니다.

## 현재 기준선

현재 저장소는 워크스페이스 구조, 공용 계약 계층, 기본 인증 보호, env 검증, migration-first 흐름, CI와 최소 테스트 기준선까지는 정리된 상태입니다.
이제부터의 작업은 “기반 공사”가 아니라, 실제 협업과 운영을 안전하게 만드는 “확정 작업”에 집중합니다.

## 우선순위

1. Git/GitHub 기준선 확정
2. 요청 단위 API 검증 보강
3. Web 인증 세션과 토큰 주입 연결
4. 핵심 입력 플로우의 실제 API 연결
5. 계약과 문서의 동기화 기준 고정
6. 배포와 운영 관점의 안전장치 추가

## 1. Git/GitHub 기준선 확정

### 왜 필요한가

아직 원격 저장소 운영 규칙이 완전히 고정된 상태는 아닙니다.
이 단계가 먼저 닫혀야 이후 구조 작업이 안전하게 누적됩니다.

### 작업

1. 첫 push 전에 업로드 대상 파일을 최종 점검한다.
2. `.env`, 로그, 테스트 산출물, 개인 설정 파일이 절대 추적되지 않도록 다시 확인한다.
3. `main` 브랜치 직접 push 금지, PR 기반 머지, CI 필수 통과를 기본 정책으로 확정한다.
4. GitHub가 인식하는 PR 템플릿 파일을 추가해 PR 설명 형식을 고정한다.
5. GitHub가 인식하는 CODEOWNERS 파일 위치를 택하고, 기본 리뷰 책임 범위를 정한다.
6. 첫 기준 커밋 이후 태그 또는 릴리즈 네이밍 규칙을 정한다.

### 완료 기준

- 첫 push 시 민감 정보가 포함되지 않는다.
- `main`은 보호 브랜치로 운영된다.
- 모든 변경은 PR과 CI를 통과하는 흐름으로만 들어간다.

## 2. 요청 단위 API 검증 보강

### 왜 필요한가

현재 테스트는 핵심 회귀를 막는 데는 충분하지만, 컨트롤러-가드-DTO-응답 배선 문제까지 자동으로 막지는 못합니다.
협업이 시작되면 이 레이어에서 나는 실수가 가장 자주 문제를 만듭니다.

### 작업

1. `POST /auth/login` 성공/실패 요청 테스트를 추가한다.
2. 보호 엔드포인트에 대한 `401` 응답 테스트를 추가한다.
3. `GET /auth/me` 요청 테스트를 추가한다.
4. `POST /transactions`, `POST /recurring-rules`에 대해 DTO validation과 소유권 검증이 함께 동작하는 요청 단위 테스트를 추가한다.
5. 테스트용 DB 초기화 방식과 fixture 전략을 정리한다.
6. 이 테스트를 CI 기본 체인에 포함한다.

### 완료 기준

- 인증 가드, DTO validation, 현재 사용자 주입 흐름이 자동 검증된다.
- 서비스 단위 테스트만 통과하고 실제 HTTP 요청은 깨지는 상태를 줄일 수 있다.

## 3. Web 인증 세션과 토큰 주입 연결

### 왜 필요한가

API는 기본적으로 보호되지만, 현재 Web은 실제 로그인 상태를 기준으로 API를 쓰는 흐름이 완전히 연결되어 있지 않습니다.
이 부분이 닫혀야 구조가 “보여주기용”이 아니라 “실사용 가능한 구조”로 확정됩니다.

### 작업

1. 로그인 성공 후 access token을 어떤 방식으로 유지할지 정책을 확정한다.
2. 공통 fetch 계층에 `Authorization: Bearer <token>` 주입 흐름을 추가한다.
3. `401` 응답 시 처리 정책을 정한다.
4. 현재 사용자 조회와 로그인 상태 초기화 흐름을 만든다.
5. 인증이 필요한 화면에서 비로그인 상태를 다루는 기본 UX를 정한다.

### 완료 기준

- Web에서 실제 로그인 후 보호 API를 정상 호출할 수 있다.
- 토큰 누락이나 만료 시 동작이 예측 가능하다.

## 4. 핵심 입력 플로우의 실제 API 연결

### 왜 필요한가

조회 중심 구조는 어느 정도 갖춰졌지만, 핵심 입력 플로우가 실제 mutation으로 닫혀야 프로젝트 구조의 완성도가 올라갑니다.
여기가 비어 있으면 계약과 상태 흐름이 실제 사용 상황에서 검증되지 않습니다.

### 작업

1. 거래 입력 폼을 실제 `POST /transactions`와 연결한다.
2. 반복규칙 입력 폼을 실제 `POST /recurring-rules`와 연결한다.
3. 성공 후 목록/요약 데이터가 갱신되도록 query invalidation 흐름을 정리한다.
4. 실패 시 사용자에게 보이는 에러 메시지 기준을 통일한다.
5. 로컬 demo fallback이 mutation 흐름을 가리지 않도록 확인한다.

### 완료 기준

- 사용자가 Web에서 거래와 반복규칙을 실제로 등록할 수 있다.
- 입력 후 화면 갱신이 일관되게 동작한다.

## 5. 계약과 문서의 동기화 기준 고정

### 왜 필요한가

지금은 구조가 잘 잡혀 있지만, 시간이 지나면 `contracts`, Swagger, README, API 문서가 따로 흐를 위험이 있습니다.
협업이 길어질수록 “무엇이 진짜 기준인가”를 명확히 해두는 것이 중요합니다.

### 작업

1. 요청/응답 변경 시 `packages/contracts`를 먼저 수정하는 규칙을 더 명시적으로 고정한다.
2. API 문서와 Swagger 중 어떤 것을 1차 기준으로 볼지 정한다.
3. `README`, `API.md`, `PROJECT_PLAN.md`, `VALIDATION_NOTES.md`의 역할 경계를 명확히 유지한다.
4. 구조 결정이 장기 영향이 있으면 ADR을 추가하는 규칙을 계속 유지한다.
5. 문서 변경이 필요한 유형을 PR 템플릿 체크리스트에 포함한다.

### 완료 기준

- 계약 변경 시 함께 수정해야 할 문서가 명확하다.
- 현재 상태 문서와 계획 문서가 서로 충돌하지 않는다.

## 6. 배포와 운영 관점의 안전장치 추가

### 왜 필요한가

현재는 개발과 구조 정리에 강하지만, 운영 관점의 체크리스트는 아직 얇습니다.
이 부분이 정리되어야 “누가 배포해도 같은 결과”에 가까워집니다.

### 작업

1. 배포 전에 필요한 env 키와 GitHub Secrets 목록을 정리한다.
2. build 산출물 기준의 스모크 검증 방법을 정의한다.
3. DB migration 배포 순서를 문서화한다.
4. 시드 데이터와 운영 데이터의 경계를 다시 확인한다.
5. 장애 시 확인 순서와 최소 복구 절차를 간단히 정리한다.

### 완료 기준

- 배포 전 체크할 항목이 문서로 존재한다.
- migration과 env 누락으로 인한 운영 실수를 줄일 수 있다.
